
### 3.3 测试场景执行指南

#### 场景 1：端到端研究流程（核心场景）

**目标**：用户独立完成从需求输入到报告输出的全流程

**步骤**：
1. 创建研究（使用预设模板或自定义）
2. 生成虚拟消费者（50-100个）
3. 查看 CBC 问卷设计
4. 触发模拟作答
5. 运行 HB 分析
6. 查看属性重要性看板
7. 运行市场份额模拟
8. 导出报告

**成功标准**：用户可在无指导情况下完成全部步骤

**支持策略**：
- 第 1-2 步：提供操作视频链接
- 第 3-4 步：在线答疑
- 第 5-8 步：结果解读支持

#### 场景 2：情景模拟（What-if）

**目标**：用户测试不同产品配置的市场份额

**步骤**：
1. 进入"市场份额模拟器"页面
2. 选择已完成的分析结果
3. 配置 3 个竞争产品场景
4. 运行模拟
5. 调整其中一个产品的价格
6. 重新运行，观察份额变化

**成功标准**：份额变化方向符合直觉，数值在合理范围

**支持策略**：
- 提供 2-3 个预设场景供参考
- 解释"相对吸引力"概念
- 强调不要用于真实销量预测

#### 场景 3：细分群体探索

**目标**：用户探索不同画像群体的偏好差异

**步骤**：
1. 进入"细分群体比较"页面
2. 选择研究项目
3. 输入分析结果ID
4. 选择两个细分群体（如"一线城市年轻家庭"vs"新一线品质追求者"）
5. 运行对比分析
6. 查看统计显著性结果

**成功标准**：能发现至少 1 个有价值的细分差异

**支持策略**：
- 解释 p 值和显著性概念（通俗版）
- 帮助识别"有价值"的差异

#### 场景 4：报告解读

**目标**：非技术用户阅读并理解分析报告

**步骤**：
1. 在"分析任务状态"页面找到已完成的分析
2. 点击"报告"按钮查看自动生成的报告
3. 回答 3 个关于关键发现的问题
4. 判断报告是否可直接用于汇报

**成功标准**：回答正确率 ≥ 80%，可直接用于汇报

**支持策略**：
- 提供问题清单和参考答案
- 收集"看不懂"的具体部分

## 四、支持体系

### 4.1 支持渠道

| 渠道 | 用途 | 响应时间 | 负责人 |
|------|------|---------|--------|
| 飞书/微信群 | 日常答疑、快速问题 | 2小时内 | 小培 |
| 在线文档 | 常见问题自助查询 | 即时 | 小培（维护） |
| 预约 1v1 | 复杂问题深度讲解 | 预约后 4 小时内 | 小培 |
| 电话/语音 | 紧急技术问题 | 1小时内 | 小端（技术） |

### 4.2 常见问题预案

| 问题 | 预案 |
|------|------|
| "系统太慢/卡住" | 检查后端服务状态，必要时重启；准备降级方案（mock数据演示） |
| "分析一直不完成" | 检查任务状态，解释HB分析需要时间；提供已完成的结果供查看 |
| "结果看不懂" | 引导查看培训材料；提供 1v1 讲解；记录具体不懂的部分 |
| "发现明显错误" | 立即记录，分类问题；如是技术问题转小端/小数 |
| "操作步骤太多，记不住" | 提供简化版操作清单；收集流程优化建议 |
| "和我知道的市场情况不一样" | 记录具体差异；区分"模型问题"vs"反直觉但正确" |

```markdown

- 日期: ___________
- 场景: ___________

## 问题描述
- 发生了什么: ___________
- 预期结果: ___________
- 实际结果: ___________

## 严重程度
- [ ] CRITICAL（阻碍测试继续）
- [ ] HIGH（严重影响体验）
- [ ] MEDIUM（有影响但可继续）
- [ ] LOW（建议/优化）

## 分类
- [ ] 业务直觉偏差
- [ ] 系统功能缺陷
- [ ] 用户体验问题
- [ ] 偏见/伦理问题
- [ ] 性能问题
- [ ] 文档/培训不足
```

### 5.2 反馈处理流程

```
反馈收集
   │
   ▼
小培初步分类
   │
   ├── 操作问题 → 小培即时解答
   ├── 业务直觉偏差 → 小验判断
   │   ├── 统计事实 → 准备教育材料
   │   └── 模型偏差 → 转小数/小示
   ├── 系统功能缺陷 → 转小端
   ├── 用户体验问题 → 转小前
   ├── 偏见/伦理问题 → 转小伦
   └── 性能问题 → 转小控/小速
   │
   ▼
记录到缺陷跟踪表
   │
   ▼
每日 17:00 汇总会议
   │
   ▼
问题分级 + 分配修复
   │
   ▼
修复后通知原反馈者验证
```

## 六、验收判定标准

### 6.1 通过标准

| 维度 | 标准 | 权重 |
|------|------|------|
| 功能完整性 | 全部 4 个场景可完成 | 30% |
| 结果准确性 | 属性重要性排序与业务认知偏差 ≤ 10% | 25% |
| 系统可用性 | 平均操作成功率 ≥ 80% | 20% |
| 报告可读性 | 理解度 ≥ 80% | 15% |
| 用户满意度 | 总体满意度 ≥ 4.0/5 | 10% |

### 6.2 通过条件

- **全部 CRITICAL 问题修复**
- **HIGH 问题修复率 ≥ 80%**
- **加权总分 ≥ 80%**
- **小验签署验收通过**

### 6.3 不通过处理

如未通过：
1. 小培整理未通过原因
2. 小P组织修复计划会议
3. 确定修复优先级和时间线
4. 修复后重新进行 UAT（或部分场景复测）
