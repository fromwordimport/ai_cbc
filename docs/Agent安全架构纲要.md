## 一、安全目标与原则

### 1.1 安全目标

| 目标 | 说明 | 优先级 |
|------|------|--------|
| **防止提示注入** | 阻止通过输入操控Agent行为 | P0 |
| **防止数据泄露** | 保护API密钥、商业数据、消费者画像 | P0 |
| **确保服务可用** | 防止DoS攻击和资源滥用 | P0 |
| **审计可追溯** | 所有操作可追踪、可审计 | P1 |
| **供应链可信** | 第三方组件来源可信、版本可控 | P1 |

### 1.2 安全原则

1. **纵深防御**：多层防护，单点失效不导致系统失陷
2. **最小权限**：Agent和工具仅拥有必要的权限
3. **默认拒绝**：未明确允许的请求一律拒绝
4. **不可信输入**：所有外部输入视为不可信，必须经过校验
5. **安全左移**：安全审查嵌入每个开发阶段，非事后补丁

| 排名 | 风险 | 影响 | 应对策略 | 状态 |
|------|------|------|---------|------|
| 1 | **提示注入攻击** | Agent行为失控、系统信息泄露 | 输入清洗 + 语义检测 + 关键词过滤 | 已设计 |

## 三、防御策略决策

### 3.1 三层防御架构

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1: 输入层防御（第一道防线）                            │
├─────────────────────────────────────────────────────────────┤
│  • 输入长度限制（按场景设定最大值）                            │
│  • 禁止字符过滤（空字符、替换字符等）                          │
│  • 危险关键词模式匹配（15+注入模式）                           │
│  • Unicode归一化（防止同形字符攻击）                           │
│  • 速率限制（IP级/用户级/全局级）                              │
├─────────────────────────────────────────────────────────────┤
│  Layer 2: 处理层防御（核心防线）                              │
├─────────────────────────────────────────────────────────────┤
│  • Agent沙箱隔离（受限执行环境）                               │
│  • 工具调用白名单（仅允许预定义操作）                          │
│  • 上下文隔离（消费者间数据不泄露）                            │
│  • 输出格式强制约束（JSON Schema校验）                         │
│  • 语义注入检测（轻量LLM辅助判断）                             │
├─────────────────────────────────────────────────────────────┤
│  Layer 3: 输出层+监控层（兜底防线）                            │
├─────────────────────────────────────────────────────────────┤
│  • 输出内容审查（敏感词+语义过滤）                             │
│  • 系统信息泄露检测（Prompt/密钥/配置）                        │
│  • 全量审计日志（correlation_id全链路追踪）                    │
│  • 异常行为检测（规则引擎+统计模型）                           │
│  • 实时告警（安全事件5分钟内响应）                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 关键安全决策

| 决策项 | 决策 | 理由 | 替代方案 |
|--------|------|------|---------|
| **输入校验方式** | 规则引擎为主（快）+ LLM辅助（准） | 平衡效率与准确性 | 纯LLM检测（成本高） |
| **工具调用权限** | 白名单机制，显式允许 | 最小权限原则 | 黑名单机制（易遗漏） |
| **密钥存储** | HashiCorp Vault | 行业标准，支持动态轮换 | 环境变量（不推荐） |
| **API密钥轮换** | 90天强制轮换 | 平衡安全与运维成本 | 30天（太频繁）/ 180天（风险高） |
| **日志保留** | 热存储30天 + 冷存储1年 | 满足审计与合规需求 | 永久保留（成本高） |
| **部署模式** | 私有部署优先 | 数据不出境，满足合规 | 公有云SaaS（数据风险） |

### 5.2 一票否决条件

小安在以下情况下有权要求停止项目推进：

1. **密钥泄露事件**：任何API密钥或敏感配置泄露
2. **防御绕过**：红队测试发现P0级防御绕过且48小时内无法修复
3. **审计失效**：日志系统不可用或审计追踪断裂
4. **权限失控**：发现Agent或工具拥有超出设计范围的权限
