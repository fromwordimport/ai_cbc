## 一、文档审阅结论

### 1.1 现有文档覆盖度评估

| 维度 | 现有文档 | 覆盖度 | 主要 gaps |
|------|---------|--------|-----------|
| **部署架构** | `docs/系统部署与运维架构.md` + `consumer-simulation/15-运维手册.md` | 中高 | 两文档存在架构不一致（运维手册含Kafka，部署架构文档无Kafka）；缺少阶段化落地路径 |
| **CI/CD 流水线** | 系统部署文档第2章 | 中 | 流程完整但缺少阶段1前置条件（代码仓库、测试框架、容器化基线） |
| **监控告警** | 两文档均有指标体系 | 高 | 指标定义存在重复和命名不一致，需统一 |
| **模型版本管理** | 系统部署文档第4章 | 中 | 缺少模型A/B测试基础设施和效果回传机制 |
| **成本熔断** | `docs/项目成本预算方案.md` + `docs/成本管控与预算模型.md` | 高 | 需与部署侧监控告警打通，形成自动闭环 |
| **安全运维** | `docs/Agent安全架构纲要.md` | 中高 | 安全门禁与CI/CD门禁的集成方式待明确 |
| **容灾备份** | 两文档均有涉及 | 中 | 缺少RTO/RPO量化目标和恢复演练计划 |

### 1.2 关键发现

1. **架构不一致风险**：`15-运维手册.md` 引入 Kafka Cluster 作为消息队列，但 `系统部署与运维架构.md` 的核心服务层未包含 Kafka，数据层也未明确消息队列选型。阶段3/4需统一为 **Redis Queue / Celery**（MVP）或 **Kafka**（生产）。
2. **成本-监控闭环待打通**：小控的成本熔断器（`CostCircuitBreaker`）与小维的 Prometheus 告警规则（`HighCostPerPersona`）是两套独立机制，需通过统一事件总线集成。
3. **CI/CD 前置条件缺失**：当前流水线假设代码仓库、测试框架、容器化已就绪，但项目处于阶段1前置，这些基线尚未建立。
4. **模型路由缺少观测回环**：`ModelRouter` 可根据预算状态降级，但缺少降级后的质量指标回传（如真实性评分变化），无法验证降级策略是否可接受。
5. **安全门禁与部署门禁未联动**：红队测试通过率 ≥95% 是阶段2门禁，但 CI/CD 中仅做"安全测试（红队用例抽样）"，缺少明确的阻断条件。

## 二、部署架构初步方案

### 2.1 设计原则

| 原则 | 说明 |
|------|------|
| **阶段演进** | MVP用Docker Compose；当前生产环境使用 Azure VM + Docker Compose，Kubernetes 作为未来高可用扩展选项 |
| **成本优先** | 所有基础设施选型需符合 ¥25,000/3个月 的总预算约束 |
| **安全左移** | CI/CD 流水线嵌入安全扫描、偏见检测、成本预算检查 |
| **可观测性优先** | 先建监控和告警，再上生产流量 |
| **模型可切换** | LLM调用层抽象统一，支持多供应商、多模型动态路由 |

**MVP 组件说明**：

| 组件 | 选型 | 理由 |
|------|------|------|
| API 框架 | FastAPI | 异步支持好，Prometheus指标集成成熟 |
| 任务队列 | Celery + Redis | 复杂度低，满足MVP并发需求，避免引入Kafka |
| 主存储 | MongoDB | 适配画像/问卷/回答的半结构化数据 |
| 分析元数据 | PostgreSQL | 分析结果结构化存储，支持复杂查询 |
| 对象存储 | MinIO | 兼容S3 API，本地部署零成本 |
| 缓存 | Redis | 复用同一实例，分db使用 |
| 监控 | Prometheus + Grafana + Loki | 开源栈，与Docker Compose集成简单 |
| 反向代理 | Nginx | 成熟稳定，支持SSL和限流 |

#### 阶段3-4：生产演进（Azure VM + Docker Compose，K8s 作为未来扩展）

> **当前状态**：生产环境已确定运行在 Azure 云服务器（VM）上，通过 `cd-azure-b2ats.yml` 自动 SSH 部署。Kubernetes 作为未来高可用、多可用区部署的扩展选项，在阶段4暂不实装。

在MVP基础上增加：

| 新增能力 | 选型 | 触发条件 |
|---------|------|---------|
| 生产部署 | Azure VM + Docker Compose（当前） | 当前实际方案 |
| 容器编排 | Kubernetes (K3s/EKS/ACK)（未来） | 日请求 > 1000 或需要高可用 |
| 服务网格 | Istio / Linkerd（可选） | 多服务间需要细粒度流量控制 |
| 外部负载均衡 | ALB / CLB | 多可用区部署 |
| 托管数据库 | MongoDB Atlas / 阿里云MongoDB | 数据量 > 100GB 或需要自动备份 |
| 托管Redis | ElastiCache / 阿里云Redis | 生产高可用需求 |
| 对象存储 | AWS S3 / 阿里云OSS | 替代MinIO，降低运维负担 |
| 密钥管理 | HashiCorp Vault / 云KMS | 满足90天轮换和审计要求 |
| 链路追踪 | Jaeger / Tempo | 多Agent协作调试需要 |
| 日志平台 | Loki + Grafana 或 ELK | 日志量 > 10GB/天 |

### 4.2 关键告警规则（按优先级）

| 告警名 | 表达式 | 级别 | 通知人 | 自动动作 |
|--------|--------|------|--------|---------|
| `HighCostPerPersona` | `aicbc_cost_per_persona_cny > 5` | Warning | 小控、小P | 无 |
| `CostDegradeThreshold` | 日费用 > ¥228 (95%) | Critical | 小控、小P、小应 | 触发模型降级 |
| `CostFuseThreshold` | 日费用 > ¥240 (100%) | Critical | 全员 | 暂停新任务 |
| `LowAuthenticityScore` | `aicbc_authenticity_score_average < 8` | Critical | 小测、小示、小P | 建议暂停生成 |
| `HighErrorRate` | 5min错误率 > 10% | Critical | 小维、小端、小P | 无 |
| `LLMAPIRateLimit` | 429响应率 > 0 | Warning | 小维、小应 | 无 |
| `QueueBacklog` | 队列长度 > 100 持续10min | Warning | 小维、小端 | 建议扩容Worker |
| `InjectionAttackDetected` | 5min拦截 > 5次 | Critical | 小安、小维、小P | 无 |
| `ModelDegradeQualityImpact` | 降级后真实性分下降 > 10% | Warning | 小控、小应、小示 | 无 |

### 4.3 成本-监控闭环设计

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  LLM调用    │────→│ CostTracker │────→│ 时序数据库   │
│  (各Agent)  │     │ (每次调用)  │     │ (Prometheus) │
└─────────────┘     └─────────────┘     └──────┬──────┘
                                               │
                                               ▼
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ 模型路由器   │←────│ 熔断决策器   │←────│ 告警规则引擎 │
│ (执行降级)  │     │ (check_and_act)│    │  (Prometheus)│
└─────────────┘     └─────────────┘     └─────────────┘
        │
        ▼
┌─────────────┐
│ 质量回传     │
│ (真实性评分) │
└─────────────┘
```

**关键集成点**：
- `CostTracker` 每次LLM调用后推送 `aicbc_cost_per_persona_cny` 到 Prometheus
- Prometheus 告警触发时，通过 Alertmanager webhook 调用 `CostCircuitBreaker.check_and_act()`
- 熔断器根据级别执行：告警 → 通知 / 降级 → 切换模型 / 熔断 → 暂停任务
- 降级后，系统持续监控 `aicbc_authenticity_score_average`，若下降超过10%触发 `ModelDegradeQualityImpact` 告警

### 5.3 模型切换流程

| 切换场景 | 触发条件 | 审批人 | 回滚条件 |

| 成本降级 | 费用达预算95% | 自动（小控知情） | 费用降至80%以下或质量下降 > 10% |
| 故障切换 | 某模型API连续失败 > 10次/5min | 自动（小维知情） | 原模型恢复且稳定5min |
| A/B测试 | 配置即代码中启用 | 小应 | 实验结束或显著负向 |
