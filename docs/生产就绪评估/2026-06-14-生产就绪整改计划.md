# AI_CBC 生产就绪整改计划

## Context

AI_CBC MVP v0.1.0 在 2026-06-14 生产就绪评估中被判定为**不可以从 mock 开发转入实际生产环境**。核心阻塞项包括：

- 数据全内存存储，无持久化
- 安全红队防御成功率仅 56%（目标≥95%）
- 6 条公平性硬规则未嵌入 Prompt
- 测试覆盖率 28%，前端 0%
- 前端 10 个 P0 问题，12 个后端端点未暴露
- NLScenarioParser / ReportBuilder / CBCVisualizer / LatentClassModel 文档定义但代码零实现
- 无正式压力测试报告

本计划的目标是通过**按模块逐个交付**的方式，将系统整改为可转入生产环境的版本。用户已确认：

- 持久化数据库：**MongoDB**（使用现有 Beanie ODM / docker-compose 服务）
- 缺失四大模块：**P0 必须实现**
- 交付方式：**按模块逐个交付**，不严格预设总周期

---

## 整改策略

1. **先基线，后功能**：先完成安全、合规、数据持久化三大基线，再补齐前端 API 与缺失模块。
2. **内存存储保留 fallback**：改造 `store.py` 时保留内存实现作为 `ENVIRONMENT=development` 回退，降低迁移风险。
3. **前后端契约以数据字典为唯一真相源**：所有字段变更同步更新 `docs/数据字典.md`、`api/schemas.py`、`analysis/models.py`、`frontend/src/types/api.ts`。
4. **测试先行**：每个持久化/安全模块在实现时同步补充测试，不接受新代码零测试。
5. **每个模块独立验收**：每完成一个模块即进行单元测试、集成测试、代码审查，通过后再进入下一模块。

## 风险与回滚

| 风险 | 缓解措施 |
|------|----------|
| MongoDB 迁移导致数据丢失 | 保留内存 fallback；迁移前备份；灰度切换 |
| 安全加固引入性能瓶颈 | 每个 sanitize 操作加性能基准；必要时加缓存 |
| 前端测试环境搭建复杂 | 使用 Vitest（与 Vite 同生态） |
| 公平性规则误杀正常输出 | 增加白名单机制；人工审核首批结果 |
| 缺失模块实现延期 | 每个模块独立验收，延期不影响其他模块 |

**回滚机制**：
- 数据层：`USE_MEMORY_STORE=1` 回退到内存存储。
- API 变更：保持向后兼容，新增字段均为 optional。
- 每个模块完成后打 tag：`v0.2.0-m1` ~ `v0.2.0-m10`。
