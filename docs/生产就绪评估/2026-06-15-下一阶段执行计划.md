## 一、计划目标

当前项目处于 **"代码基线已补齐，生产依赖未真实验证，正式否决尚未解除"** 的状态。本计划的目标是：

1. 在真实生产依赖栈（MongoDB + Redis + Celery + Docker/K8s）上验证系统可行性。
2. 完成测试、安全、性能、公平性、隐私合规、前端覆盖等全部生产准入门禁。
3. 收集足够证据，召开复验会议，书面解除 小伦、小验 的否决权，并让小速确认是否行使/解除阶段 4 性能否决权。
4. **不进入生产环境**，只进入受控 Staging 验证阶段。

## 三、阶段划分与时间线

### 第一阶段：Staging 环境基线验证（第 1-2 周）

**目标**：让系统在真实生产依赖栈上跑起来，并修复测试挂起问题。

**关键交付**：
- Docker Compose 全栈可启动且健康检查通过
- MongoDB 路径读写验证通过
- 全量测试（含 coverage）稳定通过
- 性能压测脚本可在生产栈上运行

### 第二阶段：门禁补齐与证据收集（第 2-3 周）

**目标**：完成安全、公平性、隐私合规、前端 Staging 冒烟等全部验证。

**关键交付**：
- 6 条公平性硬规则 100% 嵌入 Prompt 并通过 redteam 测试

- 安全门禁（bandit + pip-audit + npm audit）集成 CI

### 第三阶段：性能压测与 UAT（第 3-4 周）

**目标**：完成生产栈压测和业务 UAT，准备复验会议。

**关键交付**：

### 第四阶段：复验与决策（第 4 周末）

## 七、风险与回滚策略

| 风险 | 影响 | 缓解措施 | 负责人 |
|------|------|----------|--------|
| BE-1 coverage 挂起根因复杂 | 阻塞 CI 门禁与后续所有任务 | 优先投入小端 + 小测；必要时拆分测试套件逐模块定位 | 小端 |
| MongoDB/Redis/Celery 路径存在隐藏 bug | 生产栈验证失败 | 保留 `USE_MEMORY_STORE=1` 回退；逐项接口对比内存与 MongoDB 行为 | 小端 |
| 生产栈压测不达标 | 小速行使否决权 | 提前在本地 Docker Compose 做 50 并发预压测，预留优化时间 | 小速 |
| 公平性规则误杀正常输出 | 画像生成成功率下降 | 增加白名单机制；小示参与首批结果人工审核 | 小伦/小示 |
| UAT 反馈大量前端问题 | 小验拒签 | 小前提前完成 Staging 冒烟，小培准备培训材料 | 小前/小培 |
| LLM API 成本超支 | 小控熔断触发 | 压测与 UAT 阶段使用小批量样本；监控 CostTracker | 小控 |

**回滚机制**：
- 数据层：`USE_MEMORY_STORE=1` 回退到内存存储。
- 部署层：K8s 保留上一版本 ReplicaSet，可快速回滚。
- 配置层：所有新增字段保持 optional，向后兼容。
