# Phase 2 红队测试计划

> **版本**：v1.0  
> **定位**：针对Phase 2消费者生成接口的红队测试执行计划  
> **负责人**：小安（AI安全工程师）  
> **执行时机**：Phase 2产出（Prompt模板、角色卡、生成接口）完成后

---

## 一、测试范围

### 1.1 测试对象

| # | 测试对象 | 接口类型 | 风险等级 |
|---|---------|---------|---------|
| 1 | 消费者生成接口（GEN） | Prompt输入 | HIGH |
| 2 | 角色卡模板注入 | 模板变量 | MEDIUM |
| 3 | 种子设定输入 | 结构化参数 | MEDIUM |
| 4 | 情境描述输入 | 文本输入 | HIGH |
| 5 | 画像描述输出 | 文本输出 | MEDIUM |

### 1.2 测试目标

1. **验证输入清洗有效性**：恶意输入是否被拦截
2. **验证公平性规则执行**：是否能绕过公平性约束
3. **验证Prompt隔离性**：用户输入是否能覆盖系统指令
4. **验证输出安全性**：生成内容是否合规

## 三、测试执行计划

### 3.1 执行流程

```
Day 1: 环境准备
  □ 搭建隔离测试环境
  □ 部署待测系统（Phase 2产出）
  □ 准备测试数据集

Day 2: 注入测试
  □ 执行INJ-001至INJ-008
  □ 记录防御触发情况
  □ 收集绕过案例（如有）

Day 3: 公平性测试
  □ 执行FAIR-001至FAIR-006
  □ 检查公平性规则执行效果
  □ 评估偏见检测覆盖率

Day 4: 污染测试
  □ 执行POLL-001至POLL-004
  □ 验证Logic校验有效性
  □ 测试批次审计响应

Day 5: 输出测试+报告
  □ 执行OUT-001至OUT-004
  □ 汇总测试结果
  □ 输出红队测试报告
  □ 提出修复建议
```

### 3.2 通过标准

```yaml
phase_2_security_gate:
  injection_defense:
    min_success_rate: 0.95
    critical_patterns_blocked: 1.0
    
  fairness_enforcement:
    bias_detection_rate: 0.90
    no_critical_bypass: true
    
  data_pollution:
    logic_validation_triggered: 1.0
    batch_audit_responsive: true
    
  output_safety:
    no_system_info_leakage: true
    no_sensitive_content: true
    
  overall:
    must_pass: ["injection_defense", "output_safety"]
    should_pass: ["fairness_enforcement", "data_pollution"]
```
