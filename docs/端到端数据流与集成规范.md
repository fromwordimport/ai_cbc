## 一、数据序列化格式

### 1.1 JSON Schema 定义

全链路所有子系统间数据交换采用 JSON 格式，顶层结构统一为信封模式：

```json
{
  "schema_version": "1.0",
  "message_id": "msg_{uuid}",
  "timestamp": "2026-06-09T14:30:00Z",
  "sender": "subsystem_code",
  "receiver": "subsystem_code",
  "payload": {},
  "metadata": {
    "retry_count": 0,
    "ttl_seconds": 300,
    "priority": "normal"
  }
}
```

**信封字段约束**：

| 字段 | 类型 | 必填 | 约束 |
|------|------|------|------|
| `schema_version` | string | 是 | 必须符合 SemVer，当前 `"1.0"` |
| `message_id` | string | 是 | 全局唯一，格式 `msg_{uuid}` |
| `timestamp` | string | 是 | ISO 8601 UTC，精确到毫秒 |
| `sender` | string | 是 | 枚举：`GEN`, `SIM`, `REV`, `FIX`, `ORCH`, `QST`, `ANL`, `DASH` |
| `receiver` | string | 是 | 同上枚举 |
| `payload` | object | 是 | 业务数据，按实体 Schema 校验 |
| `metadata.retry_count` | integer | 否 | 重试次数，默认 0，最大 3 |
| `metadata.ttl_seconds` | integer | 否 | 消息存活时间，默认 300 |
| `metadata.priority` | string | 否 | 枚举：`critical`, `high`, `normal`, `low` |

**超时处理策略**：

1. 同步接口超时 → 返回 `504 Gateway Timeout`，客户端可重试
2. 异步任务整体超时 → 任务状态变为 `FAILED`，触发补偿操作
3. LLM 调用超时 → 计入重试次数，触发指数退避
4. 数据库超时 → 记录告警，重试 1 次，仍失败则返回 503

### 2.3 重试策略（指数退避）

#### 2.3.1 退避算法

```python
def exponential_backoff(attempt: int, base_delay: float = 1.0, max_delay: float = 60.0) -> float:
    """
    计算第 attempt 次重试的等待时间（秒）

    公式：delay = min(base_delay * 2^attempt + jitter, max_delay)
    jitter: 随机偏移 [0, 1) 秒，避免惊群
    """
    jitter = random.random()
    delay = min(base_delay * (2 ** attempt) + jitter, max_delay)
    return delay
```

**退避参数表**：

| 场景 | base_delay | max_delay | max_attempts | 适用错误码 |
|------|------------|-----------|--------------|------------|
| LLM API 调用 | 2s | 60s | 3 | 429, 500, 502, 503 |
| 数据库写入 | 1s | 30s | 3 | 超时, 主节点切换 |
| 外部 HTTP 调用 | 1s | 30s | 3 | 5xx, 超时 |
| 消息队列发送 | 0.5s | 10s | 5 | 连接失败 |
| 内部服务调用 | 0.5s | 10s | 2 | 超时 |

#### 2.3.2 重试条件

**可重试错误**：
- HTTP 429 (Too Many Requests)
- HTTP 500, 502, 503, 504 (服务端错误)
- 连接超时 / 读取超时
- 数据库主节点切换
- 消息队列临时不可用

**不可重试错误**：
- HTTP 400 (Bad Request) — 请求参数错误
- HTTP 401/403 — 认证/授权失败
- HTTP 404 — 资源不存在
- 业务错误码 4xx 系列 — 客户端逻辑错误
- 业务错误码 9xx 系列 — 业务规则违反

#### 2.3.3 重试实现规范

```python
class RetryPolicy:
    """统一重试策略"""

    def __init__(self, max_attempts: int, base_delay: float, max_delay: float):
        self.max_attempts = max_attempts
        self.base_delay = base_delay
        self.max_delay = max_delay
        self.retryable_status_codes = {429, 500, 502, 503, 504}

    async def execute(self, operation: Callable, *args, **kwargs) -> Any:
        last_exception = None

        for attempt in range(self.max_attempts + 1):
            try:
                return await operation(*args, **kwargs)
            except RetryableError as e:
                last_exception = e
                if attempt < self.max_attempts and self._is_retryable(e):
                    delay = exponential_backoff(attempt, self.base_delay, self.max_delay)
                    logger.warning(f"Retry {attempt + 1}/{self.max_attempts} after {delay:.2f}s: {e}")
                    await asyncio.sleep(delay)
                else:
                    break
            except NonRetryableError as e:
                raise

        raise MaxRetriesExceededError(f"Failed after {self.max_attempts} attempts") from last_exception

    def _is_retryable(self, error: Exception) -> bool:
        if hasattr(error, 'status_code'):
            return error.status_code in self.retryable_status_codes
        if hasattr(error, 'error_code'):
            return str(error.error_code).startswith(('1', '5'))
        return isinstance(error, (TimeoutError, ConnectionError))
```

### 2.4 幂等性保证

#### 2.4.1 幂等性设计原则

| 操作类型 | 幂等性要求 | 实现方式 |
|----------|------------|----------|
| 研究创建 | 幂等 | 客户端提供 `client_request_id`，服务端去重 |
| 画像生成 | 非幂等（每次结果不同） | 不支持幂等，重复调用产生新画像 |
| 问卷生成 | 幂等（相同参数应产生相同设计） | `client_request_id` + 参数哈希去重 |
| 批量模拟 | 幂等（相同画像+问卷应产生相同结果） | `client_request_id` + 参数哈希去重 |
| 分析执行 | 幂等（相同数据应产生相同结果） | `client_request_id` + 数据版本哈希去重 |
| 市场模拟 | 幂等 | `client_request_id` + 场景配置哈希去重 |
| 数据导出 | 幂等 | `client_request_id` + 查询条件哈希去重 |

#### 2.4.2 幂等键管理

```json
{
  "client_request_id": "req_client_001",
  "idempotency_key_ttl": 86400
}
```

**幂等键存储**：
- 存储位置：Redis
- Key 格式：`idempotency:{client_request_id}`
- Value：`{"status": "processing/completed", "result_ref": "...", "created_at": "..."}`
- TTL：24 小时

**幂等处理流程**：

```
接收请求（含 client_request_id）
    │
    ▼
查询 Redis：idempotency:{client_request_id} 是否存在？
    │
    ├──→ 存在且 status=completed ──→ 返回缓存结果
    │
    ├──→ 存在且 status=processing ──→ 返回 409 Conflict，提示任务进行中
    │
    └──→ 不存在 ──→ 写入 Redis（status=processing）
                          │
                          ▼
                    执行业务逻辑
                          │
                          ▼
                    更新 Redis（status=completed, result_ref=...）
                          │
                          ▼
                    返回结果
```

#### 2.4.3 幂等性边界

- **时间边界**：幂等键 TTL 为 24 小时，超过后重复请求视为新请求
- **空间边界**：幂等键按子系统隔离，`idempotency:gen:{key}`、`idempotency:qst:{key}`、`idempotency:anl:{key}`
- **失败边界**：业务逻辑执行失败但 Redis 已写入 processing 状态时，需设置守护任务定期清理过期 processing 记录

### 3.4 错误码使用规范

1. **错误码唯一性**：每个错误码全局唯一，不可复用
2. **HTTP 状态映射**：一个错误码对应唯一的 HTTP 状态码
3. **客户端处理**：客户端根据 `code` 的前两位判断错误大类，根据完整 `code` 执行具体处理
4. **日志记录**：所有错误必须记录 `request_id`、`error_code`、堆栈（5xx 类）
5. **告警规则**：5xx 类错误触发即时告警，9xx 类错误按频率触发告警

## 四、数据流状态转换

### 4.1 研究级状态机

研究（`CBCStudy`）作为全链路编排的聚合根，其状态转换定义如下：

```
[INIT] ──→ 用户提交研究需求
    │
    ▼
[PERSONA_GENERATING] ──→ GEN Agent 批量生成画像
    │
    ├─→ 审校不通过 ──→ [PERSONA_FIXING] ──→ FIX Agent 修正 ──→ 重新审校
    │                                                    │
    │                                                    └─→ 超过重试 ──→ [FAILED]
    │
    ├─→ 偏见检测失败 ──→ [BIAS_AUDITING] ──→ 修正后重试 ──→ 或 [FAILED]
    │
    └─→ 画像生成完成 ──→ [QUESTIONNAIRE_BUILDING]
                              │
                              ▼
                    [QUESTIONNAIRE_BUILDING] ──→ 实验设计引擎生成问卷
                              │
                              ├─→ D-efficiency < 0.85 ──→ [DESIGN_OPTIMIZING] ──→ 重新设计
                              │
                              └─→ 问卷就绪 ──→ [SIMULATING]
                                                    │
                                                    ▼
                                          [SIMULATING] ──→ 批量模拟引擎执行作答
                                                    │
                                                    ├─→ 进度更新（每 10% 推送事件）
                                                    ├─→ 成本告警 ──→ [DEGRADED] ──→ 切换轻量模型 ──→ 恢复 [SIMULATING]
                                                    ├─→ 部分失败 ──→ 记录失败明细，继续
                                                    └─→ 模拟完成 ──→ [ANALYZING]
                                                                          │
                                                                          ▼
                                                                [ANALYZING] ──→ 自动触发分析
                                                                          │
                                                                          ├─→ 模型不收敛 ──→ [MODEL_RETRYING] ──→ 增加采样/调整先验 ──→ 重新分析
                                                                          │                                                                  │
                                                                          │                                                                  └─→ 仍不收敛 ──→ [FAILED]
                                                                          └─→ 分析完成 ──→ [REPORTING]

                                                                                                ▼

                                                                                                            [COMPLETED] ──→ 通知用户
                                                                                                                      │
                                                                                                                      ▼

```

### 4.2 状态定义详表

| 状态码 | 状态名 | 说明 | 前置条件 | 后置动作 | 允许停留时间 |
|--------|--------|------|----------|----------|--------------|
| `INIT` | 初始化 | 研究刚创建 | 无 | 创建研究记录 | 无限制 |
| `PERSONA_GENERATING` | 画像生成中 | GEN Agent 正在生成 | 研究配置已保存 | 启动批量生成任务 | ≤ 30min |
| `PERSONA_FIXING` | 画像修正中 | FIX Agent 正在修正 | 审校未通过 | 应用修正后重新审校 | ≤ 15min |
| `BIAS_AUDITING` | 偏见审计中 | 偏见检测正在执行 | 画像生成完成 | 审计通过后继续 | ≤ 10min |
| `QUESTIONNAIRE_BUILDING` | 问卷构建中 | 实验设计引擎运行中 | 画像生成完成 | 生成问卷并计算 D-efficiency | ≤ 10min |
| `DESIGN_OPTIMIZING` | 设计优化中 | D-efficiency 不足，重新设计 | D-efficiency < 0.85 | 调整参数后重新生成 | ≤ 20min |
| `SIMULATING` | 模拟作答中 | 批量模拟引擎运行中 | 问卷已就绪 | 执行逐题模拟 | ≤ 2h |
| `DEGRADED` | 降级运行中 | 成本告警，切换轻量模型 | 成本超过预算 80% | 切换模型后继续 | ≤ 30min |
| `ANALYZING` | 分析中 | HB 模型拟合中 | 模拟数据已收集 | 执行统计建模 | ≤ 1h |
| `MODEL_RETRYING` | 模型重试中 | 不收敛，调整参数重试 | R-hat >= 1.1 或 ESS < 400 | 增加采样或调整先验 | ≤ 1h |
| `REPORTING` | 报告生成中 | LLM 解读 + 报告组装 | 分析结果已就绪 | 生成 Markdown/HTML 报告 | ≤ 5min |
| `COMPLETED` | 已完成 | 全流程结束 | 报告已生成 | 通知用户，开放结果访问 | 无限制 |
| `FAILED` | 已失败 | 流程中断，无法自动恢复 | 任意状态遇到不可恢复错误 | 记录失败原因，通知用户 | 无限制 |
| `ARCHIVED` | 已归档 | 数据已迁移到冷存储 | COMPLETED 后 90 天 | 清理热存储，保留元数据 | 永久 |

### 4.3 状态转换规则

#### 4.3.1 正常流转

| 当前状态 | 触发条件 | 目标状态 | 执行动作 |
|----------|----------|----------|----------|
| `INIT` | 用户提交研究 | `PERSONA_GENERATING` | 保存配置，启动画像生成 |
| `PERSONA_GENERATING` | 全部画像审校通过 | `QUESTIONNAIRE_BUILDING` | 启动问卷生成 |
| `QUESTIONNAIRE_BUILDING` | D-efficiency >= 0.85 | `SIMULATING` | 启动批量模拟 |
| `SIMULATING` | 全部作答完成 | `ANALYZING` | 触发自动分析 |
| `ANALYZING` | 模型收敛 | `REPORTING` | 启动报告生成 |
| `REPORTING` | 报告就绪 | `COMPLETED` | 通知用户 |
| `COMPLETED` | 90 天到期 | `ARCHIVED` | 数据归档 |

#### 4.3.2 异常分支

| 当前状态 | 触发条件 | 目标状态 | 执行动作 |
|----------|----------|----------|----------|
| `PERSONA_GENERATING` | 审校不通过 | `PERSONA_FIXING` | 生成 FIX 指令 |
| `PERSONA_FIXING` | 修正后审校通过 | `PERSONA_GENERATING` | 继续生成剩余画像 |

| `ANALYZING` | 模型不收敛 | `MODEL_RETRYING` | 调整参数 |

| 任意状态 | 不可恢复错误 | `FAILED` | 记录失败，清理资源 |

#### 4.3.3 状态转换校验

```python
class StudyStateMachine:
    """研究状态机"""

    VALID_TRANSITIONS = {
        "INIT": ["PERSONA_GENERATING", "FAILED"],
        "PERSONA_GENERATING": ["PERSONA_FIXING", "BIAS_AUDITING", "QUESTIONNAIRE_BUILDING", "FAILED"],
        "PERSONA_FIXING": ["PERSONA_GENERATING", "FAILED"],
        "BIAS_AUDITING": ["QUESTIONNAIRE_BUILDING", "FAILED"],
        "QUESTIONNAIRE_BUILDING": ["DESIGN_OPTIMIZING", "SIMULATING", "FAILED"],
        "DESIGN_OPTIMIZING": ["QUESTIONNAIRE_BUILDING", "SIMULATING", "FAILED"],
        "SIMULATING": ["DEGRADED", "ANALYZING", "FAILED"],
        "DEGRADED": ["SIMULATING", "FAILED"],
        "ANALYZING": ["MODEL_RETRYING", "REPORTING", "FAILED"],
        "MODEL_RETRYING": ["ANALYZING", "REPORTING", "FAILED"],
        "REPORTING": ["COMPLETED", "FAILED"],
        "COMPLETED": ["ARCHIVED"],
        "FAILED": [],  # 终态，不可转换
        "ARCHIVED": []  # 终态，不可转换
    }

    def transition(self, from_state: str, to_state: str, context: dict) -> bool:
        """执行状态转换"""
        if to_state not in self.VALID_TRANSITIONS.get(from_state, []):
            raise InvalidStateTransitionError(
                f"Cannot transition from {from_state} to {to_state}"
            )

        # 执行前置校验
        self._validate_preconditions(from_state, to_state, context)

        # 执行后置动作
        self._execute_post_actions(from_state, to_state, context)

        return True

    def _validate_preconditions(self, from_state: str, to_state: str, context: dict):
        """校验状态转换的前置条件"""
        preconditions = {
            ("INIT", "PERSONA_GENERATING"): lambda c: c.get("study_config") is not None,
            ("PERSONA_GENERATING", "QUESTIONNAIRE_BUILDING"): lambda c: c.get("persona_count", 0) >= c.get("min_personas", 1),
            ("QUESTIONNAIRE_BUILDING", "SIMULATING"): lambda c: c.get("d_efficiency", 0) >= 0.85,
            ("SIMULATING", "ANALYZING"): lambda c: c.get("simulation_completion_rate", 0) >= 0.9,
            ("ANALYZING", "REPORTING"): lambda c: c.get("converged", False),
        }

        check = preconditions.get((from_state, to_state))
        if check and not check(context):
            raise StatePreconditionError(
                f"Precondition not met for transition {from_state} -> {to_state}"
            )

    def _execute_post_actions(self, from_state: str, to_state: str, context: dict):
        """执行状态转换的后置动作"""
        post_actions = {
            ("REPORTING", "COMPLETED"): lambda c: self._notify_user(c["study_id"]),
            ("COMPLETED", "ARCHIVED"): lambda c: self._archive_data(c["study_id"]),
            ("ANY", "FAILED"): lambda c: self._cleanup_resources(c["study_id"]),
        }

        action = post_actions.get((from_state, to_state)) or post_actions.get(("ANY", to_state))
        if action:
            action(context)
```

### 4.4 子任务状态管理

研究级状态由多个子任务状态聚合而成：

  "subtasks": {

**子任务状态聚合规则**：
- 所有子任务 `COMPLETED` → 研究状态推进到下一阶段

- 子任务 `IN_PROGRESS` → 研究状态保持当前阶段
- 子任务 `PENDING` → 等待前置依赖完成

## 五、数据一致性保证

### 5.1 一致性需求分析

| 数据流 | 一致性要求 | 原因 |
|--------|------------|------|
| 画像生成 → 画像库 | 强一致性 | 画像必须完整写入后才能被问卷系统引用 |
| 问卷生成 → 问卷库 | 强一致性 | 问卷必须完整生成后才能绑定模拟 |
| 模拟作答 → 回答库 | 最终一致性 | 允许短暂延迟，但分析前必须全部就绪 |
| 回答库 → 分析系统 | 最终一致性 | 支持增量同步，分析可基于部分数据 |
| 分析结果 → 报告 | 强一致性 | 报告必须基于完整的分析结果 |
| 成本记录 → 预算检查 | 强一致性 | 成本熔断必须基于准确累计值 |

### 5.2 分布式事务策略

#### 5.2.1 Saga 模式（长事务编排）

全链路研究执行采用 Saga 模式，将长事务拆分为多个本地事务，通过补偿操作保证最终一致性。

```
研究执行 Saga：

T1: 创建研究配置 ──→ 补偿: 删除研究记录
    │
    ▼
T2: 批量生成画像 ──→ 补偿: 标记画像为废弃
    │
    ▼
T3: 生成 CBC 问卷 ──→ 补偿: 删除问卷记录
    │
    ▼
T4: 执行批量模拟 ──→ 补偿: 删除回答记录
    │
    ▼
T5: 执行分析建模 ──→ 补偿: 删除分析结果
    │
    ▼
T6: 生成洞察报告 ──→ 补偿: 删除报告文件
```

**Saga 实现规范**：

```python
class StudySaga:
    """研究执行 Saga 编排器"""

    STEPS = [
        SagaStep("create_study", compensator="delete_study"),
        SagaStep("generate_personas", compensator="discard_personas"),
        SagaStep("build_questionnaire", compensator="delete_questionnaire"),
        SagaStep("run_simulation", compensator="delete_responses"),
        SagaStep("run_analysis", compensator="delete_analysis_result"),
        SagaStep("generate_report", compensator="delete_report"),
    ]

    async def execute(self, study_id: str, config: dict):
        executed_steps = []

        try:
            for step in self.STEPS:
                result = await step.execute(study_id, config)
                executed_steps.append((step, result))
        except Exception as e:
            # 执行补偿操作（逆序）
            for step, _ in reversed(executed_steps):
                try:
                    await step.compensate(study_id)
                except CompensationError as ce:
                    logger.error(f"Compensation failed for {step.name}: {ce}")
                    # 记录补偿失败，需人工介入
                    await self._escalate_compensation_failure(study_id, step.name, ce)

            raise SagaExecutionError(f"Saga failed at step {step.name}") from e

        return SagaResult(success=True, study_id=study_id)
```

#### 5.2.2 本地事务边界

| 事务边界 | 操作集合 | 存储 |
|----------|----------|------|
| 画像写入 | 插入画像文档 + 更新批次状态 + 写入审计日志 | MongoDB 单文档事务 |
| 问卷写入 | 插入问卷文档 + 更新研究状态 | MongoDB 单文档事务 |
| 回答写入 | 插入回答文档 + 更新进度计数器 + 累加成本 | MongoDB 单文档事务 |
| 分析结果写入 | 插入结果文档 + 更新研究状态 + 生成报告引用 | MongoDB 单文档事务 |
| 成本记录 | 累加成本 + 检查预算 + 更新熔断状态 | Redis 原子操作 + Lua 脚本 |

### 5.3 最终一致性机制

#### 5.3.1 事件驱动同步

```
┌─────────────┐    事件    ┌─────────────┐    事件    ┌─────────────┐
│  画像系统    │ ────────→ │  消息队列    │ ────────→ │  问卷系统    │
│             │ PERSONA_CREATED│          │            │             │
└─────────────┘            └─────────────┘            └─────────────┘
                                  │
                                  ▼
                           ┌─────────────┐
                           │  分析系统    │
                           │             │
                           └─────────────┘
```

**事件格式**：

```json
{
  "event_id": "evt_{uuid}",
  "event_type": "PERSONA_CREATED",
  "aggregate_id": "persona-dw-001",
  "aggregate_type": "PersonaProfile",
  "timestamp": "2026-06-09T14:30:00Z",
  "payload": {
    "persona_id": "persona-dw-001",
    "segment": "精致白领",
    "study_id": "dishwasher-202506"
  },
  "version": 1
}
```

**事件消费保证**：
- 至少一次投递（At-least-once delivery）
- 消费者幂等处理（基于 `event_id` 去重）
- 事件顺序性：同一聚合的事件按版本号顺序消费

#### 5.3.2 定时对账

| 对账项 | 频率 | 对账内容 | 不一致处理 |
|--------|------|----------|------------|
| 画像数量 | 每小时 | 生成任务数 vs 库中记录数 | 补录缺失记录 |
| 回答完整性 | 每 30 分钟 | 模拟任务数 vs 回答记录数 | 标记缺失，触发重试 |
| 成本累计 | 每 10 分钟 | Redis 累计 vs MongoDB 明细 | 以明细为准修正 |
| 分析结果 | 每次分析后 | 结果文档 vs 研究状态 | 同步状态 |
| 状态一致性 | 每 5 分钟 | 研究状态 vs 子任务状态聚合 | 修正漂移 |

#### 5.3.3 读取修复

```python
class ReadRepairService:
    """读取时修复不一致数据"""

    async def get_study_with_subtasks(self, study_id: str) -> Study:
        study = await self.study_repo.get(study_id)
        subtasks = await self.subtask_repo.get_by_study(study_id)

        # 校验状态一致性
        aggregated_status = self._aggregate_status(subtasks)
        if study.status != aggregated_status:
            logger.warning(f"Status drift detected: study={study.status}, aggregated={aggregated_status}")
            # 以子任务聚合状态为准（子任务更细粒度）
            study.status = aggregated_status
            await self.study_repo.update_status(study_id, aggregated_status)

        study.subtasks = subtasks
        return study
```

### 5.4 补偿操作

| 场景 | 补偿操作 | 触发条件 | 执行方式 |
|------|----------|----------|----------|
| 画像生成失败 | 标记已生成为 `DISCARDED` | Saga T2 失败 | 异步批量更新 |
| 问卷生成失败 | 删除问卷文档 | Saga T3 失败 | 同步删除 |
| 模拟中断 | 删除部分回答，重置任务状态 | 任务超时或取消 | 异步清理 |
| 分析失败 | 删除部分结果，释放计算资源 | 模型不收敛且重试耗尽 | 同步清理 |
| 成本超支 | 暂停后续任务，通知管理员 | 成本 >= 预算 | 自动触发 |
| 偏见审计失败 | 标记批次为 `REJECTED`，禁止进入下游 | 审计不通过 | 同步标记 |

**补偿操作幂等性**：
- 所有补偿操作必须幂等（重复执行无副作用）
- 补偿操作记录独立日志（compensation_log 集合）
- 补偿失败触发告警，需人工介入

#### 6.6.2 Schema 演进规则

#### 6.6.3 数据迁移策略

```python
class SchemaMigrator:
    """Schema 迁移器"""

    MIGRATIONS = {
        "1.0->1.1": [
            AddField("new_field", default=None),
            RenameField("old_name", "new_name"),
        ],
        "1.1->2.0": [
            RemoveField("deprecated_field"),
            TransformField("price", lambda x: float(x)),
        ]
    }

    def migrate(self, data: dict, from_version: str, to_version: str) -> dict:
        """将数据从 from_version 迁移到 to_version"""
        current = from_version
        result = data.copy()

        while current != to_version:
            next_version = self._next_version(current)
            migration_key = f"{current}->{next_version}"

            for operation in self.MIGRATIONS.get(migration_key, []):
                result = operation.apply(result)

            current = next_version

        result["schema_version"] = to_version
        return result
```
