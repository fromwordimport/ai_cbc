# 业务验收标准与KPI框架

> **版本**：v1.0  
> **定位**：定义AI_CBC系统从业务视角的"成功标准"，组织UAT验收，追踪业务价值  
> **负责人**：小验（业务需求与质量验收负责人）  
> **一票否决权**：当业务价值存疑时，有权阻止项目进入下一阶段

---

## 一、业务成功定义

### 1.1 核心问题

> **"这个系统能不能帮业务做决策？"**

所有技术指标最终必须转化为可感知的业务价值。我们定义三个层级的成功：

```
┌─────────────────────────────────────────────────────────────┐
│                    业务成功三层级                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 3: 战略成功                                          │
│  ────────────                                               │
│  • 虚拟消费者研究替代 ≥ 30% 的传统调研场景                   │
│  • 产品决策周期从4周缩短到1周                                │
│  • 年度调研成本降低 ≥ 50%                                    │
│                                                             │
│  Level 2: 项目成功                                          │
│  ────────────                                               │
│  • 分析结果通过业务验收（属性重要性偏差 ≤ 10%）              │
│  • 市场份额预测通过拟合度检验                                │
│  • 业务用户能独立操作系统并解读结果                          │
│                                                             │
│  Level 1: 技术成功                                          │
│  ────────────                                               │
│  • 系统功能完整（生成→问卷→填写→分析→报告）                │
│  • 虚拟消费者"像真人"（真实性评分 ≥ 8.5）                   │
│  • 统计模型可靠（参数恢复误差 ≤ 15%）                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 业务价值主张

| 业务痛点 | 传统方式 | AI_CBC方案 | 价值量化 |
|---------|---------|-----------|---------|
| 概念测试等待时间长 | 招募+执行+分析 = 4-6周 | 数小时出结果 | 时间缩短 **90%** |
| 小众人群难以招募 | 高成本、低响应率 | 按需生成任意画像 | 覆盖度提升 **100%** |
| 调研成本高昂 | 单个项目 ¥50,000-200,000 | 单次标准研究目标 ¥500（业务可接受上限 ¥2,000） | 成本降低 **95%** |
| 假设验证慢 | 必须等完整调研后才能调整 | 快速迭代测试假设 | 迭代速度提升 **10x** |
| 交叉分析受限 | 样本量限制细分分析 | 可生成足够样本量 | 细分深度无限制 |

## 二、业务级验收标准

### 2.1 验收维度与指标

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                      业务验收四大维度                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ① 结果准确性                    ② 洞察深度                                │
│  ─────────────                   ─────────                                 │
│  • 属性重要性排序偏差 ≤ 10%      • 能发现传统调研遗漏的细分群体             │
│  • 市场份额预测拟合度 R² ≥ 0.8   • 交叉分析维度 ≥ 5个画像特征               │
│  • 价格弹性与历史数据偏差 ≤ 20%  • 提供可行动的决策建议                     │
│                                                                             │
│  ③ 系统可用性                    ④ 业务适配性                              │
│  ─────────────                   ───────────                               │
│  • 业务用户独立操作成功率 ≥ 80%  • 支持业务常用产品类别 ≥ 3个               │
│  • 结果解读无需统计背景          • 输出格式适配现有决策流程                 │
│  • 培训后上手时间 ≤ 2小时        • 可与真实调研结果对比验证                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 各阶段验收标准

#### 阶段1：安全与业务基准设定（前置门禁）

| 验收项 | 验收标准 | 验收方法 | 负责人 |
|--------|---------|---------|--------|
| 业务需求明确性 | 研究目标、关键问题、决策场景已书面确认 | 需求评审会 | 小验 |
| 成功标准共识 | 所有利益相关方签署《成功标准书》 | 签字确认 | 小验 |
| 预算合理性 | 单次全流程成本在预算范围内 | 成本模型审核 | 小控 |

#### 阶段2：领域设计与行为仿真

| 验收项 | 验收标准 | 验收方法 | 负责人 |
|--------|---------|---------|--------|
| 属性/水平合理性 | 业务专家确认属性列表覆盖决策维度 | 专家评审 | 小联+小验 |
| 消费者画像真实性 | 盲测中 ≥ 70% 的画像被判断为"真实" | 人工盲测 | 小测+小验 |
| 行为一致性 | 同一画像在不同情境下行为逻辑自洽 | 场景测试 | 小示+小验 |

#### 阶段4：全链路串联与业务验收（核心门禁）

| 验收项 | 验收标准 | 验收方法 | 通过阈值 |
|--------|---------|---------|---------|
| **属性重要性准确性** | 与历史真实调研对比，排序偏差 | 历史数据对比 | TOP3属性排序完全一致 |
| **效用值方向正确性** | 价格效用为负、优质属性效用为正 | 方向检查 | 100%正确 |
| **市场份额预测** | 与已知市场数据对比 | 拟合度检验 | R² ≥ 0.5 且方向正确 |
| **WTP合理性** | 支付意愿在业务合理范围内 | 专家评审 | 与业务认知偏差 ≤ 30% |
| **细分群体区分度** | 不同画像群体的偏好差异显著 | 统计检验 | p < 0.05 |
| **报告可读性** | 业务用户无需解释即可理解 | 用户测试 | 理解度 ≥ 80% |
| **系统操作流畅性** | 从输入到报告输出的全流程 | 端到端测试 | 成功率 ≥ 90% |

## 三、项目KPI体系

### 3.1 KPI框架

| KPI类别 | 指标名称 | 定义 | 目标值 | 测量频率 | 负责人 |
|---------|---------|------|--------|---------|--------|
| **效率** | 替代调研比例 | 虚拟消费者研究替代传统调研的项目占比 | ≥ 30% | 月度 | 小验 |
| **效率** | 决策周期缩短 | 从需求提出到洞察输出的平均时间 | ≤ 1周 | 项目级 | 小验 |
| **效率** | 单次研究成本 | 单次CBC全流程的平均成本（业务可接受上限） | ≤ ¥2,000 | 项目级 | 小控 |
| **质量** | 结果准确性评分 | 与真实调研对比的综合准确性评分 | ≥ 8.5/10 | 项目级 | 小验 |
| **质量** | 模型可恢复性 | 已知参数恢复误差 | ≤ 15% | 季度 | 小测 |
| **质量** | 消费者真实性 | 人工盲测真实感评分 | ≥ 8.5/10 | 批次 | 小测 |
| **质量** | 偏见检测通过率 | 偏见审计批次通过率 | 100% | 批次 | 小伦 |
| **采纳** | 业务用户满意度 | 业务方对结果的满意度评分 | ≥ 4.0/5 | 项目级 | 小验 |
| **采纳** | 系统独立使用率 | 业务用户独立完成研究的比例 | ≥ 60% | 月度 | 小培 |
| **采纳** | 培训完成率 | 业务用户完成培训的比例 | ≥ 80% | 季度 | 小培 |
| **成本** | 成本节约率 | (传统成本 - AI_CBC成本) / 传统成本 | ≥ 50% | 季度 | 小控 |
| **成本** | 预算偏差率 | 实际花费 vs 预算 | ≤ 10% | 月度 | 小控 |

## 四、用户验收测试（UAT）流程

### 4.1 UAT参与者

| 角色 | 人数 | 职责 | 招募方式 |
|------|------|------|---------|
| 市场研究员 | 2-3人 | 评估分析方法的严谨性 | 内部招募 |
| 产品经理 | 2-3人 | 评估结果对决策的支持度 | 目标用户 |
| 品牌经理 | 1-2人 | 评估品牌相关洞察的合理性 | 内部招募 |
| 高管/决策者 | 1人 | 评估报告可直接用于决策 | 关键利益方 |

### 4.2 UAT测试场景

```yaml
uat_scenarios:
  scenario_001_end_to_end:
    name: "端到端研究流程"
    description: "用户独立完成从需求输入到报告输出的全流程"
    steps:
      - 输入产品属性和研究目标
      - 生成虚拟消费者（观察画像质量）
      - 生成CBC问卷（检查选择集合理性）
      - 执行模拟填写（查看回答和理由）
      - 运行分析模型（查看效用值和重要性）
      - 生成洞察报告（评估可读性和可操作性）
    success_criteria: "用户可在无指导情况下完成全部步骤"
    
  scenario_002_what_if:
    name: "情景模拟（What-if）"
    description: "用户测试不同产品配置的市场份额"
    steps:
      - 设定基准竞争情景
      - 调整自己产品的价格和配置
      - 观察份额变化
      - 验证结果是否符合业务直觉
    success_criteria: "份额变化方向符合直觉，数值在合理范围"
    
  scenario_003_segment_exploration:
    name: "细分群体探索"
    description: "用户探索不同画像群体的偏好差异"
    steps:
      - 按年龄/收入/城市筛选消费者群体
      - 对比不同群体的属性重要性
      - 发现跨群体的差异化洞察
    success_criteria: "能发现至少1个有价值的细分差异"
    
  scenario_004_report_reading:
    name: "报告解读"
    description: "非技术用户阅读并理解分析报告"
    steps:
      - 阅读自动生成的洞察报告
      - 回答关于关键发现的3个问题
      - 判断报告是否可直接用于汇报
    success_criteria: "回答正确率 ≥ 80%，可直用于汇报"
```

### 4.4 UAT结果处理流程

```
UAT执行
   │
   ▼
收集评分和"不可理喻"反馈
   │
   ▼
分类问题
   ├── 业务直觉偏差 → 小联审核 → 可能是模型问题或真实但反直觉
   ├── 系统功能缺陷 → 小端/小应修复
   ├── 用户体验问题 → 小前优化
   ├── 偏见/伦理问题 → 小伦处理
   └── 成本/性能问题 → 小控优化
   │
   ▼
问题分级
   ├── CRITICAL → 必须修复后才能上线
   ├── HIGH → 建议修复，可延期
   ├── MEDIUM → 纳入迭代计划
   └── LOW → 记录，暂不处理
   │
   ▼
验收判定
   ├── 全部CRITICAL修复 → 通过
   └── 有未修复CRITICAL → 不通过 → 返工
```

## 五、反馈收集与缺陷转化

### 5.1 "感觉不合理"反馈转译流程

```
业务用户反馈
   │ "这个价位段消费者不可能这么看重品牌"
   ▼
小培收集并结构化
   │
   ▼
小验分类判断
   │
   ├── 业务直觉 vs 统计事实 → 小联判断
   │   • 如果是统计事实 → 教育业务用户
   │   • 如果是模型偏差 → 转技术改进
   │
   ├── 虚拟消费者不真实 → 小示改进
   │   • 提示模板调整
   │   • 偏好注入修正
   │
   ├── 模型估计偏差 → 小数改进
   │   • 先验调整
   │   • 模型选择优化
   │
   └── 系统展示误导 → 小前改进
       • 可视化优化
       • 置信区间展示
```

| ID | 来源 | 描述 | 类型 | 严重程度 | 负责人 | 状态 | 验证方式 |

| BUG-001 | UAT | "40岁消费者选了最便宜选项，与收入不符" | 画像不真实 | HIGH | 小示 | 修复中 | 盲测验证 |
