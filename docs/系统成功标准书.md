## 一、成功定义

### 1.1 核心命题

> **"这个系统能不能帮业务做决策？"**

所有技术指标最终必须转化为可感知的业务价值。本文档将"成功"从抽象概念转化为可测量、可验证、可签字确认的具体标准。

### 1.2 成功三层级

```
┌─────────────────────────────────────────────────────────────┐
│                    业务成功三层级                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Level 3: 战略成功                                          │
│  ────────────                                               │
│  • 虚拟消费者研究替代 >= 30% 的传统调研场景                   │
│  • 产品决策周期从4周缩短到1周                                │
│  • 年度调研成本降低 >= 50%                                   │
│                                                             │
│  Level 2: 项目成功                                          │
│  ────────────                                               │
│  • 分析结果通过业务验收（属性重要性偏差 <= 10%）              │
│  • 市场份额预测通过拟合度检验                                │
│  • 业务用户能独立操作系统并解读结果                          │
│                                                             │
│  Level 1: 技术成功                                          │
│  ────────────                                               │
│  • 系统功能完整（生成 -> 问卷 -> 填写 -> 分析 -> 报告）      │
│  • 虚拟消费者"像真人"（真实性评分 >= 9/14）                 │
│  • 统计模型可靠（参数恢复误差 <= 15%）                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 业务价值主张

| 业务痛点 | 传统方式 | AI_CBC方案 | 价值量化 |
|---------|---------|-----------|---------|
| 概念测试等待时间长 | 招募+执行+分析 = 4-6周 | 数小时出结果 | 时间缩短 **90%** |
| 小众人群难以招募 | 高成本、低响应率 | 按需生成任意画像 | 覆盖度提升 **100%** |
| 调研成本高昂 | 单个项目 ¥50,000-200,000 | 单次标准研究目标 ¥500（业务可接受上限 ¥2,000） | 成本降低 **95%** |
| 假设验证慢 | 必须等完整调研后才能调整 | 快速迭代测试假设 | 迭代速度提升 **10x** |
| 交叉分析受限 | 样本量限制细分分析 | 可生成足够样本量 | 细分深度无限制 |

## 二、业务级验收标准

### 2.1 结果准确性

| 验收项 | 验收标准 | 验证方法 | 通过阈值 | 一票否决 |
|--------|---------|---------|---------|---------|
| **属性重要性排序** | TOP3属性符合业务直觉，排序偏差可控 | 专家评审 + 历史数据对比 | 排序完全一致（TOP3） | 否 |
| **属性重要性数值** | 各属性重要性权重与历史调研偏差 | 历史数据对比 | 偏差 <= 10% | 否 |
| **价格效用方向** | 价格系数必须为负 | 自动检查 | 100%正确 | **是** |
| **市场份额预测** | 与已知市场数据对比拟合度 | 拟合度检验 | R^2 >= 0.5 且方向正确 | 否 |
| **WTP合理性** | 支付意愿在业务合理范围内 | 专家评审 | 与业务认知偏差 <= 30% | 否 |
| **价格弹性** | 与历史数据偏差可控 | 历史数据对比 | 偏差 <= 20% | 否 |

### 2.2 洞察深度

| 验收项 | 验收标准 | 验证方法 | 通过阈值 |
|--------|---------|---------|---------|
| **细分群体差异** | 不同群体偏好差异可被统计检验确认 | 统计检验（p < 0.05）+ 专家评审 | p < 0.05 |
| **交叉分析维度** | 支持按画像特征进行多维度交叉分析 | 功能测试 | >= 5个画像特征维度 |
| **反直觉洞察** | 至少发现1个超出专家预期的有价值发现 | 专家评审 | >= 1个 |
| **决策建议** | 输出包含可行动的决策建议 | 专家评审 | 有明确建议 |

### 2.3 系统可用性

| 验收项 | 验收标准 | 验证方法 | 通过阈值 |
|--------|---------|---------|---------|
| **独立操作成功率** | 业务用户无需指导完成全流程 | UAT测试 | 成功率 >= 80% |
| **上手时间** | 培训后用户可独立操作系统 | 计时测试 | <= 2小时 |
| **结果解读** | 非技术人员可直接理解报告 | 用户测试 | 理解度 >= 80% |
| **报告可用性** | 输出报告可直接用于决策会议 | 专家评审 | 可直接使用 |

### 2.4 业务适配性

| 验收项 | 验收标准 | 验证方法 | 通过阈值 |
|--------|---------|---------|---------|
| **产品类别覆盖** | 支持业务常用产品类别 | 功能测试 | >= 3个类别 |
| **产品切换时间** | 从当前产品切换到新产品配置 | 计时测试 | <= 30分钟 |
| **决策流程适配** | 输出格式适配现有决策流程 | 专家评审 | 适配 |
| **真实调研对比** | 可与真实调研结果交叉验证 | 功能测试 | 支持对比 |

### 2.5 可接受偏差范围

| 指标 | 目标值 | 可接受上限 | 说明 |
|------|--------|-----------|------|
| 属性重要性数值偏差 | <= 10% | <= 15% | 与专家预期或历史数据对比 |
| WTP估计偏差 | <= 25% | <= 30% | 与业务认知对比 |
| 市场份额预测偏差 | <= 10个百分点 | <= 15个百分点 | 方向必须正确 |
| 消费者真实性评分 | >= 真实评分的80% | >= 75% | 盲测结果 |
| 参数恢复误差（MAPE） | <= 15% | <= 20% | 已知参数恢复测试 |

### 2.6 偏差处理流程

所有量化指标按四级阈值管理，测量结果落入不同区间触发不同处理动作：

```
测量值落在...
   │
   ├── 目标值范围内 → 绿色，正常通过
   │
   ├── 可接受上限内 → 黄色，记录并监控趋势
   │                    └── 连续2次黄色 → 触发根因分析
   │
   ├── 警戒线内 → 橙色，必须修复后方可继续
   │              └── 小P协调相关责任人制定修复计划
   │
   └── 超出警戒线 → 红色，立即停止当前批次
                    └── 小验/小伦可行使否决权
```

| 指标 | 目标值（绿色） | 可接受上限（黄色） | 警戒线（橙色） | 红色处理 |
|------|--------------|-------------------|---------------|---------|
| 属性重要性数值偏差 | <= 10% | <= 15% | > 15% | 停止批次，根因分析 |
| 市场份额预测偏差 | <= 10个百分点 | <= 15个百分点 | > 15个百分点 | 停止批次，根因分析 |
| WTP估计偏差 | <= 25% | <= 30% | > 30% | 停止批次，根因分析 |
| 消费者真实性评分 | >= 80% | >= 75% | < 75% | 停止批次，盲测复验 |
| 参数恢复误差 | <= 10% | <= 15% | > 15% | 停止批次，模型审查 |
| 单次研究成本 | <= 500元 | <= 800元 | > 800元 | 触发成本熔断 |

## 五、Must/Should/Nice-to-Have 分级

### 5.1 Must Have（必须满足）

不满足则项目不通过：

- [ ] 属性重要性排序TOP3与业务直觉一致
- [ ] 价格效用方向为负（100%正确）
- [ ] HB模型收敛（R-hat < 1.1）
- [ ] 消费者真实性盲测 >= 真实评分的75%
- [ ] 偏见审计100%通过（无CRITICAL级别偏见，HIGH级别已修正）
- [ ] 6条公平性硬规则已全部嵌入消费者生成Prompt
- [ ] 系统可用性 >= 80%（业务用户独立操作成功率）
- [ ] 单次研究成本 <= 500元
- [ ] 红队防御成功率 >= 95%

### 5.2 Should Have（期望满足）

不满足需记录原因并制定改进计划：

- [ ] 细分群体差异显著（p < 0.05）
- [ ] WTP估计偏差 <= 30%
- [ ] 市场份额预测方向正确且R^2 >= 0.5
- [ ] 报告可直接用于决策会议
- [ ] 发现至少1个反直觉洞察
- [ ] 业务用户满意度 >= 4.0/5

### 5.3 Nice to Have（加分项）

满足则提升项目评级：

- [ ] 支持实时"what-if"市场模拟
- [ ] 生成消费者决策过程可视化
- [ ] 同一套系统可快速切换到其他产品（<= 30分钟）
- [ ] 根据使用数据持续改进消费者真实性
- [ ] 支持按任意画像维度进行交叉分析

## 六、关键业务问题（必须回答）

### 6.1 首个验证场景：洗碗机定价研究

1. **在3000-6000元价位段，消费者最愿意为哪些属性付费？**
   - 预期：容量和核心功能可能是关键

2. **台式、嵌入式、水槽式三种安装方式，各自的溢价空间有多大？**
   - 预期：嵌入式溢价最高，台式门槛最低

3. **品牌vs功能，哪个对购买决策的影响更大？**
   - 预期：不同细分群体可能有显著差异

4. **超一级能耗是否足以支撑500元以上的溢价？**
   - 预期：在部分群体中可能成立

5. **三个细分群体的偏好差异有多大？是否需要推出差异化产品？**
   - 预期：差异显著，建议差异化配置

### 6.2 通用能力验证问题

1. 系统能否在2小时内完成从需求输入到报告输出的全流程？
2. 更换产品（如从洗碗机换成智能手表）后，系统能否在30分钟内重新配置？
3. 系统生成的消费者画像是否支持导出和人工审核？
4. 分析结果是否支持按任意画像维度进行交叉分析？
5. 虚拟消费者研究能否替代 >= 30% 的传统调研场景？

## 十一、自我优化声明

系统具备根据每次使用数据进行自我优化的能力：
- **偏好校准**：根据用户反馈调整消费者偏好分布
- **Prompt优化**：积累高质量画像样本，优化生成模板
- **模型迭代**：持续运行参数恢复测试，改进建模策略

但自我优化不改变以下原则：
- 公平性硬规则始终强制执行
- 偏见审计每批次必做
- 人工盲测定期执行（每迭代周期至少1次）

## 十二、使用限制与免责声明

### 12.1 适用范围

**适用场景**：
- 产品概念快速筛选
- 定价策略探索
- 属性重要性排序
- 小众人群需求挖掘
- 假设快速验证

**不适用场景**：
- 最终上市定价的唯一依据
- 精确销量预测
- 法律法规相关的合规决策
- 涉及健康安全的关键决策

### 12.2 建议的使用方式

```
AI_CBC研究结果
      |
      ├──→ 与真实消费者研究交叉验证（推荐）
      |
      ├──→ 与行业数据/竞品分析对比
      |
      └──→ 作为内部讨论和假设筛选的输入
                |
                ▼
         结合专家判断 -> 形成最终决策
```
