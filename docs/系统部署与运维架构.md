### 1.2 MVP部署（Docker Compose）

```yaml
# docker-compose.yml (MVP阶段)
version: '3.8'

services:
  # API服务
  api:
    build: ./src/api
    ports:
      - "8000:8000"
    environment:
      - MONGODB_URL=mongodb://mongo:27017/aicbc
      - REDIS_URL=redis://redis:6379
      - LLM_API_KEY=${LLM_API_KEY}
      - ENV=production
    depends_on:
      - mongo
      - redis
    volumes:
      - ./logs:/app/logs
    restart: unless-stopped
    deploy:
      resources:
        limits:
          memory: 2G
        reservations:
          memory: 512M

  # 工作者（异步任务处理）
  worker:
    build: ./src/worker
    environment:
      - MONGODB_URL=mongodb://mongo:27017/aicbc
      - REDIS_URL=redis://redis:6379
      - LLM_API_KEY=${LLM_API_KEY}
    depends_on:
      - mongo
      - redis
    restart: unless-stopped
    deploy:
      replicas: 2

  # MongoDB
  mongo:
    image: mongo:6
    ports:
      - "27017:27017"
    volumes:
      - mongo_data:/data/db
      - ./backups/mongo:/backup
    restart: unless-stopped

  # Redis
  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    volumes:
      - redis_data:/data
    restart: unless-stopped

  # Nginx（反向代理）
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf
      - ./ssl:/etc/nginx/ssl
    depends_on:
      - api
    restart: unless-stopped

  # Prometheus（监控）
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
    volumes:
      - ./prometheus.yml:/etc/prometheus/prometheus.yml
      - prometheus_data:/prometheus
    restart: unless-stopped

  # Grafana（可视化）
  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"
    volumes:
      - grafana_data:/var/lib/grafana
      - ./grafana/dashboards:/etc/grafana/provisioning/dashboards
    depends_on:
      - prometheus
    restart: unless-stopped

volumes:
  mongo_data:
  redis_data:
  prometheus_data:
  grafana_data:
```

### 1.3 生产部署（当前：Azure VM + Docker Compose）

> **当前状态**：生产环境实际部署在 Azure 云服务器（VM）上，由 `.github/workflows/cd-azure-b2ats.yml` 通过 SSH 自动拉取镜像并执行 `docker compose` 部署。`k8s/` 目录下的 manifests 仍保留，用于 CI 静态验证（`validate-k8s`）和本地 kind/minikube 测试，不再作为生产部署入口。

生产部署核心流程：

1. CI 通过后将镜像推送到 GHCR（`ghcr.io/fromwordimport/ai_cbc:<sha>`）。
2. `cd-azure-b2ats.yml` 通过 SSH 登录生产 VM。
3. 在 VM 上拉取最新代码，执行 `scripts/deploy-to-azure-b2ats.sh` 更新容器。
4. 部署后通过 `/health` 和 worker 状态做健康检查。

```yaml
# docker-compose.azure-b2ats.yml（生产实际使用）
# 与 docker-compose.yml 结构类似，资源限制和外部依赖按 Azure B2ats VM 调整
services:
  api:
    image: ghcr.io/fromwordimport/ai_cbc:${IMAGE_TAG}
    # ...
```

### 2.2 GitHub Actions配置

```yaml
# Split workflows: .github/workflows/ci.yml, cd-staging.yml, cd-azure-b2ats.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      
      - name: Install dependencies
        run: |
          pip install -r requirements-dev.txt
      
      - name: Lint
        run: ruff check src/
      
      - name: Type check
        run: mypy src/
      
      - name: Security scan
        run: |
          bandit -r src/
          safety check -r requirements.txt

  test:
    runs-on: ubuntu-latest
    needs: quality
    steps:
      - uses: actions/checkout@v4
      
      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      
      - name: Install dependencies
        run: pip install -r requirements.txt
      
      - name: Run tests
        run: pytest tests/ --cov=src --cov-report=xml --cov-fail-under=60
      
      - name: Upload coverage
        uses: codecov/codecov-action@v3
        with:
          file: ./coverage.xml

  integration-test:
    runs-on: ubuntu-latest
    needs: test
    steps:
      - uses: actions/checkout@v4
      
      - name: Run integration tests
        run: |
          docker-compose -f docker-compose.test.yml up -d
          pytest tests/integration/ -v
          docker-compose -f docker-compose.test.yml down

  build:
    runs-on: ubuntu-latest
    needs: [test, integration-test]
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      
      - name: Build Docker image
        run: |
          docker build -t aicbc/api:${{ github.sha }} .
          docker tag aicbc/api:${{ github.sha }} aicbc/api:latest
      
      - name: Scan image
        run: trivy image aicbc/api:${{ github.sha }}
      
      - name: Push to registry
        run: |
          echo ${{ secrets.REGISTRY_PASSWORD }} | docker login registry.example.com -u ${{ secrets.REGISTRY_USER }} --password-stdin
          docker push aicbc/api:${{ github.sha }}
          docker push aicbc/api:latest

  deploy:
    runs-on: ubuntu-latest
    needs: build
    if: github.ref == 'refs/heads/main'
    environment: azure-b2ats
    steps:
      - uses: actions/checkout@v4
      
      - name: Deploy to production via SSH
        uses: appleboy/ssh-action@v1.0.3
        env:
          IMAGE_TAG: ${{ github.sha }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        with:
          host: ${{ secrets.AZURE_VM_IP }}
          username: ${{ secrets.AZURE_VM_USER }}
          key: ${{ secrets.AZURE_VM_SSH_KEY }}
          envs: IMAGE_TAG,GITHUB_TOKEN
          script_stop: true
          script: |
            cd /opt/aicbc
            git remote set-url origin https://x-access-token:${GITHUB_TOKEN}@github.com/fromwordimport/ai_cbc.git
            git fetch origin master
            git reset --hard origin/master
            export IMAGE_TAG=${IMAGE_TAG}
            bash scripts/deploy-to-azure-b2ats.sh
      
      - name: Smoke test
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.AZURE_VM_IP }}
          username: ${{ secrets.AZURE_VM_USER }}
          key: ${{ secrets.AZURE_VM_SSH_KEY }}
          script_stop: true
          script: |
            cd /opt/aicbc
            for i in {1..30}; do
              if docker compose -f docker-compose.azure-b2ats.yml exec -T api python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" > /dev/null 2>&1; then
                echo "Health check passed"
                exit 0
              fi
              echo "Waiting for API health... ($i/30)"
              sleep 2
            done
            echo "ERROR: Health check failed" >&2
            exit 1
      
      - name: Notify team
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {
              "text": "🚀 AI_CBC 生产环境已部署\n版本: ${{ github.sha }}\n部署人: ${{ github.actor }}"
            }
        env:
          SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK }}
```

### 3.2 告警规则

```yaml
alerting_rules:
  # 业务告警
  - alert: HighCostPerPersona
    expr: cost_per_persona_cny > 5
    for: 1h
    labels:
      severity: warning
    annotations:
      summary: "单个消费者成本过高"
      description: "当前成本 ¥{{ $value }}/人，超过阈值 ¥5"
      action: "检查模型路由和批量优化配置"
      notify: ["小控", "小P"]

  - alert: LowAuthenticityScore
    expr: authenticity_score_average < 8
    for: 30m
    labels:
      severity: critical
    annotations:
      summary: "消费者真实性评分过低"
      description: "批次平均分 {{ $value }}，低于阈值 8"
      action: "暂停生成，检查Prompt模板"
      notify: ["小测", "小示", "小P"]

  # 系统告警
  - alert: HighErrorRate
    expr: rate(api_requests_total{status=~"5.."}[5m]) / rate(api_requests_total[5m]) > 0.1
    for: 5m
    labels:
      severity: critical
    annotations:
      summary: "API错误率过高"
      description: "5分钟错误率 {{ $value }}"
      notify: ["小维", "小端", "小P"]

  - alert: LLMAPIRateLimit
    expr: rate(llm_api_calls_total{status="429"}[5m]) > 0
    for: 1m
    labels:
      severity: warning
    annotations:
      summary: "LLM API触发速率限制"
      description: "检测到429响应"
      action: "检查并发控制和重试策略"
      notify: ["小维", "小应"]

  - alert: QueueBacklog
    expr: queue_size > 100
    for: 10m
    labels:
      severity: warning
    annotations:
      summary: "任务队列堆积"
      description: "当前队列长度 {{ $value }}"
      action: "扩容Worker或检查处理瓶颈"
      notify: ["小维", "小端"]

  # 安全告警
  - alert: InjectionAttackDetected
    expr: rate(injection_attempts_total{blocked="true"}[5m]) > 5
    for: 1m
    labels:
      severity: critical
    annotations:
      summary: "检测到注入攻击"
      description: "5分钟内拦截 {{ $value }} 次注入尝试"
      action: "检查攻击源IP，必要时封禁"
      notify: ["小安", "小维", "小P"]

  - alert: SuspiciousActivity
    expr: rate(suspicious_requests_total[5m]) > 10
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "异常请求激增"
      description: "5分钟内 {{ $value }} 次异常请求"
      notify: ["小安", "小维"]
```

### 4.2 模型版本切换

```python
class ModelRouter:
    """
    模型路由器 - 支持动态切换LLM模型
    用于成本优化、A/B测试、故障切换
    """
    
    MODEL_CONFIGS = {
        "claude-sonnet-4-6": {
            "provider": "anthropic",
            "input_cost_per_1k": 0.003,
            "output_cost_per_1k": 0.015,
            "max_tokens": 200000,
            "quality_tier": "high",
            "use_for": ["persona_generation", "choice_simulation", "deep_analysis", "default"]
        },
        "claude-haiku-4-5": {
            "provider": "anthropic",
            "input_cost_per_1k": 0.00025,
            "output_cost_per_1k": 0.00125,
            "max_tokens": 50000,
            "quality_tier": "medium",
            "use_for": ["review_scoring", "result_interpretation", "quick_tasks"]
        },
        "gpt-4o": {
            "provider": "openai",
            "input_cost_per_1k": 0.005,
            "output_cost_per_1k": 0.015,
            "max_tokens": 128000,
            "quality_tier": "high",
            "use_for": ["fallback", "comparison"]
        }
    }
    
    def route(self, task: dict) -> str:
        """
        根据任务特征和当前条件选择最佳模型
        """
        task_type = task["type"]
        complexity = task.get("complexity", "medium")
        urgency = task.get("urgency", "normal")
        budget_status = self._get_budget_status()
        
        # 预算紧张时降级
        if budget_status > 0.95:
            return "claude-haiku-4-5"
        
        # 根据任务类型路由
        if task_type == "persona_generation":
            return "claude-sonnet-4-6"
        elif task_type == "choice_simulation":
            return "claude-sonnet-4-6"
        elif task_type == "review_scoring":
            return "claude-haiku-4-5"
        elif task_type == "deep_analysis":
            return "claude-sonnet-4-6"
        else:
            return "claude-sonnet-4-6"  # 默认
    
    def switch_model(self, new_model: str, reason: str):
        """
        切换全局默认模型（用于版本升级或故障切换）
        """
        old_model = self.current_default
        self.current_default = new_model
        
        # 记录切换日志
        logger.info(f"模型切换: {old_model} -> {new_model}, 原因: {reason}")
        
        # 通知团队
        self._notify_team(f"LLM模型已切换至 {new_model}，原因: {reason}")
        
        # 更新监控标签
        self._update_metrics_label(new_model)
```

## 五、备份与灾难恢复

### 5.1 备份策略

| 数据类型 | 备份频率 | 保留期 | 存储位置 | 加密 |
|---------|---------|--------|---------|------|
| MongoDB数据 | 每日增量 + 每周全量 | 90天 | 对象存储 + 异地 | AES-256 |
| 配置文件 | 每次变更 | 永久 | Git + 对象存储 | — |
| 日志数据 | 实时归档 | 30天热存储 + 1年冷存储 | 日志服务 | TLS |
| 镜像文件 | 每次构建 | 保留最近10个版本 | 镜像仓库 | — |

### 5.2 灾难恢复流程

```
灾难发生
   │
   ▼
评估影响范围（5分钟内）
   │
   ├── 单服务故障 → 自动重启 / 流量切换
   │
   ├── 数据丢失 → 启动备份恢复
   │   • 确认备份完整性
   │   • 从对象存储恢复MongoDB
   │   • 验证数据一致性
   │
   ├── 全系统故障 → 启动灾难恢复预案
   │   • 切换至灾备环境（如有）
   │   • 或从Docker Compose快速重建
   │
   └── API密钥泄露 → 紧急轮换
       • 立即吊销泄露密钥
       • 生成新密钥
       • 滚动更新服务
       • 审计调用日志
   │
   ▼
恢复验证
   │
   ├── 健康检查通过
   ├── 核心功能验证（生成→模拟→分析）
   └── 数据一致性校验
   │
   ▼
通知团队与用户
   │
   ▼
事后复盘（24小时内）
   • 根因分析
   • 改进措施
   • 更新应急预案
```

## 六、扩容策略

> **当前状态**：生产环境运行在单台/多台 Azure VM 上，未使用 Kubernetes HPA。扩容通过提升 VM 规格或增加 VM 实例实现，具体策略由 `cd-azure-b2ats.yml` 和 `deploy-to-azure-worker.sh` 支撑。

### 6.1 扩容触发条件

| 指标 | 阈值 | 动作 | 最大规模 |
|------|------|------|---------|
| CPU利用率 | > 70% 持续5分钟 | 升级 VM 规格 / 增加 VM 实例 | 视预算 |
| 内存利用率 | > 80% 持续5分钟 | 升级 VM 规格 / 增加 VM 实例 | 视预算 |
| 队列长度 | > 50 持续10分钟 | Worker 容器扩容 | 20 Workers |
| API延迟P95 | > 5秒 持续5分钟 | 升级 VM 规格 | 视预算 |
| LLM API并发 | > 80%配额 | 启用备用模型 | — |

### 6.2 缩容策略

- CPU利用率 < 30% 持续30分钟 → 降级 VM 规格
- 队列长度 = 0 持续60分钟 → 缩容 Worker 容器
- 最小保留：API 1 容器，Worker 1 容器
