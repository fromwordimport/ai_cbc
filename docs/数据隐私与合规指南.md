# 数据隐私与合规指南

> **版本**：v1.0  
> **定位**：明确AI_CBC系统在数据收集、存储、处理、传输和销毁全生命周期中的隐私保护义务与合规要求  
> **负责人**：小伦（伦理/偏见审计与合规负责人）+ 小安（AI安全工程师）  
> **适用范围**：所有涉及虚拟消费者画像、问卷回答、分析结果及系统操作日志的处理活动  
> **合规依据**：《个人信息保护法》《数据安全法》、GDPR（如涉及跨境）

---

## 一、数据分类与敏感度定级

### 1.1 数据资产清单

| 数据类别 | 具体内容 | 敏感度 | 是否含个人信息 | 备注 |
|---------|---------|--------|-------------|------|
| **虚拟消费者画像** | Layer1-4 全层数据 | 中 | ⚠️ 含模拟人口统计信息 | 合成数据，但需防逆向关联 |
| **CBC问卷回答** | 选择记录、决策理由 | 低 | ❌ 不含 | 行为数据，无直接身份标识 |
| **分析结果** | 效用值、WTP、市场份额 | 中 | ❌ 不含 | 商业敏感，需访问控制 |
| **系统操作日志** | API调用、用户行为、审计记录 | 高 | ✅ 可能含操作者身份 | 含LLM交互内容，需脱敏 |
| **成本计费记录** | Token用量、费用明细 | 低 | ❌ 不含 | 仅含技术标识 |
| **业务用户账户** | 用户名、权限、组织架构 | 高 | ✅ 含真实个人信息 | 按企业IT规范管理 |

### 1.2 敏感度定义

- **高敏感度**：泄露/滥用将导致合规风险、商业机密泄露或个人权益受损
- **中敏感度**：泄露/滥用可能导致业务判断偏差或间接隐私风险
- **低敏感度**：公开披露不会对系统或用户造成实质损害

## 二、数据处理原则（Privacy by Design）

### 2.1 五大核心原则

| 原则 | 要求 | 实现方式 |
|------|------|---------|
| **最小必要** | 仅收集实现功能所必需的数据 | 画像生成不采集真实身份证号、手机号 |
| **目的限定** | 数据仅用于声明的研究目的 | 禁止将画像数据用于营销外呼 |
| **透明告知** | 用户知晓数据如何被使用 | 系统界面展示数据处理说明 |
| **安全保障** | 按敏感度实施分级保护 | 高敏感度数据加密存储、传输 |
| **限期留存** | 超期数据自动清理或匿名化 | 详见第4节数据保留策略 |

### 2.2 虚拟消费者数据的特殊性质

> **关键判定**：AI_CBC生成的虚拟消费者画像属于**合成数据（Synthetic Data）**，不对应真实自然人。但为防止"拟像关联"风险（即虚拟画像与真实个体高度相似导致可识别），仍按中敏感度管理。

**拟像关联防护措施**：
1. 画像生成时禁止输入任何真实个人可识别信息（PII）作为种子
2. 定期抽样比对：检查虚拟画像与公开数据库（如社交网络）的相似度
3. 发现高度相似案例（≥5个属性完全匹配真实公众人物）→ 自动触发重生成

## 三、数据主体权利响应

### 3.1 权利映射

尽管虚拟消费者非真实个人，但系统操作涉及**真实业务用户**（研究人员、管理员）和**可能的真实受访者对照数据**，需支持以下权利响应：

| 权利 | 适用对象 | 响应机制 | 时限 |
|------|---------|---------|------|
| **知情权** | 业务用户 | 系统界面展示隐私政策摘要 | 首次登录时 |
| **访问权** | 业务用户 | 提供个人操作历史、数据使用记录 | 15个工作日内 |
| **更正权** | 业务用户 | 支持修改账户信息、研究配置 | 即时 |
| **删除权** | 业务用户 | 支持删除账户及关联研究数据 | 15个工作日内 |
| **导出权** | 业务用户 | 提供研究数据的标准格式导出（JSON/CSV） | 即时 |
| **限制处理权** | 业务用户 | 支持暂停特定研究的数据处理 | 即时 |

### 3.2 删除权执行流程

```
用户提交删除请求
   │
   ▼
身份验证（多因素认证）
   │
   ▼
评估删除范围
   ├── 仅删除账户 → 保留匿名化操作日志（合规要求）
   ├── 删除账户+研究数据 → 级联删除该用户创建的全部研究
   └── 删除特定研究 → 仅删除指定研究及其衍生数据
   │
   ▼
执行删除（软删除30天后物理删除）
   │
   ▼
发送确认通知
   │
   ▼
审计记录（记录删除事件，保留1年）
```

## 四、数据保留与销毁策略

### 4.1 保留期限表

| 数据类别 | 保留期限 | 到期处置 | 例外情况 |
|---------|---------|---------|---------|
| 虚拟消费者画像（原始） | 研究完成后 **90天** | 自动匿名化后归档 | 用户主动导出备份 |
| CBC问卷回答（原始） | 研究完成后 **90天** | 自动删除 | 用户主动导出备份 |
| 分析结果（报告+数据） | 研究完成后 **1年** | 转冷存储，到期删除 | 用户标记为"长期归档" |
| 系统操作日志 | **热存储30天** + **冷存储1年** | 到期自动清除 | 安全事件调查可延长 |
| 审计日志 | **3年** | 到期评估后删除 | 监管调查期间冻结 |
| 成本计费记录 | **2年** | 到期匿名化后归档 | 财务审计需延长 |
| 用户账户信息 | 账户存续期 + **注销后1年** | 到期删除 | 法律诉讼期间冻结 |

### 4.2 自动清理机制

```python
# config/data_retention.py
DATA_RETENTION_POLICY = {
    "persona_profiles": {
        "retention_days": 90,
        "action": "anonymize_and_archive",
        "anonymization": {
            "remove_fields": ["language_samples", "dishwasher_context.purchase_constraints"],
            "generalize_fields": {
                "age": "decade_only",  # 28岁 → 20-30岁
                "income": "range_only",  # 15K-25K → 10K-30K
            }
        }
    },
    "choice_responses": {
        "retention_days": 90,
        "action": "delete",
        "cascade": True  # 级联删除关联的审校记录
    },
    "analysis_results": {
        "retention_days": 365,
        "action": "cold_storage_then_delete",
        "cold_storage_after": 90  # 90天后转冷存储
    },
    "system_logs": {
        "hot_retention_days": 30,
        "cold_retention_days": 365,
        "action": "tiered_cleanup",
        "sensitive_content_masking": True  # 日志中PII自动打码
    }
}
```

## 五、数据安全措施

### 5.1 存储安全

| 措施 | 适用数据 | 实现方式 |
|------|---------|---------|
| **静态加密** | 高敏感度数据 | AES-256-GCM，密钥由Vault管理 |
| **传输加密** | 全部数据 | TLS 1.3，强制HTTPS |
| **字段级加密** | API密钥、用户密码 | 单独加密存储，支持轮换 |
| **备份加密** | 全部备份 | 备份文件加密，异地存储 |

### 5.2 访问控制

```yaml
access_control:
  # 基于角色的访问控制（RBAC）
  roles:
    researcher:
      can_access: [own_studies, own_personas, own_results]
      cannot_access: [others_data, system_logs, admin_panel]
      
    analyst:
      can_access: [all_studies, all_results, reports]
      cannot_access: [system_logs, user_management]
      
    admin:
      can_access: [all_data, system_logs, config]
      requires: [mfa, audit_approval]
      
    auditor:
      can_access: [logs, audit_records, bias_reports]
      write_permission: false  # 只读
```

### 5.3 跨境数据传输

若系统部署涉及跨境（如LLM API调用至境外服务商）：

1. **事前评估**：进行数据出境安全评估，形成《数据出境风险自评估报告》
2. **合同约束**：与境外LLM服务商签署标准合同条款（SCCs），明确数据用途限制
3. **最小化传输**：仅传输必要的Prompt内容，禁止传输真实个人身份信息
4. **本地优先**：优先使用本地化部署的模型或境内服务商

## 六、隐私影响评估（PIA）

### 6.1 触发条件

以下变更需重新执行隐私影响评估：
- 新增数据收集字段
- 变更数据保留期限
- 引入新的第三方数据处理者
- 数据用途超出原始声明范围
- 发生数据泄露事件

### 6.2 PIA检查清单

```markdown
□ 数据收集是否最小必要？
□ 是否已告知数据主体处理目的？
□ 数据保留期限是否合理且有到期清理机制？
□ 第三方处理者是否有充分的安全保障？
□ 数据主体权利响应机制是否到位？
□ 跨境传输是否合规？
□ 发生泄露时是否有应急预案？
```

## 七、事件响应与违规处置

### 7.1 数据泄露响应分级

| 级别 | 定义 | 响应时间 | 通知义务 |
|------|------|---------|---------|
| **P0（重大）** | 真实用户PII泄露 ≥ 100条 | 2小时 | 72小时内通知监管机构和受影响用户 |
| **P1（严重）** | 真实用户PII泄露 < 100条，或分析结果大规模泄露 | 4小时 | 评估后决定是否通知 |
| **P2（一般）** | 系统日志/操作记录未授权访问 | 24小时 | 内部通报 |
| **P3（轻微）** | 单条数据异常暴露，已及时阻断 | 72小时 | 记录备查 |

### 7.2 响应流程

```
发现/报告数据安全事件
   │
   ▼
立即止损（隔离、阻断、冻结）
   │
   ▼
评估影响范围和数据类型
   │
   ▼
判定级别（P0-P3）
   │
   ▼
按级别响应
   ├── P0 → 立即通知小P、小伦、小安 → 启动监管通报流程
   ├── P1 → 通知小P、小安 → 准备事件报告
   ├── P2 → 通知小安 → 修复并记录
   └── P3 → 记录并纳入月度安全报告
   │
   ▼
事后复盘与改进
```
