## 二、跨模块系统性缺陷模式

通过对四个模块的并行审查，识别出 **5 个跨模块系统性模式**：

### 模式 1：「设计文档 → 代码」鸿沟

以下在设计文档中有完整定义（含伪代码/接口契约）但代码中**零实现**的组件：

| 缺失组件 | 定义位置 | 影响模块 |
|---------|---------|---------|
| `ReportBuilder` 类（7章报告生成器） | `01-架构.md` §8.1 | 重要性看板、市场模拟 |
| `CBCVisualizer` 类（4种可视化方法） | `01-架构.md` §8.2 | 全部三个分析模块 |
| `LatentClassModel` 类（自动分段发现） | `01-架构.md` §4.4 | 细分对比 |
| `NLScenarioParser` 类（自然语言→情景） | `01-架构.md` §6.2 | 市场模拟 |
| 画像资产管理 7 大能力（版本/去重/血缘/权限/归档） | `12-画像资产化管理规范.md` | 画像管理 |

**根因**：设计文档按"应有架构"编写，代码按"MVP 最小可用"实现，两者之间缺乏显式的优先级排序和缺口追踪。

### 模式 2：`by_segment` 空洞模式

两个独立的 API 响应模型声明了 `by_segment` 字段但**从不填充**：

```
ImportanceResponse.by_segment: dict | None  →  始终为 None  (tasks.py 不计算)
MarketSimResponse.by_segment: dict | None   →  始终为 None  (segment_filter 空操作)
```

**根因**：API 设计文档承诺了分段能力，但 Celery 任务和引擎层未实现分段计算逻辑。用户看到 API 文档中有此字段但调用永远返回 null。

### 模式 3：品类绑定模式

整个项目当前为**洗碗机单一场景**设计，跨品类复用需大量改动：

| 绑定点 | 文件 | 绑定方式 |
|--------|------|---------|
| `DishwasherContext` | `persona.py:77-82` | 画像模型中硬编码洗碗机购买情境 |
| `ProductScenario` | `models.py:145-154` | 市场模拟中硬编码 7 个洗碗机属性 |
| API 路由场景转换 | `routes.py:292-303` | 手动硬编码字段名 `price/capacity/...` |
| 前端属性下拉框 | `MarketSimulator.tsx:51-57` | `DEFAULT_LEVELS` 硬编码洗碗机水平值 |

**根因**：项目起步于洗碗机 CBC 实验，MVP 阶段未做品类抽象。通用化需要将 `DishwasherContext` → `PurchaseContext`，`ProductScenario` → 动态 `attributes` 方案。

### 模式 4：`segment_filter` 全链路空转

参数从 API 定义到引擎执行的全链路中在三处断裂：

```
Pydantic Field  →  路由未传递    →  引擎 pass    →  响应不填充  →  前端无UI
models.py       routes.py        simulator.py    models.py      (无组件)
✅ 已定义       ❌ 被丢弃        ❌ 空操作       ❌ 永为None     ❌ 不存在
```

**影响模块**：市场模拟（`MarketSimRequest.segment_filter`）、细分对比（`segment_a`/`segment_b` 可用但 LatentClass 自动分段不可用）

### 模式 5：前后端类型不一致

前端使用 mock 数据独立开发，未与真实后端 API 联调验证：

| 前端字段 | 后端字段 | 模块 |
|---------|---------|------|
| `comparisons: SegmentComparisonItem[]` | `per_attribute: list[PerAttributeTest]` | 细分对比 |
| `importance.overall` 仅用均值 | 后端返回含 CI 的 `ImportanceStats` | 重要性看板 |
| Mock 数据格式（`comparisons`） | 真实 API 格式（`per_attribute`） | 细分对比 |

## 五、审查方法论

四轮审查均采用**多层并行 Agent 审查法**：

```
审查目标确定 → 设计文档研读 → 并行 Agent 分层审查 → 跨层生命周期验证 → 汇总报告
                                  ├─ Agent-1: 数据/引擎层
                                  ├─ Agent-2: API/路由层
                                  ├─ Agent-3: 存储/模型层
                                  ├─ Agent-4: 集成/看板层 (可选)
                                  └─ Agent-5: 跨层验证 (可选)
```

**统计**：17 个 Agent，总 Token 消耗 ~123 万，总耗时 ~3,117 秒。

## 六、后续建议

1. **建立设计文档 ↔ 代码缺口追踪表** — 将模式 1 中发现的 5+ 个缺失组件纳入项目 Roadmap
2. **前后端联调验证** — 修复模式 5 中的类型不一致，确保前端能对接真实 API
3. **品类抽象专题** — 将模式 3 中的全部绑定点统一抽象为通用方案
4. **`by_segment` 专项** — 一次性修复所有模块的分段数据流
5. **价格标准化专项** — 在预处理与后处理之间建立量纲恢复机制（重要性看板 P0-1 已定位根因）
