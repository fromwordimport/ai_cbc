# 创建新研究模块 — 审查报告

> **版本**：v1.0
> **日期**：2026-06-12
> **审查方法**：端到端流程审查（前端创建UI → API → 生成器 → 存储 → 问卷生成 → 下游衔接）
> **审查范围**：前端 StudyCreate 页面、API 端点、生成器、存储层、端到端数据流

---

## 执行摘要

对 AI_CBC "创建新研究"模块进行端到端审查——从用户在前端点击"创建研究"到问卷生成完成的完整流程。

**整体评级：D（MVP 最小流程可用但配置能力为零，两段式创建存在事务性缺陷）**

| 维度 | P0-CRITICAL | P1-MAJOR | P2-MINOR | P3-SUGGESTION |
|------|-------------|----------|----------|---------------|
| 前端创建UI | 2 | 2 | 3 | 3 |
| 后端API/生成 | 2 | 3 | 2 | 2 |
| 端到端流程 | 1 | 2 | — | — |
| **合计** | **5** | **7** | **5** | **5** |

**核心结论**：创建研究的"快乐路径"（一键创建洗碗机研究 + 自动生成问卷）可以工作。但用户完全无法自定义属性和实验参数——只能获得硬编码的洗碗机配置。两段式创建（POST create → POST generate）之间存在事务性缺口：问卷生成失败后，已创建的研究成为孤儿状态（INIT），无法回滚。

> **🔧 修复状态（2026-06-12）**：2 个后端 P0 已由问卷配置管理修复解决；新增 PUT 更新端点 + StudyDetailResponse 补全 attributes；前端 P0 待后续修复

## 第一部分：前端创建 UI

### P0-CRITICAL

- ⏳ **待修复 (前端)** **[P0-UI-001]** `StudyCreate.tsx:24-38` — **创建→生成两段式流程无事务保护**。前端依次调用 `createStudy()` 和 `generateQuestionnaire()`。如果步骤 1 成功但步骤 2 失败（如设计算法崩溃、服务端错误），研究已创建但无问卷，处于 INIT 状态。前端在 catch 块中显示错误消息但没有回滚步骤 1。后端也没有孤儿研究清理机制。

- ⏳ **待修复 (前端)** **[P0-UI-002]** `StudyCreate.tsx:53-93` — **前端无法配置属性和实验参数**。表单仅 4 个字段（study_id, product_category, research_goal, target_segments），没有 attributes、design_parameters、prohibited_pairs。用户只能接受默认洗碗机配置。这是"品类绑定"系统缺陷在前端层的体现。

### P1-MAJOR

- **[P1-UI-001]** `StudyCreate.tsx:83-92` — **目标人群选项硬编码且与设计文档不一致**。下拉选项包括"一线城市年轻家庭""独居青年""银发族"等，与设计文档定义的 3 个预定义群体（精致白领/新手宝妈/Z世代租客）不一致。且这些标签在整个系统中没有对应的筛选逻辑——它们只作为字符串存储，不会实际影响画像生成或问卷分配。

- **[P1-UI-002]** `StudyCreate.tsx:31-36` — **创建成功后立即生成问卷，无中间确认步骤**。用户无法在创建研究和生成问卷之间检查和调整配置。如果自动生成的问卷有问题，用户必须先删除整个研究再重新创建。

### P2-MINOR

- **[P2-UI-001]** `StudyCreate.tsx:57` — **product_category 默认值硬编码为"洗碗机"**。使用 `initialValues={{ product_category: '洗碗机' }}`。如果用户忘记修改，会创建类别错误的研究。

- **[P2-UI-002]** `StudyCreate.tsx:62` — **前端 study_id 有 pattern 校验，但后端没有**。前端正则 `^[a-zA-Z0-9_-]+$` 正确，但这属于"前端独有验证"——恶意请求绕过前端直接调 API 可以传入任意 study_id。

- **[P2-UI-003]** `StudyCreate.tsx:38` — **创建后直接导航到首页 '/' 而非研究详情页**。用户不知道接下来该做什么——应该导航到研究详情或问卷查看页面。

### P3-SUGGESTION

- 建议增加创建进度指示器（创建中 → 生成问卷中 → 完成），而非仅 loading spinner
- 建议在研究创建后展示配置摘要并让用户确认后再生成问卷
- 建议 product_category 改为下拉选择（从已知品类列表中选），减少输入错误

## 第二部分：后端 API/生成器

### P0-CRITICAL

- ✅ **已修复（上一轮）** **[P0-API-001]** `questionnaires.py:37-48` — **`create_study` 不传递 attributes/design_parameters**。路由仅传递 4 个参数给 `generator.create_study()`，而生成器签名支持 7 个参数。用户即使通过 API 传入自定义属性也会被丢弃。这是"品类绑定"系统缺陷在后端 API 层的根源。

- ✅ **已修复（上一轮）** **[P0-API-002]** `questionnaires.py` 全文 — **study_id 未经 sanitize 直接用作 dict key**。对比 `personas.py` 调用了 `sanitize_id()`、`responses.py` 也调用了 `sanitize_id()`，问卷路由完全没有消毒保护。恶意 study_id 可注入到 store 的 dict key。

### P1-MAJOR

- ✅ **已修复** **[P1-API-001]** — **缺少 `PUT /studies/{study_id}` 更新端点**。创建后无法修改任何配置（属性、设计参数、研究目标）。用户发现配置错误后只能 DELETE 重建。

- ✅ **已修复（上一轮）** **[P1-API-002]** `store.py:28-31` — **`save_study()` 静默 upsert**。重复提交相同 study_id 不返回 409 Conflict，而是静默覆盖已有研究。前端无法区分"创建新研究"和"覆盖已有研究"。

- ✅ **已修复** **[P1-API-003]** `StudyDetailResponse` — **不包含完整 attributes 定义**。仅返回 `n_attributes` 整数和设计参数摘要，客户端无法获取属性名称、类型和水平值。

### P2-MINOR

- `questionnaires.py:160` — **状态直接设为 READY 无状态转换校验**。不检查当前状态是否允许此转换（如从 COMPLETED → READY 应被阻止）

- `generator.py:94-121` — **`create_study()` 命名有误导性**。方法只创建 Pydantic 对象（未持久化），但名称暗示会执行完整的创建流程

### P3-SUGGESTION

- 建议 `CreateStudyRequest` 扩展 `attributes` 和 `design_parameters` 可选字段
- 建议添加 `POST /studies/{study_id}/clone` 克隆已有研究

## 第三部分：端到端流程分析

### 完整用户路径

```
用户填写表单 ──→ POST /studies ──→ QuestionnaireGenerator.create_study() 
                                          ↓
                                   CBCStudy(attributes=洗碗机默认)
                                          ↓
                                   store.save_study() → 201 Created
                                          ↓
                              POST /studies/{id}/generate
                                          ↓
                              QuestionnaireGenerator.generate_questionnaire()
                                          ↓
                              D-Optimal / Balanced 算法
                                          ↓
                              QuestionnaireValidator.validate()
                                          ↓
                              store.save_questionnaire() + status→READY
                                          ↓
                              返回 GenerateQuestionnaireResponse
                                          ↓
                              导航到首页 '/'
```

### 流程断点

| 断点 | 场景 | 影响 |
|------|------|------|
| **断点 1** | createStudy 成功 → generateQuestionnaire 失败 | 孤儿研究（INIT 状态，无问卷） |
| **断点 2** | 生成成功但 D-efficiency < 0.5 | 低质量问卷入库，validator 阈值过低 |
| **断点 3** | 创建后直接导航到 '/' | 用户不知道接下来要做什么 |
| **断点 4** | 无属性/参数配置环节 | 创建后无法修改，只能删除重建 |

### 与其他模块的衔接

| 下游模块 | 衔接状态 | 说明 |
|---------|---------|------|
| 消费者模拟 | ✅ 正常 | 问卷生成后可调用 `POST /simulate-responses` |
| 数据分析 | ✅ 正常 | 问卷生成后可调用 `POST /analyze` |
| 市场模拟 | ✅ 正常 | 分析完成后可调用 `POST /simulate-market` |
| 问卷导出 | ⚠️ 简化 | 导出端点存在但缺失格式/编码选项 |

## 第四部分：修复优先级

### 🔴 第一批：阻断性

| 优先级 | 问题 | 修复难度 |
|--------|------|---------|
| 1 | study_id sanitize（后端所有 5 个端点） | 低 |
| 2 | study_id 后端 Pydantic 约束（min_length/max_length/pattern） | 低 |
| 3 | PUT/PATCH 更新端点（让用户可以修改配置） | 中 |
| 4 | 两段式事务保护（生成失败时提示用户而非静默孤儿） | 中 |

### 🟠 第二批：功能缺口

| 优先级 | 问题 | 修复难度 |
|--------|------|---------|
| 5 | API 和前端支持自定义 attributes/design_parameters | 高 |
| 6 | 前端增加属性配置界面（动态添加/编辑属性+水平） | 高 |
| 7 | StudyDetailResponse 包含完整 attributes | 低 |
| 8 | 创建→生成之间增加确认步骤 | 中 |
