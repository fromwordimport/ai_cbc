# 总览模块 — 审查报告

> **版本**：v1.0
> **日期**：2026-06-12
> **审查方法**：前后端全链路审查（Dashboard / Layout / Router / 后端聚合端点）
> **审查范围**：`Dashboard.tsx`、`Layout.tsx`、`router.tsx`、`appStore`、后端 `getStudies`/`getPersonas` API

---

## 执行摘要

对 AI_CBC "总览"模块进行了前端 Dashboard 页面、Layout 导航框架、路由体系的全链路审计。

**整体评级：C-（骨架完整，但数据不实时 + 状态映射不一致 + 统计信息过度精简）**

| 维度 | P0 | P1 | P2 | P3 |
|------|----|----|----|-----| 
| 总览 Dashboard | 0 | 3 | 4 | 2 |
| 导航 Layout/Router | 0 | 1 | 1 | 2 |
| 后端聚合 | 0 | 2 | 0 | 1 |
| **合计** | **0** | **6** | **5** | **5** |

**核心结论**：总览页面骨架合理——4 个统计卡片 + 研究列表 + 侧栏导航 + 头部任务计数，功能布局清晰。但数据是一次性加载无自动刷新、状态标签映射与后端不一致（`DRAFT`/`RUNNING` 永远不会出现）、缺少成本/告警/最近活动等关键总览信息，是一个纯前端页面（无后端聚合端点）。

## 第一部分：各维发现

### P0-CRITICAL

无。总览模块是纯信息展示页，不存在数据丢失或安全漏洞级别的阻断性问题。

### P1-MAJOR（6项）

- **[P1-OV-001]** `Dashboard.tsx:64-71` — **状态 Tag 映射与后端不一致**。前端 `colorMap` 定义了 `DRAFT/RUNNING/FAILED` 等状态，但后端 `CBCStudy.status` 的 `StudyStatus` 枚举为 `INIT/DESIGNING/READY/COMPLETED`。`DRAFT`、`RUNNING`、`FAILED` 状态在该表中永远不会出现（后者属于分析任务的状态枚举）。正确的映射应为 `INIT→default, DESIGNING→processing, READY→success, COMPLETED→default`。

- **[P1-OV-002]** `Dashboard.tsx:25-39` — **总览数据无自动刷新**。页面在 `useEffect` 中一次性拉取 studies 和 personas 数据，之后不再更新。用户从其他页面（如创建研究后导航回来）回到 Dashboard 时数据已过期。`useEffect` 依赖 `[setStudies]` 而非定时轮询或路由焦点事件。

- **[P1-OV-003]** `Dashboard.tsx:28` — **persona 计数只取 page_size=1**。`getPersonas(1, 1)` 仅获取第 1 页 1 条数据，依赖 `res.total` 获取总数。虽可工作，但如果服务端 `PersonaStore.list_all()` 有 bug（如返回空结果但 total 不正确），计数会误导用户。

- **[P1-OV-004]** `Dashboard.tsx:81` — **"查看分析"按钮直接导航到重要性看板**。`navigate(`/importance?study=${record.study_id}`)` 跳转到 ImportanceDashboard 页面，但该页面还需要 `analysis_id` 才能查看分析结果。仅仅携带 `study_id` 意味着用户需要在该页面上手动选择分析任务——这不直观。

- **[P1-OV-005]** — **缺少后端 Dashboard 聚合 API**。总览页面的 4 个统计指标分别来自 `/studies` 和 `/personas` 两个独立 API 调用。缺少 `GET /dashboard/summary` 端点一次性返回：研究总数、画像总数、待处理任务数、近 7 天活动趋势、成本摘要等聚合信息。

- **[P1-OV-006]** `Layout.tsx:126-130` — **头部 `runningJobs` Badge 来源不可靠**。`runningJobs` 来自 `useAppStore`，仅由 `ImportanceDashboard` 和 `AnalysisStatus` 页面的 poll 逻辑填充。用户在总览页不访问分析相关页面时，Badge 始终为 0——不管后端实际有多少任务在运行。

### P2-MINOR（5项）

- **[P2-OV-001]** `Dashboard.tsx:106` — **"研究项目"卡片显示 `studies.length`**。这是当前内存中的研究数量（取决于 store 中加载了多少页），而非后端实际总数。`StudyListResponse.total` 被丢弃了。

- **[P2-OV-002]** `Dashboard.tsx:123` — **"问卷设计"卡片逻辑问题**。`studies.filter(s => s.status === 'READY').length` 仅统计 READY 状态。`COMPLETED` 状态的研究同样有问卷（可能已完成分析），应被计入。但前端 `StudySummary` 不含 `status` 字段——需要确认 `status` 是否从后端返回。

- **[P2-OV-003]** `Dashboard.tsx:141` — **"研究项目列表"卡片的 extra 按钮固定为"属性重要性看板"**。与 Dashboard 总览页定位不符——建议改为"创建新研究"。

- **[P2-OV-004]** `Layout.tsx:37-72` — **侧栏有 11 个菜单项但总览缺少核心信息**。总览为入口页但未见：系统健康状态指示、最近创建的研究（按时间排序）、成本今日摘要、最新分析结果摘要。侧栏导航完整但 Dashboard 内容密度低。

- **[P2-OV-005]** `Dashboard.tsx:88` — **表格不显示研究创建时间**。`created_at` 字段在 `StudySummary` 中存在但表格未展示。

### P3-SUGGESTION（5项）

- 建议 Dashboard 增加自动轮询（30-60 秒间隔），或至少页面焦点恢复时刷新
- 建议研究列表默认按创建时间倒序排列
- 建议总览页增加"最近活动"时间线（创建研究 → 生成画像 → 生成问卷 → 分析完成的事件流）
- 建议头部 Badge 增加点击跳转到分析任务状态页
- 建议后端实现 `GET /dashboard/summary` 聚合端点

## 第二部分：路由体系评估

### 路由结构

```
/ (Dashboard)                    ← 总览入口
├── /studies/new                 ← 创建研究
├── /personas                    ← 画像管理
├── /questionnaires              ← 问卷配置
├── /interview                   ← 对话实验室
├── /importance                  ← 属性重要性看板
├── /segment-comparison          ← 细分群体比较
├── /market-simulator            ← 市场份额模拟器
├── /analysis-status             ← 分析任务状态
└── /settings                    ← 系统设置
```

**评价**：路由结构清晰，12 个页面通过 lazy loading + Suspense 按需加载。侧栏导航完整覆盖所有核心功能模块。

**问题**：嵌套路由（`/studies/:studyId/questionnaire`、`/studies/:studyId/responses`、`/personas/:personaId`）在侧栏无高亮——导航到详情页时侧栏菜单项全部熄灭，导航上下文丢失。
