# 市场份额模拟器模块 — 审查报告

> **版本**：v1.0
> **日期**：2026-06-12
> **审查方法**：3层并行审查（模拟引擎 / API+模型+存储 / 集成+看板+可视化）
> **审查范围**：3份设计文档 + 10个代码文件 + 前端 MarketSimulator 页面

---

## 执行摘要

对 AI_CBC "市场份额模拟器"模块进行了从核心算法 → API/存储 → 前端看板的全链路审计。

**整体评级：C（引擎算法正确但 API 层存在品类绑定和全链路空转问题）**

| 审查层 | P0-CRITICAL | P1-MAJOR | P2-MINOR | P3-SUGGESTION |
|--------|-------------|----------|----------|---------------|
| 模拟引擎层 | 2 | 5 | 5 | 4 |
| API/模型/存储层 | 4 | 3 | 9 | 4 |
| 集成/看板/可视化层 | 4 | 6 | 5 | 5 |
| **合计（去重后）** | **~7** | **~12** | **~18** | **~13** |

**核心结论**：好消息是——模拟引擎的数学是正确的，且**价格标准化在模拟层是自洽的**（两端使用相同的 `effects_encode_price()` 函数，不存在重要性看板中发现的那种量纲断裂）。但坏消息是——**API 层硬编码洗碗机属性导致模块无法跨品类复用**，`segment_filter`/`by_segment` 从 API 定义到引擎执行再到前端展示**全链路空转**，设计文档定位为核心组件的 `NLScenarioParser` **完全缺失**。

## 第一部分：各层关键发现

### Layer 1 — 模拟引擎层

**评级：B-**（算法正确，但缺少防御性校验）

#### P0-CRITICAL（2项）

- **[P0-ENG-001]** `market_simulator.py:21-24, 89-92` — **属性顺序未校验**。`util.values @ scenario_matrix.T` 的矩阵乘法依赖 `util.columns` 的顺序与 `encode_profile(self.attributes)` 的输出列顺序完全一致。**没有任何验证**确保两者一致。如果顺序不同（如 HB 模型传入的 `_feature_cols` 顺序与 attributes 不同），price 系数会乘以 brand 编码值，brand 系数会乘以 price z-score，**静默产生完全错误的效用计算**。

- **[P0-ENG-002]** `market_simulator.py:75-92` — **docstring 公式与实际代码不一致**。注释声称 `U = X @ beta.T`（维度 n_scenarios × n_resp），实际代码执行 `util.values @ scenario_matrix.T`（维度 n_resp × n_scenarios）。互为转置，但公式标注与实际运算不同，误导开发者。

#### P1-MAJOR（5项）

- `segment_filter` 参数是空操作——`_compute_utilities()` 中实现为一个 `pass`
- `sensitivity_analysis` 缺少设计文档要求的 `share_change` 列
- `none_option` 效用固定为 0，缺少根据观测"都不选"率的校准机制
- 输出缺少设计文档要求的 `share_std` 列
- `sensitivity_analysis` 多次调用 `simulate_share` 重复编码，性能浪费

#### 量纲一致性验证 ✅

**价格标准化在模拟层是自洽的：**

| 环节 | 编码方式 | 量纲 |
|------|---------|------|
| HB 训练数据 | `effects_encode_price()` = z-score | 标准化 |
| HB 输出系数 | `beta_price` 在 z-score 尺度 | 标准化 |
| 情景矩阵构建 | 同样 `effects_encode_price()` | z-score |
| **结论** | **两端使用相同函数 + 相同 attributes** | **自洽** |

**但设计文档 6.1 节伪代码对价格使用原始值（`row.append(value)`），与实际实现的 z-score 编码不一致——设计文档应更新。**

### Layer 2 — API/模型/存储层

**评级：C-**（品类绑定是核心问题）

#### P0-CRITICAL（4项）

- **[P0-API-001]** `models.py:154-163` + `routes.py:317-328` — **`ProductScenario` 硬编码洗碗机 7 个属性**。`capacity`/`installation`/`features`/`brand`/`energy` 对手机/汽车/护肤品完全不适用。API 端点**只能在洗碗机研究中工作**。

- **[P0-API-002]** `routes.py:309-313` — **空效用矩阵导致未处理异常**。`individual_utilities` 为空 dict 时创建 (0,0) DataFrame，`@ scenario_matrix.T` 维度不匹配抛 500。

- **[P0-API-003]** `routes.py:317-328` — **情景属性值未验证**。`encode_profile()` 抛 `ValueError` 时（属性缺失/水平不匹配）未被路由捕获，直接 500。

- **[P0-API-004]** 同 P0-API-001 根源——**情景字段名必须与 Attribute.id 完全一致**，但无验证机制。

#### P1-MAJOR（3项）

- `by_segment` 永不为 None——`segment_filter` 从请求到执行全线空转
- `NLScenarioParser` 完全缺失——设计文档的"自然语言→结构情景"桥梁不存在
- `segment_filter` 参数被接受但无功能效果（应返回 501 或实现）

#### P2-MINOR（9项）

- 手动 dict 转换应用 `model_dump()` 替代
- `ScenarioShare` 缺少 `share_std` 字段
- 缺少 `list_market_sims()` 方法
- 缺少 GET 端点查询已保存的模拟结果
- `sim_id` 不返回给客户端——用户无法追踪
- `uuid.hex[:8]` 碰撞风险（32 位空间）
- 无缓存策略
- `_verify_study_ownership` 冗余查询
- 设计文档 `share_std` 与实际代码不一致

### Layer 3 — 集成/看板/可视化层

**评级：C-**（数据流断裂 + 关键组件缺失）

#### P0-CRITICAL（4项）

#### P1-MAJOR（6项）

- `_generate_report()` 完全缺少市场模拟章节

- `CBCVisualizer.plot_market_simulation()` 未实现（Python 端无可视化）
- `ReportBuilder` 类完全未实现（含 `_build_market_sim_section()`）
- `simulation/__init__.py` 为空文件

#### P2-MINOR（5项）

- 前端柱状图缺少误差线（CI 仅在 tooltip 中）
- 前端缺少 `segment_filter` 和 `by_segment` 交互
- 前端缺少敏感度分析交互（引擎已实现但无 API 端点和 UI）
- 前端属性水平下拉框使用硬编码默认值（洗碗机）
- 列名 `"name"` vs 设计文档 `"scenario"` 不一致

## 第二部分：跨层断裂分析

### 断裂 #1：`segment_filter` / `by_segment` 全链路空转

```
API定义          路由传递          引擎执行        响应填充        前端展示
segment_filter → 被丢弃      →    pass空操作  →   by_segment=None → 无UI
(models.py:172)  (routes.py)     (simulator:83)   (models.py:188)   (无组件)
```

**这是与 `ImportanceResponse.by_segment` 相同的系统性缺陷模式——字段定义存在但不被任何代码路径填充。**

### 断裂 #2：品类绑定

```
ProductScenario  →  路由手动转换  →  encode_profile  →  情景矩阵
hardcoded 7字段    hardcoded 7键    通用编码器          待计算
(price/capacity/   (price/capacity/  (category-agnostic)
 installation/      installation/
 features/brand/    features/brand/
 energy)            energy)
                   ↑
              这两层硬编码使整个 API 只能在洗碗机研究中工作
```

### 断裂 #3：自然语言接口缺失

```
用户自然语言                      NLScenarioParser          结构化情景
"华为2999 vs 小米3999 vs 苹果4999"  →  [缺失]           →  [无法到达]
设计文档7.2节示例流程              设计文档6.2节定义       MarketSimulator
                                  代码中完全不存在
```
