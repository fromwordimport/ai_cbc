## 第一部分：各维发现

### P0-CRITICAL（1项）

- ⏳ **待修复 (前端)** **[P0-SET-001]** `Settings.tsx:39-41` — **前端设置仅保存到 localStorage，与后端完全隔离**。用户在前端修改 LLM 模型提供商、温度、预算等配置后点击"保存设置"，数据只写入浏览器 `localStorage`。后端 `Settings` 由环境变量/`.env` 文件驱动，完全不受前端影响。后端没有任何 `PUT /settings` API 端点。这意味着：(1) 前端设置页面是装饰性的；(2) 用户换浏览器/设备后设置丢失；(3) 多人协作时设置不共享。

### P1-MAJOR（5项）

- **[P1-SET-001]** `Settings.tsx:19-23` — **前端默认 LLM 配置与后端完全脱节**。前端 `DEFAULT_LLM.model_provider='deepseek'`（提供商选择列表包含 DeepSeek/Qwen/GLM），但后端 `Settings` 仅配置了 `AnthropicSettings` 和 `OpenAISettings`（无 DeepSeek/Qwen/GLM）。前端和后端支持的模型集合交集为空。

- **[P1-SET-002]** `Settings.tsx:27-29` — **成本预算为前端本地值，不读取后端真实预算**。`DEFAULT_SYSTEM.cost_budget_daily=50`、`cost_budget_monthly=1000` 是硬编码的前端默认值。后端 `CostFuseSettings.daily_cny=1000`、`monthly_cny=20000` 是真实熔断阈值。前端显示和编辑的预算与后端熔断系统完全无关——即便前端设了 ¥50/天，后端仍在 ¥1000/天触发熔断。

- ✅ **已修复** **[P1-SET-003]** `settings.py:91-92` — **`cramers_v_threshold=0.1` 与设计文档不一致**。代码阈值 0.1，但设计文档 `虚拟消费者公平性规范.md` 第 2.1 节指定为 0.2（Cramér's V ≥ 0.2 触发 reject_and_regenerate）。0.1 比 0.2 更严格 2 倍，可能产生大量误报（社会科学中 V < 0.2 被视为弱关联）。此值在代码中从未被实际使用（Cramér's V 计算未实现），但仍需对齐。

- ✅ **已修复** **[P1-SET-004]** `settings.py:112-124` — **`SECRET_KEY` 有注释声明保留用途但全线未使用**。代码注释详细列出 JWT/CSRF/cookie 签名等未来计划，但 `secret_key` 当前没有任何代码消费。默认值 `dev-secret-change-in-prod` 在 production 环境下不变——validator 不对 production 做特殊要求。如果忘记修改，生产环境将使用开发密钥。

- ✅ **已修复** **[P1-SET-005]** — **前端缺少 `PUT /settings` 后端 API**。整个后端没有任何设置管理端点。用户无法通过 API 查询或修改 LLM 配置、成本阈值、认证设置。

### P2-MINOR（5项）

- **[P2-SET-001]** `settings.py:29` — **`model_audit: "claude-haiku-4-5"`** — 模型名称可能已过时。当前 Claude 模型系列为 Opus 4.7 / Sonnet 4.6 / Haiku 4.5，`claude-haiku-4-5` 需要确认是否是有效模型 ID。

- **[P2-SET-002]** `settings.py:87-88` — **`AuthenticitySettings` 硬编码阈值 9/12/14**。这些值在 `persona.py:117` 的 `Field(ge=0, le=14)` 和 `consumer-simulation/14-测试规范.md` 中一致，但如果在 Settings 中修改，需要同步更新 Pydantic 模型的 Field 约束。

- **[P2-SET-003]** `Settings.tsx:56-63` — **`getCostStatus` 和 `getHealthCheck` 失败时静默设为 null**。前端 `fetchStatus()` 对每个 API 调用使用 `.catch(() => null)`，两个都失败时才显示错误。一个成功一个失败时，用户看到半页数据，不知道另一半加载失败。

- **[P2-SET-004]** `Settings.tsx:73` — **30 秒轮询健康状态**。硬编码的 30 秒轮询用于健康检查，但对成本数据的实时性要求更高（熔断状态可能快速变化）。成本状态信息也 30 秒更新一次。

- **[P2-SET-005]** `settings.py:119-124` — **`SECRET_KEY` min_length=1 过于宽松**。最小长度 1 的密钥在安全上是无意义的。真正的密钥（JWT/CSRF/cookie 签名）至少需要 32 字符（256 位）。

### P3-SUGGESTION（4项）

- 建议后端实现 `GET/PUT /admin/settings` API 端点，前端通过 API 读写配置
- 建议前端 LLM 提供商列表从后端动态获取，而非硬编码
- 建议 `settings.py` 中未使用的配置添加 `deprecated` 标记或 TODO
- 建议增加设置变更审计日志（谁在何时修改了什么配置）

## 第二部分：后端配置体系评估

### 结构评分：A-

```
Settings (顶层, .env 驱动)
├── LLMSettings         (temperature, max_tokens, timeout, retries)
├── AnthropicSettings   (api_key, models × 3 用途)
├── OpenAISettings      (api_key, model)
├── DatabaseSettings    (MongoDB URL/DB/pool + Redis URL)
├── CostFuseSettings    (single/daily/weekly/monthly 预算 + degrade model)
├── StudySettings       (默认 n_choice_sets/n_alternatives/sample_size/d_efficiency)
├── AuthenticitySettings(pass/excellent/max 阈值)
├── BiasAuditSettings   (KS p-threshold, Cramér's V, entropy)
├── celery_broker_url   (Redis URL)
└── metrics/slow_request (监控路径 + 慢请求阈值)
```

**优点**：
- `pydantic-settings` 子配置类 + `env_prefix` 分层清晰
- `@lru_cache` 单例避免重复加载
- `is_production` 属性统一环境判断
- `extra="ignore"` 防止未声明环境变量污染配置

**缺点**：
- `SECRET_KEY` 预留但未使用（代码注释过度承诺）
- 部分配置与设计文档不一致（cramers_v_threshold）
- 无运行时热重载——修改 `.env` 需重启服务

## 第三部分：前端设置页面 vs 后端配置对比

| 配置项 | 前端 Settings.tsx | 后端 settings.py | 一致性 |
|--------|------------------|------------------|--------|
| LLM 提供商 | DeepSeek/OpenAI/Anthropic/Qwen/GLM | Anthropic/OpenAI 仅两类 | ❌ 不一致 |
| LLM 模型 | deepseek-chat/gpt-4o/claude-sonnet 等 | claude-sonnet-4-6/gpt-4o/claude-haiku | ⚠️ 部分重叠 |
| 温度 | Slider 0-2 | 0.3 default | ❌ 来源不同 |
| 日预算 | ¥50 default | ¥1000 default | ❌ 完全无关 |
| 月预算 | ¥1000 default | ¥20000 default | ❌ 完全无关 |
| 存储位置 | localStorage | .env 环境变量 | ❌ 隔离 |

**结论：前端 Settings.tsx 和后端 settings.py 是两个完全独立的系统，没有任何数据共享通路。**
