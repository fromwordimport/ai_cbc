# 分析任务状态模块 — 审查报告

> **版本**：v1.0
> **日期**：2026-06-12
> **审查方法**：2层并行审查（任务生命周期状态机 / API模型前端集成）
> **审查范围**：3份设计文档 + 10个代码文件 + 前端组件

---

## 执行摘要

对 AI_CBC "分析任务状态"模块进行了从 Celery 任务生命周期 → 状态机 → API → 前端轮询的全链路审计。

**整体评级：D（存在架构级缺陷——进程间存储隔离导致异步任务系统完全不可用）**

| 审查层 | P0-CRITICAL | P1-MAJOR | P2-MINOR | P3-SUGGESTION |
|--------|-------------|----------|----------|---------------|
| 任务生命周期与状态机 | 3 | 8 | 5 | 4 |
| API/模型/前端集成 | 3 | 8 | 9 | 4 |
| **合计（去重后）** | **5** | **~15** | **~13** | **~8** |

**核心结论**：模块的 API 设计和前端轮询逻辑是合理的，但存在一个**架构级缺陷**——`AnalysisStore` 是进程内内存存储，FastAPI 进程和 Celery worker 进程各自持有独立实例，**API 永远无法感知 Celery worker 的任务状态变更**。整个异步分析任务系统仅在开发 mock 模式下能工作，一旦接入真实 Celery 即完全失效。

---

## 第一部分：各层关键发现

### Layer 1 — 任务生命周期与状态机层

**评级：D（架构缺陷 + 代码 bug）**

#### P0-CRITICAL（3项）

- **[P0-LIFE-001]** `store.py:182-190` + `tasks.py:95` — **AnalysisStore 进程间隔离——架构级缺陷**。`AnalysisStore` 使用 `dict` + `threading.Lock` 实现，通过模块级 `get_analysis_store()` 单例访问。FastAPI 进程和每个 Celery worker 进程各自持有**完全独立**的 store 实例。API 的 `/status` 端点永远看不到 Celery 写入的状态变更（永远返回 QUEUED），`/analysis/{id}` 永远 404。**当前仅在 `dev_server_with_mocks.py`（完全绕过真实分析路由和 Celery）下能正常工作。**

- **[P0-LIFE-002]** `tasks.py:177` — **`price_std` NameError 导致所有分析任务必然崩溃**。第 177 行使用 `price_std=price_std` 调用 `compute_importance()`，但 `price_std` 变量在第 212-217 行才被定义。`NameError` 使任务在拟合完成后必抛异常，所有 HB/MNL 分析任务无法成功完成。

- **[P0-LIFE-003]** `tasks.py:43` — **硬超时 SIGKILL 无状态恢复**。`time_limit=600` 触发 SIGKILL 时进程立即死亡，不执行任何 Python 代码，`update_job_status("FAILED")` 不会被调用。job 永久停留在 RUNNING 成为僵尸。

#### P1-MAJOR（8项）

- `update_job_status()` 无状态转换合法性校验——COMPLETED→RUNNING 等非法转换未被阻止
- 任务幂等性缺失——无分布式锁、无 `max_retries`/`autoretry_for` 配置
- 结果持久化无事务性——7 个 `save_*()` 独立调用，中间失败产生僵尸数据
- HB 拟合期间进度不可观测——25%→75% 是一个巨大的盲区（占总耗时 80-95%）
- 模型不收敛仍标记 COMPLETED——违反 Must-Have 验收标准（R-hat < 1.1 100% 通过）
- 状态枚举缺少 CANCELLED 和 TIMED_OUT
- 异常处理覆盖不完整——`json.loads` 在 try 块之前
- 缺少 Celery 自动重试配置

### Layer 2 — API/模型/前端集成层

**评级：C-**

#### P0-CRITICAL（3项）

- **[P0-API-001]** `routes.py:170-183` — **运行中任务返回 404 而非 409**。`get_analysis_result` 直接查找结果存储，任务未完成时返回 404。前端收到 404 触发全局错误拦截器显示"资源不存在"。设计文档明确要求 `ANALYSIS_006 → 409 Conflict`。

- **[P0-API-002]** `api.ts:170` vs routes.py — **前端 `status` 类型遗漏 `QUEUED`**。后端返回 `QUEUED`，前端类型仅为 `PENDING|RUNNING|COMPLETED|FAILED`。代码、前端类型、设计文档三者相互矛盾。

- **[P0-API-003]** `ImportanceDashboard.tsx:229-230` — **完成时调用 `removeJob` 导致跨页面状态丢失**。分析完成后从共享 store 移除 job，用户切换到 AnalysisStatus 页面看不到已完成任务。

#### P1-MAJOR（8项）

- 缺少 `GET /studies/{study_id}/analyses` 列表端点
- 缺少取消/删除分析任务的端点（无 Celery `revoke` 支持）
- 设计文档 6 个错误码（ANALYSIS_001~006）**全部未实现**
- `latent_class` 被静默降级为 HB
- 前端 `SegmentComparisonResponse` 类型与后端完全不一致
- 前端 `ImportanceStats` 类型缺少 5 个字段
- 设计文档 `PENDING` vs 代码 `QUEUED` 状态不一致
- 无结果清理/淘汰机制

## 第二部分：架构缺陷详解

### 进程间存储隔离（P0-LIFE-001）

这是本次审查发现的**最严重问题**：

```
FastAPI 进程                    Celery Worker 进程
┌──────────────┐               ┌──────────────┐
│ AnalysisStore │               │ AnalysisStore │
│ _jobs: {     │               │ _jobs: {     │
│   analysis-1 │               │   (空)       │
│   status=QUEUED              │              │
│ }             │               │ }            │
│              │               │              │
│ POST/analyze │               │ tasks.py:    │
│   save_job() │               │   update(RUNNING)
│              │               │   save_result()
└──────────────┘               └──────────────┘
        │                              │
        └──────── 完全隔离 ─────────────┘
               无共享存储
```

**影响**：
- `GET /status` 永远返回 QUEUED
- `GET /analysis/{id}` 永远 404
- 前端轮询永远看不到任务进展
- 当前仅在 `dev_server_with_mocks.py` 下工作

**修复方向**：必须将 `AnalysisStore` 替换为外部存储（Redis/PostgreSQL/MongoDB），或使用 Celery 的 result backend（Redis `CELERY_RESULT_BACKEND`）作为中间层。

### 僵尸状态全集

| 僵尸场景 | 触发 | job 状态 | 用户视角 |
|---------|------|---------|---------|
| 硬超时 | HB > 600s, SIGKILL | RUNNING | 一直转圈 |
| 保存失败 | save_importance() 异常 | RUNNING | 部分数据可用 |
| 进程隔离 | Celery→独立store | QUEUED | 看不到任何进展 |
| 重传覆盖 | Worker A 崩溃, B 重执行 | RUNNING | 结果可能变化 |
| price_std bug | NameError | FAILED | 明明应该成功 |
