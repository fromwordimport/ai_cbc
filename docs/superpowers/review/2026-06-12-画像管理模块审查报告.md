# 虚拟消费者画像管理模块 — 全面审查报告

> **版本**：v1.0
> **日期**：2026-06-12
> **审查方法**：5层模块化分层审查 + 端到端生命周期追踪验证（6个并行Agent）
> **审查范围**：7份设计文档 + 17个代码文件 + 配置文件

---

## 执行摘要

对 AI_CBC "虚拟消费者画像管理"模块进行了全面审计，覆盖数据模型、生成引擎、存储/检索、API、安全/偏见/成本 5 个维度，并通过端到端生命周期追踪验证了跨层数据流完整性。

**整体评级：D（有条件通过，需重大修复）**

| 维度 | P0-CRITICAL | P1-MAJOR | P2-MINOR | P3-SUGGESTION |
|------|-------------|----------|----------|---------------|
| Layer 1 数据模型 | 2 | 5 | 8 | 9 |
| Layer 2 生成引擎 | 0 | 5 | 6 | 6 |
| Layer 3 存储/检索 | 0 | 7 | 4 | 5 |
| Layer 4 API 层 | 0 | 6 | 7 | 6 |
| Layer 5 安全/偏见/成本 | 2 | 5 | 5 | 5 |
| 生命周期追踪 | 7 (跨层) | 5 (跨层) | 1 (跨层) | — |
| **合计（去重后）** | **7** | **~28** | **~30** | **~31** |

**核心结论**：当前实现是一个质量良好的 MVP 骨架——CRUD 正确性、线程安全、LLM 安全防护（提示词泄露检测、成本熔断）均通过审查。但与设计文档之间存在系统性鸿沟：`12-画像资产化管理规范` 定义的 7 大管理能力中 6 项未实现，4层数据模型与 `02-画像生成` 设计规格严重不对齐，偏见审计管线仅实现了 1/4 步骤，且存在 7 个跨层数据断裂点。

## 第一部分：各层审查发现

### Layer 1 — 数据模型层

**评级：C**

#### P0-CRITICAL

- **[P0-DM-001]** `src/aicbc/api/schemas.py:98` — `PersonaSummary.life_stage` 被错误赋值为 `profile.segment`。`life_stage`（人生阶段，如"初入职场单身"）和 `segment`（细分群体，如"初入职场单身-一线城市"）是两个不同概念，被强制等同。下游任何依赖 `life_stage` 的过滤/分析/展示功能均会得到错误数据。
  - **修复**：在 `Layer1Demographics` 中新增 `life_stage` 字段，将 `from_profile()` 改为 `life_stage=profile.layer1_demographics.life_stage`。

- **[P0-DM-002]** `src/aicbc/core/models/persona.py:12-19` — `Layer1Demographics` 完全缺失"生命周期"子结构（`life_stage` + `brand_relationship_stage`）。`02-阶段一-画像生成.md` 明确要求 Layer 1 包含此字段。此缺失直接导致 P0-DM-001 的赋值错误。
  - **修复**：在 `Layer1Demographics` 中添加 `life_stage: str` 和 `brand_relationship_stage: str` 字段。

#### P1-MAJOR

- **[P1-DM-001]** `Layer2Behavior` 缺失 02 文档要求的多个关键字段：支付方式、促销参与深度、凑单行为、信任模式、搜索/浏览行为、品牌转换原因、NPS 值。
- **[P1-DM-002]** `Layer4Scenarios` 与 02 文档规格完全不对齐：缺失 5 个场景反应（压力大/朋友推荐/大促限时限量/发现买贵了/产品故障售后）和人物小传三段式结构（过去/现在/未来）。
- **[P1-DM-003]** `Layer1Demographics` 缺失完整家庭结构字段：子女数量、子女年龄范围、是否与父母同住、宠物拥有情况。
- **[P1-DM-004]** 12维标签体系中维度 4-12（需求与痛点、数字化行为、触点与旅程、动态生命周期、审美与感官、圈层归属、语言风格、可持续与隐私）在 Pydantic 模型中无对应结构化字段。
- **[P1-DM-005]** `language_samples` 强制要求 3 条，与 02 文档"3-5 句"不一致。设计文档列出了 5 种具体语言样本类型但无法全部持久化。

#### P2-MINOR（8项）

- `Layer1Demographics.city` 字段命名歧义（存的是城市层级而非具体城市），缺失籍贯字段
- `income` 仅表示个人年收入，缺失家庭年收入和可支配收入
- `TensionCombination.narrative_explanation` 双重校验（Field min_length + field_validator）存在行为差异
- `TensionPair.clamp_tension` 静默截断超出 [0,1] 的值，隐藏调用方错误
- `SeedConfig.anxieties` 去重逻辑与 `max_length=3` 语义歧义
- `PersonaSummary.from_profile(profile: Any)` 无运行时类型校验
- `language_samples` 20-60 字校验使用 `len()`，将标点符号计入字数

### Layer 2 — 生成引擎层

**评级：C+**

#### P1-MAJOR

- **[P1-GE-001]** `configs/prompts/persona_generation.txt` 模板中缺少 `{{correction_feedback}}` 占位符。`_build_prompt()` 构造了 feedback_block 并设置替换，但因模板无对应占位符，来自 Harness 校验管道的修正反馈被静默丢弃。反馈修正链路完全断裂。
  - **修复**：在模板中添加 `{{correction_feedback}}` 占位符，解除 `layer_num == 1` 限制。

- **[P1-GE-002]** Jinja2 控制标签 `{% if previous_layers %}` 泄露到 LLM 提示词中。清理正则仅匹配相邻的 `{% if %}{% endif %}`，而模板中二者之间有实际文本内容，导致 `{% if previous_layers %}` 始终残留在最终提示词中。
  - **修复**：切换到 Jinja2 模板引擎，或彻底移除模板中的条件标签。

- **[P1-GE-003]** `SeedGenerator.generate_seed()` 缺少张力不足时的重新采样逻辑。`07-Harness` 和配置文件要求 `tension_min: 0.3`、`retry_on_low_tension: true`、`max_seed_attempts: 10`，但代码计算 `tension_score` 后无条件返回，可能生成张力为 0 的"和谐种子"。
  - **修复**：在 `generate_seed()` 中添加重试循环，tension < 0.3 重新采样，最多 10 次。

- **[P1-GE-004]** City Tier 严重简化。`消费者画像.md` 定义 10 个城市层级，代码仅保留 5 个，遗漏"农村"和"海外"——二者在中国消费者研究中是独立的、行为差异显著的群体。
  - **修复**：拆分为完整的城市层级列表，补充"农村"和"海外"。

- **[P1-GE-005]** PersonaProfile 数据模型缺少"场景反应库"和"人物小传"字段。Prompt 模板第 5 条要求 LLM 输出三段式小传，但因无对应 JSON Schema 字段，LLM 输出即使包含这些内容也会在解析中被忽略。
  - **修复**：在 PersonaProfile 中增加 `scene_reactions: SceneReactions` 和 `mini_biography: MiniBiography` 字段。

#### 严重遗漏

- **v1.2 偏差控制指令未注入 Prompt**：`05-Prompt模板库.md` v1.2 新增的职业-收入硬性约束、性别去刻板化指令（女性>=30%参数党等）、语言偏见过滤——这些关键的偏见控制规则完全没有注入到代码生成的 Prompt 中。当前 Prompt 模板仍停留在 v1.0 时代的简洁版本。

#### P2-MINOR（6项）

- `_sample_extra_tags` 标签池遗漏"潮流时尚""自然环保""身份象征/品牌型"等多个标签
- `_aggregate_tension` 使用硬编码 e 常量而非 `math.e`
- 缓存键不包含 temperature/max_tokens
- 语言样本数量硬编码 3 条 vs 设计文档 3-5 条

### Layer 3 — 存储/检索层

**评级：D**

CRUD 正确性和线程安全通过审查。但 `12-画像资产化管理规范` 定义的 7 大管理能力中 6 项完全未实现。

#### P1-MAJOR（7项）

- **[P1-ST-001]** 画像生命周期管理完全缺失 — `PersonaProfile` 无 `status` 字段，无状态机（DRAFT→REVIEWED→PUBLISHED→ARCHIVED→DEPRECATED）
- **[P1-ST-002]** 版本控制完全缺失 — 无语义化版本号，`save()` 直接覆盖旧数据，无变更历史
- **[P1-ST-003]** 去重策略全部缺失 — 无哈希比对、无 Embedding 相似度检测、无 persona_fingerprint
- **[P1-ST-004]** 血缘追踪完全缺失 — 无派生/衍生/引用/影响关系记录
- **[P1-ST-005]** 检索维度严重不足 — 仅 4 个固定 AND 过滤，无范围匹配/全文检索/语义检索/OR 组合，无 PersonaSearchEngine
- **[P1-ST-006]** 权限与协作机制完全缺失 — 无五级角色定义，API 路由无任何认证/授权中间件
- **[P1-ST-007]** 归档与清理机制完全缺失 — 纯内存存储，无自动归档（90天无使用），进程重启数据丢失

#### P2-MINOR（4项）

- 倒排索引精确匹配与 Phase 2 substring 过滤语义冲突——这是 P0 级检索缺陷，详见生命周期验证
- 模块级单例懒初始化存在 race condition（低概率）
- `save()` 无输入校验，`persona.layer1_demographics` 为 None 时会抛异常

#### P1-MAJOR（6项）

### Layer 5 — 安全/偏见/成本横切层

**评级：C**

基础安全防御（输入消毒、LLM 输出泄露检测、成本熔断、速率限制）已就位。但偏见审计管线存在关键缺口。

#### P0-CRITICAL

- **[P0-SEC-001]** `src/aicbc/api/routes/personas.py:136-146` — **REJECT without REGENERATE**。偏见审计 FAILED 时执行 `continue`（跳过存储），但未触发重新生成。违反 `虚拟消费者公平性规范.md` 第 2.1 节统一要求的 `reject_and_regenerate` 处置动作。导致有效样本量不可控。
  - **修复**：在 `continue` 前增加重新生成循环（重置种子→重新生成→重新审计，最多重试 3 次）。

- **[P0-SEC-002]** 偏见审计缺少 CRITICAL 严重级别。设计文档要求四级严重度（LOW/MEDIUM/HIGH/CRITICAL），CRITICAL 级命中触发小伦一票否决权。代码仅可实现 HIGH/MEDIUM/LOW 三级。SP-003/SP-006/SP-012 等本应为 CRITICAL 的模式被降级为 "high"，一票否决权机制完全失效。
  - **修复**：增加 "critical" 级别，按设计文档映射 CRITICAL 模式，实现批次暂停逻辑。

#### P1-MAJOR（5项）

- 偏见审计管线仅实现 1/4 步骤（关键词匹配），缺失 Embedding 语义相似度、统计分布检验（KS/卡方/Fisher）、Cramér's V 关联分析、Shannon 熵多样性计算
- 刻板印象模式库覆盖不足——缺失婚育（4个）和教育/社会经济（4个）类别
- 速率限制器非线程安全——并发请求可绕过限制
- LLMClient.generate() 未对 study_id 做独立安全消毒（纵深防御不完整）
- BiasAuditResult 未实现设计文档要求的效应量优先判定（Cramér's V ≥0.2 触发）

#### 正面发现

- 成本熔断四级机制（NORMAL/WARNING/DEGRADE/FUSE/EMERGENCY）完整实现，含状态持久化和自动降级
- `sanitize_id` / `sanitize_text` 输入消毒在 API 入口正确调用
- LLM 输出泄露检测（SEC-011）在 LLMClient 层正确拦截
- 生产环境错误消息脱敏逻辑正确

## 第二部分：生命周期追踪验证

**整体评分：D**

追踪一个画像从请求到删除的 7 个阶段，发现 13 个跨层数据断裂点，其中 7 个 P0 级。

### 跨层断裂 TOP 7

| 编号 | 描述 | 阶段 | 严重度 |
|------|------|------|--------|
| **CB-1** | `SchemaValidator.VALID_CITIES` / `SeedGenerator.CITY_TIERS` / LLM 生成的 city 值三套词汇表不兼容，导致几乎所有画像的 `city` 字段触发 schema 校验失败 | 3 | P0 |
| **CB-2** | 倒排索引精确匹配 + Phase 2 substring 过滤双重语义：非精确查询使索引返回空集，Phase 2 无法补救 | 5 | P0 |
| **CB-3** | 场景反应库和人物小传在整个生成/存储/检索链路中完全缺失——这是整个生命周期中最大的数据丢失 | 2→4→5 | P0 |
| **CB-4** | Bias FAILED → REJECT 但无 REGENERATE：失败画像被丢弃但未重试 | 3→4 | P0 |
| **CB-5** | `PersonaProfile` 无 status/version 字段，生命周期管理完全缺失，删除后无法恢复 | 4→7 | P0 |
| **CB-6** | `life_stage` 语义加载到 `segment` 字符串拼接上，`PersonaSummary.life_stage = profile.segment` 混淆概念 | 1→2→5 | P0 |
| **CB-7** | 偏见审计无 CRITICAL 级别，无法区分可修复偏见与严重伦理违规 | 3 | P0 |

### 各阶段数据流

```
阶段0 请求入口    ✅ PASS  — sanitize_id 正确执行，返回值被实际使用
阶段1 种子生成    ❌ FAIL  — tension=0 无拒绝；life_stage 来源与存储不匹配
阶段2 画像生成    ❌ FAIL  — 场景反应库/人物小传丢失；correction_feedback 链路断裂
阶段3 校验评分    ❌ FAIL  — VALID_CITIES 三套词汇表冲突；校验失败不阻断存储
阶段4 存储        ⚠️ PASS — 线程安全但非原子操作（索引失败不回滚）
阶段5 检索        ❌ FAIL  — 索引精确匹配/子串过滤语义冲突；life_stage 混淆
阶段6 后续使用    ✅ PASS  — persona_id 正确传递到模拟系统
阶段7 生命周期结束 ❌ FAIL  — 无归档/无软删除/无版本回滚/孤儿引用
```
