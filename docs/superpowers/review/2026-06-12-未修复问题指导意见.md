## 二、根因分类与指导意见

### A 类：前端独立开发脱离真实 API（5 项 P0）

**起因**：前端使用 mock 数据独立开发，未与真实后端 API 联调。

**影响**：细分对比页面接真实 API 即崩溃；对话实验室多轮对话完全无记忆；系统设置的修改对系统零影响。

**指导意见**：

> **优先级：第一批修复。** 这类问题的共同特点是无后端依赖，纯前端修改。参考已完成的各模块后端 P0 修复标准，按以下顺序处理：
>
> 1. **细分对比字段对齐**（P0-VIZ-001, P0-VIZ-002）— 把 `comparisons` 改为 `per_attribute`，分段选项从 API 动态获取。1-2 工时。
> 2. **分析任务状态类型对齐**（P0-API-002, P0-API-003）— 前端 `status` 类型加 `QUEUED`，移除 `removeJob`。1 工时。
> 3. **对话实验室 context 对齐**（P0-CHAT-001）— 前端维护对话历史数组并传递。2-3 工时。
> 4. **系统设置接入后端 API**（P0-SET-001）— 前端改为从 `GET /admin/settings` 读写。需后端已实现此端点。2-3 工时。
>
> **判定标准**：修复完成后，在无 mock 的环境下（连接真实后端）手动测试每个页面的一次完整操作流程。

### B 类：品类绑定（2 项 P0）

**起因**：系统起步于洗碗机 CBC 实验，画像管理、市场模拟、问卷配置、创建研究全部为洗碗机硬编码。

**影响**：系统无法用于任何其他产品品类的真实研究。

**指导意见**：

> **优先级：最高架构优先级，但分步实施。** 这是整个项目最大的架构债务。
>
> **第一步**：创建新研究的 API 和前端形态已支持 `attributes` 和 `design_parameters` 可选参数，前端需配套开发属性配置界面（动态添加/编辑属性+水平值+禁止组合）。这是品类通用化的核心瓶颈——用户必须能通过前端完整配置研究参数。
>
> **第二步**：将画像管理的 `DishwasherContext` 改为 `PurchaseContext`（含 `product_category` 动态字段）、市场模拟的 `ProductScenario` 已动态化（上一轮 P0 已修复），验证全链路可通。
>
> **不建议**：一次性全部重写为通用方案。建议先支持 2-3 个品类（洗碗机+手机+护肤品）验证通用性，再完全抽象。

### C 类：架构级缺陷（2 项 P0）

**起因**：P0-LIFE-001（进程间存储隔离）是开发期架构决策（进程内内存存储+异步 Celery worker）的直接后果。P0-VIZ-002（NLScenarioParser）是设计文档定义了"LLM 实现"的组件但代码中零实现。

**指导意见**：

> **P0-LIFE-001 进程间存储隔离**：当前通过 Celery AsyncResult 桥接 + celery_task_id 追踪已实现缓解方案。真正的根治需要将 AnalysisStore 迁移到 Redis——但这应作为 Phase 3（生产化部署）的专项工作，而非当前迭代的阻碍。**当前缓解方案可接受，建议提升为 P1 并推迟到生产化阶段。**
>
> **NLScenarioParser**：这是 LLM Agent 对话流程的关键桥梁。实现路径：(a) 利用已有的 LLM 调用链（`LLMClient.generate_json`），传入属性定义 + 用户自然语言，让 LLM 输出结构化 `ProductScenario` JSON。(b) 这会引入额外 LLM API 调用成本——需要评估其业务价值。（c）**建议**：在 MVP 阶段，前端提供表单式情景编辑器（已存在）作为主路径；NLScenarioParser 作为增强功能推迟。

### D 类：画像资产管理缺失（7 项 P1）

**现状**：`12-画像资产化管理规范` 定义了完整的资产管理生命周期，但代码中版本控制、去重、血缘追踪、权限管理、归档清理全部未实现。当前 PersonaStore 是纯内存 CRUD。

**指导意见**：

> **分三阶段补齐：**
>
> **阶段 A（必须现在做）**：`status` 字段 + 哈希去重。画像已有 `bias_audit_status`，增加 `lifecycle_status`（DRAFT→REVIEWED→PUBLISHED→ARCHIVED）。哈希去重实现简单（`hashlib.sha256` 对字段排序后的 JSON 做指纹），能在生成阶段阻断重复画像。**3-4 工时。**
>
> **阶段 B（Phase 2.5）**：版本控制 + 检索增强。语义化版本号 + 版本历史存储。检索增加全文搜索（画像小传/语言样本关键词）。
>
> **阶段 C（Phase 3 生产化）**：血缘追踪 + 权限管理 + 归档。这些是生产环境多租户场景的需求，MVP 阶段可声明不支持。
>
> **判定标准**：阶段 A 完成后，系统能追踪画像生命周期状态并阻止完全重复的画像入库。

### E 类：设计文档→代码鸿沟（多个模块）

**现状**：`ReportBuilder`、`CBCVisualizer`、`LatentClassModel`、`NLScenarioParser` 等组件在设计文档有完整伪代码但代码中零实现。

**指导意见**：

> 这些组件分两类处理：
>
> **第一类（建议实现）**：`LatentClassModel`——这是细分群体对比的核心增强。实现后可自动发现新的人群细分（不再仅依赖预定义 segment 标签）。与现有 `segment_comparison.py` 的统计引擎配合使用，是差异化能力。**优先级：P1，建议下个迭代实现。**
>
> **第二类（建议推迟）**：`ReportBuilder`、`CBCVisualizer`——当前前端 ECharts 已替代了大部分可视化需求，LLM Agent 的 `_generate_report()` 提供了基础报告。后端 Python 端的可视化/报告可用于 PDF 导出场景，但当前用户主要通过 Web 前端消费结果——推迟到 Phase 3。
>
> **原则**：设计文档中的每个"应有组件"必须在代码中有对应状态——要么已实现、要么有 TODO + 优先级标注、要么明确声明推迟并记录原因。消除"设计文档定义但代码中完全找不到任何痕迹"的不确定性。

## 四、总结

经过连续审查和修复，10 个模块的 **56+ P0 问题中约 46 个已解决（82%）**。剩余未修复问题呈现清晰的分布规律：

- **50%（5/10）的未修复 P0 属于"前端独立开发脱离真实 API"**
- **20%（2/10）属于"品类绑定"**
- **20%（2/10）属于"架构级缺陷"**
- **10%（1/10）属于"前后端类型不一致"**

**建议下一步聚焦 A 类问题（5 项 P0 前端修复），预计 10-12 工时可全部解决。B 类（品类绑定）随前端属性配置界面同步推进。C 类架构缺陷可接受当前缓解方案，推迟到 Phase 3。**
