# AI_CBC 测试架构治理设计

> **版本**：v1.0
> **日期**：2026-06-18
> **负责人**：小测（QA Engineer）
> **相关角色**：小端（Backend）、小前（Frontend）、小安（Security）、小速（Performance）
> **状态**：待实施

## 1. 背景与目标

### 1.1 背景

当前 AI_CBC 项目测试模块存在以下问题：

- 测试文件堆积在 `tests/` 根目录，职责边界模糊
- 多个模块重复覆盖同一功能（Bias Audit、HB 恢复/收敛、Cost Fuse 等）
- `tests/performance/` 被 `pyproject.toml` 默认排除，CI/CD 从未运行
- Nightly 工作流声明运行完整测试，但默认 `addopts` 过滤掉所有 `@pytest.mark.slow` 测试
- Trivy 镜像扫描在 CI 中非阻塞、在 `security-scan.yml` 中阻塞，策略不统一
- 前端覆盖率阈值文档（50%）与配置（60%）不一致

### 1.2 目标

通过彻底重构测试架构，实现：

1. **清晰分层**：按测试金字塔组织单元/集成/E2E/红队/性能测试
2. **消除冗余**：合并重复测试模块，删除过时/无用测试
3. **修复 CI/CD 冲突**：让 CI 快速反馈、Nightly 全量回归、Security-scan 策略一致
4. **提升质量**：每个测试满足单一职责、独立、可重复、边界/异常覆盖
5. **防杀虫剂**：建立测试数据工厂、mutation testing、季度审查机制

## 2. 设计原则

本设计遵循以下原则：

- **战略层**：尽早测、聚焦高风险区、承认不可能穷举、定期更新用例防“杀虫剂”
- **用例层**：单一职责、完全独立、可重复、自动断言、重点测边界和异常
- **自动化层（FIRST）**：快速、独立、可重复、自验证、及时编写
- **架构层**：坚持测试金字塔（大量单元测试 + 少量集成/E2E）、只测行为不测实现、隔离不稳定依赖、把测试代码当产品维护
- **TDD**：红→绿→重构，让测试倒逼出高内聚、低耦合的良好设计

## 5. 标记策略

### 5.1 核心 Mark

| Mark | 含义 | 执行场景 |
|---|---|---|
| `unit` | 纯单元，无外部依赖 | CI/PR、本地快速反馈 |
| `integration` | 需要 DB/Redis/Celery/API | CI/PR |
| `e2e` | 完整端到端链路 | CI/PR（精简）、Nightly |
| `slow` | 执行 >30s | Nightly |
| `security` | 快速安全校验 | CI/PR |
| `redteam` | 慢速对抗/渗透/偏见审计 | Nightly |
| `performance` | 性能/负载测试 | Nightly 单独 job |
| `external` | 需要真实外部服务 | 按需/手动 |

### 5.2 规则

- 每个测试至少有一个分层 mark（`unit`/`integration`/`e2e`）
- `slow` 可与 `unit`/`integration`/`e2e` 叠加
- 尽量用 mock 替代 `external`，减少真实 LLM 依赖

### 5.3 pyproject.toml 配置

```toml
[tool.pytest.ini_options]
asyncio_mode = "auto"
testpaths = ["tests"]
python_files = ["test_*.py", "*_test.py"]
addopts = "-v"
markers = [
    "unit: pure unit tests with no external dependencies",
    "integration: tests requiring DB/Redis/Celery/API",
    "e2e: end-to-end full pipeline tests",
    "slow: tests taking >30s (deselect with '-m \"not slow\"')",
    "security: fast security validation tests",
    "redteam: slow adversarial/penetration/bias tests",
    "performance: load and regression tests",
    "external: tests requiring real external services (LLM/Mongo/Redis)",
]
norecursedirs = [".git", ".tox", "dist", "build"]
```

## 7. CI/CD 分层执行设计

### 7.1 CI / PR 执行范围

```bash
pytest tests/ \
  -v \
  -m "(unit or integration or security) and not slow and not redteam and not performance" \
  --timeout=120 \
  --cov=src \
  --cov-report=xml \
  --cov-report=html \
  --cov-fail-under=60 \
  --junitxml=test-results/fast.xml
```

### 7.2 Redteam Fast

```bash
pytest tests/redteam/ \
  -v \
  -m "security and not slow" \
  --timeout=120 \
  --junitxml=test-results/redteam-fast.xml
```

### 7.3 Nightly 全量

```bash
pytest tests/ \
  -v \
  -m "not performance" \
  --timeout=300 \
  --cov=src \
  --cov-report=xml \
  --cov-report=html \
  --junitxml=test-results/full.xml
```

### 7.4 Nightly Performance

```bash
# 1. Locust 压测
locust -f scripts/locust_baseline.py \
  --host http://staging.aicbc.example.com \
  --run-time 5m \
  --users 20 \
  --spawn-rate 5 \
  --headless \
  --html reports/locust-report.html \
  --csv reports/locust

# 2. pytest 性能回归断言
pytest tests/performance/ \
  -v \
  --timeout=300 \
  --junitxml=test-results/performance.xml
```

### 7.5 质量门禁

| 门禁 | 阈值 | 位置 |
|---|---|---|
| ruff lint | 0 错误 | CI quality |
| ruff format | 0 差异 | CI quality |
| mypy | 0 错误（分阶段落地） | CI quality |
| bandit high | 0 | CI quality |
| pytest fast coverage | ≥ 60% | CI fast-test |
| frontend coverage | statements/branches/functions/lines ≥ 60% | CI frontend |
| pip-audit | HIGH/CRITICAL = 0 | CI security + security-scan |
| npm audit | high = 0 | CI frontend + security-scan |
| Trivy | CRITICAL/HIGH = 0 | CI security + security-scan |

### 7.6 Trivy 统一策略

CI 和 `security-scan.yml` 中统一使用：

```yaml
exit-code: '1'
severity: 'CRITICAL,HIGH'
```

## 9. 验收标准

| 维度 | 标准 |
|---|---|
| 结构 | `tests/` 下仅存在 `unit/`、`integration/`、`e2e/`、`redteam/`、`performance/`、`manual/`、`support/` |
| 标记 | 每个测试函数都有且仅有一个主分层 mark |
| 数量 | 后端测试文件从 52 个精简到 35-40 个 |
| 时间 | CI fast-test ≤ 10 分钟；Nightly 全量 ≤ 60 分钟 |
| 覆盖 | 后端覆盖率 ≥ 60%；前端覆盖率 ≥ 60%；重构后覆盖率不下降 |
| 独立 | 任意两个测试可以单独运行，不依赖执行顺序 |
| 门禁 | ruff、bandit、pip-audit、npm audit、Trivy 均阻塞 |
| 性能 | Nightly performance job 同时输出 Locust 报告和 pytest 回归断言 |
| 文档 | `tests/CLAUDE.md`、`frontend/CLAUDE.md`、`docs/测试验证方案.md` 同步更新 |

## 10. 风险与回滚

| 风险 | 缓解措施 |
|---|---|
| 文件移动导致 PR 冲突 | 每阶段 PR 尽快合并；其他开发者暂停新增测试 |
| 合并测试丢失用例 | 合并前逐条对比原文件 test 函数清单 |
| CI 时间变长 | 阶段 2 后监控 fast-test 时间，必要时调整 mark |
| mypy 无法一次达标 | 分两步：先统计修复，再开启阻塞 |
| performance 依赖 staging | Locust 阶段 `continue-on-error: true`；pytest 回归用本地 CSV |

**回滚策略**：每阶段独立分支 + PR，可单独 revert；阶段 3 合并前建 `backup/tests-pre-merge` 标签。

## 11. 防“杀虫剂”机制

| 机制 | 落地方式 |
|---|---|
| 定期审查 | 每季度由 小测 牵头审查 10% 用例 |
| 数据轮换 | `factories.py` 支持参数化随机种子 |
| 边界扩展 | 每次 bug fix 必须补充边界/异常测试 |
| Mutation testing | 阶段 4 引入 `mutmut`，试点 `unit/` |
| 新功能强制测试 | PR 模板增加测试覆盖 checklist |
| 覆盖率不下降 | CI 中 `--cov-fail-under=60` 强制门禁 |
