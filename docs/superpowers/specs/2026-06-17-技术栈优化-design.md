# AI_CBC 技术栈优化设计

> **版本**：v1.0
> **定位**：技术栈选型优化与迁移兼容设计
> **日期**：2026-06-17
> **状态**：待审批

---

## 一、背景与约束

AI_CBC 当前处于免费层运行阶段，核心约束为：

- **Render**：免费 Web Service，仅支持单个容器，资源有限，冷启动明显。
- **MongoDB**：MongoDB Atlas 免费层（M0），连接数与存储受限。
- **Redis**：Upstash Redis 免费层，每日命令数与内存受限。

同时，项目计划在后续阶段迁移到**个人云服务器**全量自托管。因此本次优化需要在两个目标之间取得平衡：

1. 免费层阶段获得**最好的性能与体验**。
2. 迁移到自托管时**改动最小**，能够复用同一套代码与镜像。

## 二、优化目标（按优先级排序）

| 优先级 | 目标 | 说明 |
|--------|------|------|
| 1 | 性能 | API 响应、分析任务执行、页面加载尽可能快 |
| 2 | 成本 | 不超出 Render / Atlas / Upstash 免费额度 |
| 3 | 运维 | 减少外部依赖，方便后续迁移 |
| 4 | 开发体验 | 保持本地开发与生产行为一致 |

## 三、最终选定方案：可迁移的免费层优化

### 3.1 核心思路

- **Render 单容器继续承载 API + Worker + Beat**：通过 `supervisord` 运行，符合免费层限制。
- **前端迁到 Cloudflare Pages**：静态托管 + CDN，不受 Render 休眠影响，加载最快。
- **Redis 仍作为 Celery broker**：同时用于缓存和限流；通过配置压缩命令消耗。
- **MongoDB 作为主数据与任务状态存储**：减少 Redis 数据写入压力。
- **代码层保持可拆分**：API、Worker、Beat 的启动入口独立，迁移后可直接拆成独立容器。

### 3.2 当前免费层架构

```
用户
  │
  ▼
Cloudflare Pages（前端静态资源）
  │
  ▼
Render Web Service（supervisord 单容器）
  ├── API 进程（FastAPI，不导入 pymc/pandas/numpy）
  ├── Worker 进程（Celery，导入重依赖）
  └── Beat 进程（Celery beat）
  │
  ├──→ MongoDB Atlas（主数据、任务状态）
  └──→ Upstash Redis（Celery broker、缓存、限流）
```

### 3.3 迁移后自托管架构

```
用户
  │
  ▼
Nginx（HTTPS、静态前端、反代 API）
  │
  ├──→ API 容器（FastAPI）
  ├──→ Worker 容器（Celery）
  ├──→ Beat 容器（Celery beat）
  │
  ├──→ MongoDB 容器
  └──→ Redis 容器
  │
  └──→ Prometheus + Grafana 容器（可选）
```

迁移时仅需：

1. 把 Render 上的 supervisord 启动改为 `docker-compose.yml` 多容器启动。
2. 修改 `MONGODB_URL`、`REDIS_URL` 指向本地容器。
3. 前端构建产物由 Nginx 托管（也可继续用 Cloudflare Pages）。

## 四、组件职责

### 4.1 API 进程

- 负责 HTTP 请求处理、路由、依赖注入、中间件。
- 不导入 `pymc`、`pandas`、`numpy` 等重依赖，使用懒加载。
- 读写 MongoDB，读写 Redis 缓存，下发 Celery 任务。
- 暴露 `/health` 与 `/ready` 探针。

### 4.2 Worker 进程

- Celery worker，消费 `analysis`、`default` 队列。
- 执行 HB/MCMC 采样、虚拟消费者批量生成、偏见审计等重任务。
- 结果写回 MongoDB，失败状态也写回 MongoDB。

### 4.3 Beat 进程

- Celery beat 调度器。
- 触发定时清理、成本统计、审计任务。
- 免费层与 API/Worker 共用容器；迁移后独立容器。

### 4.4 前端

- React + Vite 构建的静态 SPA。
- 免费层使用 Cloudflare Pages；迁移后由 Nginx 托管。
- API 基础地址通过构建时环境变量注入。

### 4.5 存储与缓存

| 组件 | 用途 | 免费层注意事项 |
|------|------|----------------|
| MongoDB | 主数据、任务状态、分析结果 | 注意连接数上限与存储容量 |
| Redis | Celery broker、结果后端、缓存、限流 | 注意每日命令数上限 |

## 五、数据流

### 5.1 创建研究

1. 前端 POST `/api/v1/studies`。
2. API 校验请求体，写入 MongoDB。
3. 返回 `study_id`。

### 5.2 生成虚拟消费者

1. 前端 POST `/api/v1/studies/{id}/generate-personas`。
2. API 向 Redis broker 发送 Celery 任务。
3. Worker 消费任务，调用 LLM 批量生成 `PersonaProfile`。
4. Worker 将结果写入 MongoDB，更新任务状态。
5. 前端轮询任务状态，获取结果。

### 5.3 运行 HB 分析

1. 前端 POST `/api/v1/studies/{id}/analyze`。
2. API 写入分析任务记录，向 Redis broker 发送 Celery 任务。
3. API 返回 `202 Accepted` + `task_id`。
4. Worker 执行 `pymc` 采样，计算 R-hat、ESS、WTP 等指标。
5. Worker 将结果写回 MongoDB。
6. 前端轮询 `/api/v1/tasks/{task_id}` 获取结果。

### 5.4 定时任务

1. Beat 按配置 schedule 向 Redis broker 发送任务。
2. Worker 消费并执行清理、成本统计、审计任务。

## 六、Redis 使用策略

虽然希望降低 Redis 命令消耗，但 Celery 需要 broker，因此 Redis 仍承担 broker 角色。优化手段：

- `task_ignore_result=True`：大部分任务不需要结果后端。
- `result_expires=300`：必须保留的结果 5 分钟后过期。
- 缓存使用短 TTL（60s ~ 300s）。
- 限流按 IP + 路径聚合，减少命令数。
- 避免把大对象存入 Redis，优先存入 MongoDB。

## 七、性能优化点

### 7.1 容器瘦身

- Dockerfile 中删除开发依赖，只安装运行所需包。
- 使用 `orjson` 替代默认 JSON 序列化。
- 对 `src/` 和 `configs/` 预编译字节码，减少冷启动导入时间。
- API 进程懒加载 `pymc`、`pandas`、`numpy`。

### 7.2 MongoDB 优化

- 为高频查询字段添加索引：`study_id`、`task_id`、`status`、`created_at`。
- 使用投影减少返回字段。
- 批量写入虚拟消费者画像，减少网络往返。

### 7.3 Worker 优化

- Worker 并发数在免费层保持为 1，避免 OOM。
- HB 采样使用合理的链数与迭代数，在统计可靠性与耗时之间平衡。
- 任务失败重试 3 次，避免无效重试消耗资源。

### 7.4 前端优化

- 构建产物启用 gzip/brotli 压缩（`vite-plugin-compression2`）。
- 路由懒加载已实施，保持即可。
- 静态资源托管到 Cloudflare Pages，利用全球 CDN。

## 八、错误处理

### 8.1 API 层

- 统一 FastAPI exception handler，返回结构化错误响应。
- `/health` 只做自身健康检查，`/ready` 检查 MongoDB 与 Redis 连接。
- 异步任务统一返回 `202 Accepted`，失败状态通过任务查询接口暴露。

### 8.2 Worker 层

- Celery 自动重试，指数退避，最大 3 次。
- 重试耗尽后写入 MongoDB `dead_letters` 集合。
- `CostFuse` 拦截异常 LLM 调用峰值，防止免费额度被快速耗尽。

### 8.3 存储层

- MongoDB 写操作使用 `w=majority`（Atlas 默认）。
- Redis 连接断开后自动重连。

## 九、测试策略

### 9.1 单元测试

- API 路由、schema、依赖注入覆盖。
- Worker 任务使用 `CELERY_TASK_ALWAYS_EAGER=True` 同步执行。
- Mock LLM 调用与外部存储。
- 保持覆盖率 ≥ 60%。

### 9.2 集成测试

- CI 中启动 MongoDB、Redis service 容器。
- 验证 API → Redis broker → Worker → MongoDB 全链路。
- 验证 Celery beat schedule 正确触发。

### 9.3 迁移验证

- 本地运行完整 `docker-compose.yml`。
- 验证 supervisord-render 与 compose 多容器行为一致。
- 验证 Nginx 静态前端 + 反代 API。

### 9.4 前端测试

- Vitest + React Testing Library 保持不变。
- Mock API 轮询与任务状态。
- 验证构建产物在 Cloudflare Pages 和 Nginx 均可运行。

### 9.5 关键测试场景

- Worker 崩溃恢复。
- Render 休眠后冷启动恢复。
- Upstash Redis 命令上限内正常运行。
- 迁移后同一镜像零改动部署。

## 十二、风险与权衡

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Render 单容器资源竞争 | Worker 与 API 争抢 CPU/内存 | Worker 并发保持 1，API 懒加载重依赖 |
| Upstash Redis 命令超限 | 任务队列阻塞 | 调低结果保留时间，合并限流键，监控命令量 |
| MongoDB Atlas 连接数耗尽 | API 无法访问数据库 | 限制连接池大小，避免长连接泄漏 |
| Render 休眠导致 Beat  missed | 定时任务延迟 | 非关键定时任务可接受延迟；关键任务迁移后改善 |
| 迁移时配置遗漏 | 服务启动失败 | 使用 `.env.example` 模板，逐条核对 |
