# 虚拟消费者真实性治理设计

> **版本**：v1.0  
> **日期**：2026-06-26  
> **负责人**：小应（LLM 应用）、小示（行为工程）  
> **状态**：设计已确认，待实施计划  

---

## 一、背景与问题

当前后端共生成 19 个虚拟消费者，前端显示 14 个，平均真实性评分 7.5/14，仅 3 个（21%）超过及格线 9 分。使用 DeepSeek-v4-flash 多次复现均如此。更严重者，出现“学生住宿舍想买洗碗机”这类明显荒诞画像，且该画像评分高达 10 分。

这表明当前系统存在三类不真实，按业务优先级排序：

1. **情境荒诞**：产品情境（`dishwasher_context`）与人口统计、居住条件脱节。
2. **画像模板化 / 没灵魂**：输出像标签集合，缺少具体故事和异常值解释。
3. **语言 AI 感**：语言样本像例句，而非真实人物会说的话。

---

## 二、设计目标与原则

### 2.1 目标

让虚拟消费者首先成为**真实、立体、有内在矛盾的人**，其购买情境、语言、行为都从这个“人”自然推导出来。评分器、prompt、校验器都是为这一原则服务的工具，而非优化目标。

### 2.2 核心原则

- **人先于产品**：先成为真实的人，再决定要不要买洗碗机。
- **叙事即内核**：`mini_biography` 和 `scene_reactions` 是画像的第一公民，四层标签从叙事中获得解释。
- **评分器是探针**：用于诊断“哪里假”，不用于“刷分”。
- **反馈要具体**：自校正必须告诉生成器“错在哪”，而不是只给总分。

## 三、总体架构

### 3.1 新流程

```
[Seed] → [Layer 1-4] → [Narrative Core]
                              ↓
                  [ProductContextDeriver]
                              ↓
                    [Plausibility Gate]
                              ↓
                    [LanguageSampleGenerator]
                              ↓
                    [AuthenticityScorer]（诊断）
                              ↓
                    [Bias Audit] → 输出
```

### 3.2 关键组件

| 组件 | 职责 | 新增/修改 |
|------|------|----------|
| `ProfileGenerator` | 生成四层画像 | 修改：先生成叙事核心，再推导产品情境 |
| `NarrativeCoreGenerator` | 生成 `mini_biography` + `scene_reactions` | 新增 |
| `ProductContextDeriver` | 从人格推导洗碗机情境 | 新增 |
| `PlausibilityValidator` | 检查人—场景—产品一致性 | 新增 |
| `NarrativeConsistencyChecker` | 检查叙事是否解释四层标签 | 新增 |
| `LanguageSampleGenerator` | 从叙事核心生成语言样本 | 新增（替代 `_generate_auxiliary` 中的语言样本部分） |
| `AuthenticityScorer` | 诊断性真实评分 | 修改：弱化关键词计数，新增情境合理性和叙事深度维度 |
| `ConsumerGeneratorAgent` |  orchestrate 生成与自校正 | 修改：以 plausibility 为硬门槛，弱化总分门槛 |

## 四、情境荒诞治理

### 4.1 问题定位

`dishwasher_context` 当前由 `_generate_auxiliary` 用一个独立 prompt 基于“四层概要”顺便生成，没有判断“这个人是否真的可能考虑洗碗机”。

### 4.2 `ProductContextDeriver`

**输入**：四层画像 + 叙事核心。  
**输出**：

```python
class DerivedProductContext(BaseModel):
    eligibility: Literal["not_applicable", "latent_need", "actively_considering"]
    reason: str
    dishwasher_context: DishwasherContext
```

**推导示例**：

| 画像条件 | eligibility | reason |
|---------|-------------|--------|
| 学生 + 学校宿舍 | `not_applicable` | 无独立厨房，无法安装 |
| 租房独居 + 厨房极小 | `latent_need` | 空间受限，搬家不便 |
| 已婚有孩 + 自有住房 | `actively_considering` | 具备安装条件和家庭需求 |

当 `eligibility == "not_applicable"` 时，`dishwasher_context` 应明确写出约束，例如 `["无独立厨房，无法安装"]`，而不是编造决策因素。

### 4.3 `PlausibilityValidator`

**规则结构**：

```python
class PlausibilityRule(BaseModel):
    rule_id: str
    severity: Literal["hard", "soft"]
    condition: Callable[..., bool]
    message_template: str
```

**规则示例**：

| 规则 ID | 严重度 | 触发条件 | 反馈 |
|--------|--------|---------|------|
| PLA-001 | hard | 学生 ∧ 住宿舍 ∧ 有洗碗机需求 | 学生住宿舍通常无独立厨房和安装条件，请重新考虑该画像是否真会考虑洗碗机。 |
| PLA-002 | hard | 年收入 3 万以下 ∧ actively_considering 单价 4000 元以上洗碗机 | 低收入画像若 actively_considering 高价洗碗机，需有借贷/赠礼/合租分摊叙事。 |
| PLA-003 | soft | 退休 + 独居 + 大厨房 ∧ actively_considering | 需有“子女推荐/不想弯腰洗碗”等触发事件。 |

**规则存储**：`configs/plausibility_rules/*.yaml`，由业务角色（小联/小示）维护，不硬编码在 Python 中。

### 4.4 集成点

`PlausibilityValidator` 在 `ProductContextDeriver` 之后、`LanguageSampleGenerator` 之前运行。hard 规则失败直接触发重生成，并附带具体规则反馈。

## 五、画像模板化 / 没灵魂治理

### 5.1 叙事核心成为必填字段

`PersonaProfile` 中：

```python
mini_biography: MiniBiography          # 从 Optional 改为必填
scene_reactions: SceneReactions        # 从 Optional 改为必填
```

### 5.2 `NarrativeCoreGenerator`

用专门的一次 LLM 调用，基于完整四层画像生成：

- `MiniBiography`：过去（决定性经历）、现在（日常状态与核心矛盾）、未来（焦虑与渴望）。
- `SceneReactions`：压力大、朋友推荐、大促限量、发现买贵、售后故障五个场景的反应。

### 5.3 Prompt 要求

1. **决定性经历**：必须是一个具体事件，而非泛泛背景。
2. **异常值解释**：至少有一个标签与人生阶段/收入/城市期待不符，并解释原因。
3. **场景反应具体**：不是抽象描述，而是该人物在具体瞬间会怎么做。

### 5.4 `NarrativeConsistencyChecker`

轻量检查叙事是否解释了四层标签：

```python
class NarrativeConsistencyCheck(BaseModel):
    unexplained_tags: list[str]
    contradiction_score: float  # 0-1
```

关键行为标签指 `layer2_behavior` 中的 `decision_style`、`price_sensitivity` 和 `layer3_psychology` 中的 `tension_combination.labels`。若其中任意一项在 `mini_biography` 中无对应解释，标记为 `unexplained_tag`。2 个以上未解释触发“叙事不足”反馈。

### 5.5 反模板约束

在 `persona_generation.txt` 中增加：

- 禁止“典型都市白领/普通上班族/一般消费者”等无特征描述。
- 每个人口统计字段必须包含至少一个具体细节。
- 职业必须给出具体行业和一个小习惯。

### 5.6 批次级去重

`generate_batch` 中比较同一批次 `mini_biography.past` 与 `secret_motivation` 的语义相似度（使用 LLM 嵌入或轻量文本相似度，具体实现待定）。相似度 > 0.7 时，给后续画像的 seed 生成加反馈：“请避免与已有画像相似的童年经历和隐藏动机”。

## 六、语言 AI 感治理

### 6.1 从叙事核心生成长语言样本

`LanguageSampleGenerator` 输入：

- `mini_biography`
- `scene_reactions`
- `layer2_behavior.decision_style`
- `layer3_psychology.defense_mechanism`

输出：3 条语言样本，分别对应三个真实情境：

1. 日常聊天中聊到洗碗机。
2. 评价一款具体洗碗机时。
3. 被推销员介绍洗碗机时。

### 6.2 声音驱动

根据 `decision_style` 和 `defense_mechanism` 给 LLM 声音提示，例如：

| 决策风格 | 声音提示 |
|---------|---------|
| 参数党 | 会提 1-2 个具体参数，但承认“别的没研究” |
| 冲动型 | 短句、情绪词、事后可能后悔 |
| 拖延比较型 | 开头犹豫、结尾没结论 |
| 口碑党 | 引用“我朋友/同事说” |

### 6.3 语言一致性校验

检查：

- 是否包含营销术语、过度理性、全能专家表述（硬性惩罚）。
- 是否与小传中的价值观/行为冲突。

不检查：

- 是否包含“吧/呢/啊”“但是/不过”等具体口语词（避免为了过评分器而堆砌）。

## 七、评分器重新定位

### 7.1 使命变化

| 旧定位 | 新定位 |
|--------|--------|
| 及格门槛 | 诊断探针 |
| 关键词计数 | 语义一致性 + 真实性信号 |

### 7.2 维度调整

**新增**：

- **情境合理性**：基于 `PlausibilityValidator`（0/1/2）。
- **叙事深度**：`mini_biography` 是否有决定性经历、异常值解释（0/1/2）。

**弱化**：

- **语言自然度**：仅惩罚营销术语和过度理性，不再要求特定口语词数量。
- **社会摩擦感**：从数犹豫词改为检测犹豫、知识边界或面子考量是否存在。

### 7.3 及格线调整

- `plausibility_passed` 是**硬门槛**。
- `authenticity_score` 用于监控和人工抽检，不设单一重生成阈值。
- 只有当某维度连续出现 0 分（如连续 3 个画像语言都带营销术语）时，才触发专项反馈。

## 八、数据流与错误处理

### 8.1 自校正循环

```
生成结果
   ↓
PlausibilityValidator
   ↓ hard 失败
   → 反馈具体规则
   → 重生成
   ↓ 通过
NarrativeConsistencyChecker
   ↓ 关键标签未解释
   → 反馈“请在小传中解释 X”
   → 重生成
   ↓ 通过
AuthenticityScorer（诊断）
   ↓
输出 persona + 评分详情
```

### 8.2 `_should_correct` 新逻辑

```python
if not plausibility_passed:
    return True, plausibility_feedback
if narrative_under_explained:
    return True, narrative_feedback
if authenticity_score < 6:  # 仅明显虚假时触发
    return True, authenticity_feedback
return False, ""
```

### 8.3 错误处理

| 场景 | 处理 |
|------|------|
| LLM 不返回叙事核心 | fallback 填充，但标记 `status=DRAFT`，不进入分析 |
| `ProductContextDeriver` 无法推断 | 默认 `not_applicable`，避免胡说 |
| Plausibility 连续 3 次失败 | 停止重生成，记录失败队列，人工复核 |
| Bias 审计失败 | 保持现有最高优先级，直接重生成 |

## 九、测试与验收

| 测试类型 | 内容 | 通过标准 |
|---------|------|---------|
| 单元测试 | `PlausibilityValidator` 各规则命中/未命中 | 学生宿舍场景必须返回 hard 失败 |
| 单元测试 | `ProductContextDeriver` 推导 | 学生宿舍返回 `not_applicable` |
| 集成测试 | 端到端生成 10 个画像 | `plausibility_passed` ≥ 95%；`authenticity_score` 平均分仅用于监控，不设硬性阈值 |
| 人工抽检 | 随机抽取 20 个画像盲评 | 业务方（小验/小联）认为“像真人”比例 ≥ 80% |
| 回归测试 | 旧及格线 9 分场景 | 新流程下仍通过 plausibility + narrative 检查 |
