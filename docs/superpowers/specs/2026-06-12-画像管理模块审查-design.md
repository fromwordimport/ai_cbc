## 三、审查维度（5层 + 1验证）

### Layer 1 — 数据模型层

审查 `PersonaProfile` 及其子模型与 `消费者画像.md` 12维标签体系的对齐程度。

**检查项**：
- 模型字段是否覆盖标签体系的核心维度
- `language_samples` 20-60字约束、`tension_combination` 50字下限的正确性
- `bias_audit_status` 枚举值与文档一致性
- `GenerationMetadata` 成本精度
- Pydantic validators 的边界条件覆盖

### Layer 2 — 生成引擎层

审查 `SeedGenerator` → `ProfileGenerator` 的完整生成链路。

**检查项**：
- 加权采样逻辑与设计分布的对齐
- `TENSION_DEFINITIONS` 是否覆盖文档附录A全部矛盾组合
- 4层顺序生成是否保证"上层解释下层异常"
- Prompt模板变量替换的完整性（Jinja2-style）
- 每层 fallback 策略的合理性
- LLM缓存机制正确性

### Layer 3 — 存储/检索层

审查 `PersonaStore` 实现与 `12-画像资产化管理规范.md` 的差距。

**检查项**：
- CRUD 正确性与线程安全
- 倒排索引在增/删/改时的一致性维护
- 查询过滤正确性（SEC-014 study_id 前缀匹配）
- **差距分析**：版本控制、去重、血缘追踪、权限管理、归档/清理——代码实现了多少？

### Layer 4 — API 层

审查 REST 端点设计与实现质量。

**检查项**：
- RESTful 规范（端点命名、状态码、分页）
- 请求验证（count上限、study_id消毒）
- 错误处理分层（CostFuse快速失败、生产脱敏）
- 响应模型合理性（Summary vs Detail）
- 批量生成的原子性问题

### Layer 5 — 安全/偏见/成本横切层

审查三大横切关注点的实现。

**检查项**：
- 偏见审计：生成→评分→拒绝→统计→告警完整链路
- 输入消毒：`sanitize_id` 注入防护
- 成本熔断：`CostFuseError` 在生成链路的传播与停止语义
- 生产环境错误脱敏
- 安全架构文档要求与代码对齐

### 最终验证 — 生命周期追踪

追踪一个画像的完整生命周期：
```
Seed生成 → LLM生成4层 → Schema校验 → Logic校验
→ 真实性评分 → 偏见审计 → 存储（含索引更新）
→ API检索 → 模拟使用 → 版本演进 → 归档/清理
```
验证每个节点的数据完整性和错误处理。

## 四、评判标准

| 级别 | 标志 | 定义 |
|------|------|------|
| P0-CRITICAL | 🔴 | 安全漏洞、数据丢失风险、核心设计原则违背、API契约破裂 |
| P1-MAJOR | 🟠 | 功能缺失、规范-代码严重不一致、错误处理缺陷 |
| P2-MINOR | 🟡 | 代码异味、命名不一致、边界条件处理不优雅 |
| P3-SUGGESTION | 🔵 | 改进建议，非必须修复 |

**通过标准**：无 P0-CRITICAL，P1-MAJOR ≤ 5个，每个 P1+ 附修复方案。

## 五、执行方案

5个并行 Agent 各负责一层审查，完成后由第6个 Agent 执行生命周期验证，最终汇总为审查报告。

```
Agent-1 (数据模型层) ──┐
Agent-2 (生成引擎层) ──┤
Agent-3 (存储/检索层) ──┼──→ Agent-6 (生命周期验证) ──→ 汇总报告
Agent-4 (API层)       ──┤
Agent-5 (安全/偏见/成本) ─┘
```

每个 Agent 输出：
- 发现清单（级别、文件:行号、描述、修复建议）
- 规范-代码对齐矩阵（PASS / PARTIAL / FAIL / NOT_IMPLEMENTED）

## 六、输出

审查报告写入 `docs/superpowers/review/2026-06-12-画像管理模块审查报告.md`，包含：

1. 执行摘要
2. 各层发现详情（按严重程度排序）
3. 规范-代码对齐矩阵
4. 生命周期追踪结果
5. 修复优先级建议
6. 附录：审查覆盖的文件清单和审查指令
