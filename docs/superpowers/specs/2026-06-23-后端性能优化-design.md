# AI_CBC 后端性能优化设计

> **版本**：v1.0
> **定位**：Azure B2ats v2 双机部署下的后端性能优化与可观测性补齐
> **日期**：2026-06-23
> **状态**：已审批

---

## 一、背景与目标

AI_CBC 后端目前已具备较好的异步基础（Motor、Celery、ORJSONResponse、Redis 限流可选、真实健康检查等），但缺少系统性的性能基线，且存在若干已知的低效点。本次优化在 **Azure B2ats v2 双机部署** 场景下，以 **"先测量、后优化"** 为原则，通过建立基准测试和补齐监控，再对测量出的瓶颈做低风险代码优化，最终提升 API 响应速度、降低资源占用，并确保业务逻辑不受影响。

### 1.1 核心原则

1. **先测量，后优化**：所有代码改动都应有基准数据前后对比。
2. **低风险优先**：不改动业务逻辑、API 契约、统计算法。
3. **端到端可验证**：基准脚本、监控面板、告警、CI 门禁一次性补齐。
4. **Azure B2ats v2 友好**：优化项在主 VM 与 Worker VM 上均可独立生效，不依赖共享文件系统或集群改造。

### 1.2 目标指标

| 指标 | 当前状态 | 目标 |
|------|----------|------|
| Dashboard `/dashboard/summary` P95 | 未测量 | P95 < 800ms（缓存命中） / < 2s（缓存未命中） |
| 研究列表 `/studies` P95 | 未测量 | P95 < 300ms |
| 画像批量生成 `/personas/generate` | 同步阻塞 | 改为异步任务，API 响应 < 500ms |
| HB 分析任务（N=150） | 未测量 | 在 Worker 2G/1CPU 限制下稳定完成，不 OOM |
| API 进程内存占用 | 未测量 | 通过懒加载和投影降低 ≥ 10% |
| Docker 镜像体积 | 未测量 | 记录基线，API target 准备就绪 |

### 1.3 适用范围

- **新增/修改**：`scripts/benchmark_*.py`、`src/aicbc/monitoring/*`、`src/aicbc/api/routes/*`、`src/aicbc/llm/client.py`、`src/aicbc/core/store_mongo.py`、`src/aicbc/core/cache.py`、`docker/*`、`.github/workflows/ci.yml` 等。
- **不做改动**：`tests/` 下测试文件（除非用户明确授权）、业务规范文档、HB 统计算法。

## 二、性能诊断与基准测试

### 2.1 基准脚本

| 脚本 | 用途 | 关键指标 |
|------|------|----------|
| `scripts/benchmark_api.py` | 压测高频 API | RPS、P50/P95/P99、错误率 |
| `scripts/benchmark_worker.py` | 触发并监控分析任务 | 任务排队时间、执行时间、成功率 |
| `scripts/profile_endpoint.py` | 单端点火焰图 | 调用栈耗时 |
| `scripts/benchmark_image.py` | 测量 Docker 镜像大小与启动时间 | 镜像体积、冷启动到 `/health` 可用时间 |

`scripts/benchmark_api.py` 覆盖场景：
- 场景 1：50 并发，持续 60s，压测 `/health`、`/ready`、`/dashboard/summary`、`/studies`、`/personas`。
- 场景 2：10 并发，持续 60s，压测 `/personas/{id}/converse`（使用 mock LLM 服务端，避免真实 API 费用）。
- 场景 3：20 并发，持续 30s，压测 `/studies/{id}/analysis/{id}/status`（只读轮询）。

`scripts/benchmark_worker.py` 覆盖场景：
- 创建 study → 上传 150 样本数据集 → 触发 HB 分析 → 轮询至完成 → 记录总耗时、Worker CPU/内存峰值（通过 Prometheus 抓取）。

### 2.2 性能分析工具

- **py-spy**：对 API/Worker 进程采样，生成火焰图，识别 CPU 热点。
- **memray**：识别内存泄漏和高内存占用函数（重点看 `analysis/tasks.py` 和 `hb_engine.py`）。
- **Prometheus + Grafana**：记录测试期间 RPS、P95、队列长度、Worker 内存曲线。

### 2.3 基线报告模板

每次跑完基准后输出 Markdown 报告到 `reports/performance/YYYY-MM-DD-baseline.md`，包含：
- 环境信息（VM 规格、Docker 版本、依赖版本）。
- 测试配置（并发数、持续时间、数据集规模）。
- 关键指标表格。
- 火焰图/内存图截图路径。
- 已识别瓶颈 Top 5。

### 2.4 CI 集成

在 `.github/workflows/ci.yml` 新增 `performance-baseline` job：
- 仅在 PR 修改 `src/` 或 `docker/` 时触发。
- 启动 MongoDB/Redis service 容器，运行简化版基准（30s/低并发）。
- 将结果与 `reports/performance/latest-baseline.md` 对比，QPS 退化 > 15% 或 P95 上升 > 30% 时阻断合并。

## 三、代码层优化

### 3.1 Dashboard 聚合优化

**现状**：`/dashboard/summary` 每次查询所有研究（`page_size=100`），在 Python 中遍历统计状态并排序。

**优化**：
- 使用 MongoDB 聚合管道在数据库端完成计数，API 只读取聚合结果。
- 研究列表查询添加投影，仅返回 `study_id`、`product_category`、`status`、`created_at`。
- 缓存 TTL 从 10s 提升到 30s（读多写少场景）。

**影响文件**：
- `src/aicbc/monitoring/health.py`
- `src/aicbc/core/store_mongo.py`（新增 `alist_studies_summary` / `acount_studies_by_status`）
- `src/aicbc/core/cache.py`

### 3.2 画像批量生成异步化

**现状**：`/personas/generate` 同步生成所有画像，大 count 时长时间阻塞 HTTP 请求，且容易触发超时。

**优化**：
- 新增 Celery 任务 `run_persona_generation_task`，在 Worker 中执行批量生成。
- API 端点改为：校验参数 → 创建任务记录 → enqueue → 立即返回 `202 Accepted` + `task_id`。
- 前端轮询 `/personas/generation/{task_id}/status` 获取进度。
- 保留小批量（如 `count <= 5`）同步快速路径，避免小请求任务化开销。

**影响文件**：
- `src/aicbc/api/routes/personas.py`
- `src/aicbc/agents/consumer_generator.py`（新增支持 Celery 的批量生成入口）
- `src/aicbc/analysis/tasks.py`（新增任务）
- `src/aicbc/core/models/db_documents.py`（新增 `PersonaGenerationJobDocument`）

### 3.3 LLM 客户端异步化

**现状**：`LLMClient` 使用同步 `httpx.Client`，`generate_async` 通过 `asyncio.to_thread` 包装，仍有线程池开销。

**优化**：
- 新增 `httpx.AsyncClient` 用于异步路径。
- `generate_async` 直接走异步 HTTP，减少线程切换。
- 同步 `generate` 保留，兼容 Celery Worker 和现有同步调用方。
- 连接池参数可配置：`LLM_HTTP_MAX_CONNECTIONS`、`LLM_HTTP_MAX_KEEPALIVE`。

**影响文件**：
- `src/aicbc/llm/client.py`
- `src/aicbc/config/settings.py`

### 3.4 MongoDB 查询投影与连接池

**优化**：
- 在 `MongoPersonaStore.alist_all`、`MongoQuestionnaireStore.alist_studies`、`MongoAnalysisStore.alist_jobs_by_study` 中，当调用方只需要摘要信息时，使用投影减少 `data` 大字段传输。
- 为高频查询字段创建/确认索引：`persona_id`、`study_id`、`response_id`、`analysis_id`、`status`、`created_at`。
- 在 `settings.py` 中增加 MongoDB 连接池参数：
  - `MONGODB_MIN_CONNECTIONS=10`
  - `MONGODB_MAX_IDLE_TIME_MS=60000`
  - `MONGODB_WAIT_QUEUE_TIMEOUT_MS=5000`
  - `MONGODB_SERVER_SELECTION_TIMEOUT_MS=5000`

**影响文件**：
- `src/aicbc/core/store_mongo.py`
- `src/aicbc/core/models/db_documents.py`
- `src/aicbc/config/settings.py`

### 3.5 缓存策略增强

**优化**：
- 将 `TimedCache` 扩展为支持 Redis 作为可选后端（通过 `USE_REDIS_CACHE=true`），多 Pod 场景下共享缓存。
- 默认保持进程内缓存（B2ats 单 Pod 足够）。
- 对 `/studies`、`/personas` 列表增加 30s 缓存；写操作后显式失效。

**影响文件**：
- `src/aicbc/core/cache.py`
- 各路由文件中的缓存失效调用点

### 3.6 轻量优化项

- **GZip 压缩级别**：从 5 调到 6（CPU 增加可接受，响应体积再降 ~5%）。
- **中间件顺序调整**：`MetricsMiddleware` 放到 `RateLimitMiddleware` 之前，确保被限流请求也被计数。
- **慢请求阈值**：`SLOW_REQUEST_THRESHOLD` 从 5.0s 调到 2.0s（更敏感地发现退化）。

**影响文件**：
- `src/aicbc/main.py`
- `src/aicbc/config/settings.py`

## 四、部署与配置优化

### 4.1 API 与 Worker 镜像拆分（后续迭代）

**现状**：同一 Dockerfile 同时包含 API 和 Worker 所需依赖，API 镜像包含 pymc/pandas/numpy，体积大、启动慢。

**本轮做法**：
- 在 `docker/Dockerfile` 中增加多 target 结构（`api`、`worker`），为后续拆分做准备。
- **不实际切换** `docker-compose.yml` 引用，避免在代码优化尚未稳定时引入部署风险。
- 记录镜像体积基线，待代码稳定后再正式启用拆分。

**影响文件**：
- `docker/Dockerfile`（仅增加 target，不破坏现有构建）

### 4.2 资源限制补全

**优化**：
- 为 `beat` 服务补充资源限制：512M 内存 / 0.5 CPU。
- 为 `mongo`、`redis` 补充 CPU 限制：各 1.0 CPU。
- K8s `configmap.yaml` 中 `API_WORKERS` 保持 4，但增加注释说明可根据 Pod CPU limits 动态调整。

**影响文件**：
- `docker-compose.yml`
- `docker-compose.azure-b2ats.yml`
- `k8s/deployment.yaml`
- `k8s/configmap.yaml`

### 4.3 Graceful Shutdown

**现状**：`lifespan` 中 shutdown 仅打印日志，滚动更新时可能中断正在处理的 LLM 长连接请求。

**优化**：
- 在 `main.py` lifespan 中注册 `SIGTERM` 处理器，收到信号后设置 `app.state.shutting_down = True`。
- 中间件检测到 `shutting_down` 时返回 503（readiness probe 失败），停止接收新请求。
- 等待现有请求完成（最多 30s）后再关闭 MongoDB 连接。

**影响文件**：
- `src/aicbc/main.py`

### 4.4 环境变量与配置

新增/调整以下环境变量（均带默认值，向后兼容）：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `MONGODB_MIN_CONNECTIONS` | 10 | 最小连接池大小 |
| `MONGODB_MAX_IDLE_TIME_MS` | 60000 | 连接最大空闲时间 |
| `MONGODB_WAIT_QUEUE_TIMEOUT_MS` | 5000 | 等待连接超时 |
| `LLM_HTTP_MAX_CONNECTIONS` | 20 | LLM 异步 HTTP 连接池 |
| `USE_REDIS_CACHE` | false | 是否启用 Redis 共享缓存 |
| `DASHBOARD_CACHE_TTL` | 30 | Dashboard 缓存秒数 |
| `SLOW_REQUEST_THRESHOLD` | 2.0 | 慢请求阈值 |
| `API_GRACEFUL_SHUTDOWN_TIMEOUT` | 30 | 优雅关闭等待秒数 |

**影响文件**：
- `src/aicbc/config/settings.py`
- `.env.example`

## 五、监控与告警

### 5.1 Prometheus Recording Rules

新增 `docker/prometheus-rules.yml`：

```yaml
groups:
  - name: aicbc_performance
    interval: 15s
    rules:
      - record: aicbc:api_request_p95_seconds
        expr: histogram_quantile(0.95, rate(aicbc_api_request_duration_seconds_bucket[5m]))
      - record: aicbc:api_request_p99_seconds
        expr: histogram_quantile(0.99, rate(aicbc_api_request_duration_seconds_bucket[5m]))
      - record: aicbc:api_requests_per_second
        expr: rate(aicbc_api_requests_total[1m])
      - record: aicbc:llm_p95_seconds
        expr: histogram_quantile(0.95, rate(aicbc_llm_latency_seconds_bucket[5m]))
```

## 七、风险与回滚

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 异步化画像生成后前端未适配轮询 | 功能不可用 | 保留同步小批量路径；提供新端点但不删除旧字段 |
| LLM AsyncClient 与现有同步调用冲突 | Worker 或测试失败 | 同步 `generate` 保留，AsyncClient 仅用于 `generate_async` |
| MongoDB 投影遗漏字段导致序列化错误 | 部分端点 500 | 增加投影回归测试；逐步灰度放开 |
| 缓存 TTL 变长导致数据新鲜度问题 | 用户看到旧数据 | Dashboard 30s、列表 30s，写操作主动失效 |
| Graceful Shutdown 超时过短 | 长连接被强制关闭 | 默认 30s，可配置 |
| 监控告警误报 | 噪音 | 初始阶段告警先设为 warning，观察一周后再升 critical |

**回滚策略**：所有改动通过 feature branch 提交，每个任务独立 commit；如生产异常，优先回滚对应 commit，再排查问题。

## 八、实施顺序

1. **Phase 1：基准与监控**（1-2 天）
   - 编写基准脚本并建立 baseline。
   - 补齐 Prometheus rules、Alertmanager、Grafana dashboard。
   - 新增业务指标埋点。

2. **Phase 2：代码优化**（3-5 天）
   - Dashboard 聚合优化。
   - MongoDB 投影与连接池。
   - LLM AsyncClient。
   - 缓存策略增强。
   - 轻量优化项（GZip、中间件顺序、慢请求阈值）。

3. **Phase 3：部署配置与优雅关闭**（1-2 天）
   - 资源限制补全。
   - Graceful Shutdown。
   - Dockerfile target 准备（不启用）。
   - `.env.example` 更新。

4. **Phase 4：测试验证与 CI 集成**（2-3 天）
   - 单元/集成测试。
   - CI 性能回归门禁。
   - Staging 端到端验证。
   - 输出最终性能报告。
