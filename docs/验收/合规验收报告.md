## 二、逐项审查详情

### 2.1 优雅降级与容错

**审查标准**：依赖服务故障时核心流程是否可有限使用；熔断/重试/降级策略是否生效。

#### ✅ 已实现的措施

| 组件 | 实现 | 说明 |
|------|------|------|
| **LLM 调用重试** | `src/aicbc/llm/client.py` | 指数退避重试（`2^(attempt-1)` 秒），默认 `max_retries=3` |
| **成本熔断** | `src/aicbc/cost/fuse.py` | 五级熔断状态（NORMAL → WARNING → DEGRADE → FUSE → EMERGENCY），超预算时自动降级模型或阻断调用 |
| **模型降级** | `CostFuse.pre_call_check()` | 预算达 95% 时自动切换至轻量模型（`claude-haiku-4-5`） |
| **调用阻断** | `CostFuseError` | 预算达 100% 时阻断非紧急调用，达 120% 进入 EMERGENCY 锁定 |
| **LRU 缓存** | `LLMClient` | 128 条线程安全缓存，命中缓存时不触发成本 |
| **全局异常处理** | `src/aicbc/main.py` | `global_exception_handler`：生产环境仅返回 `"Internal server error"`，防止信息泄露 |
| **速率限制** | `RateLimitMiddleware` | 全局 120 req/min + 敏感端点 10 req/min |
| **安全响应头** | `SecurityHeadersMiddleware` | X-Content-Type-Options、X-Frame-Options、X-XSS-Protection、Referrer-Policy、Permissions-Policy |
| **生成层容错** | `ProfileGenerator` | 单层解析失败时回退至静态 fallback 字典，不中断整体流程 |

#### ⚠️ 存在的缺陷

| 缺陷 | 位置 | 风险 | 修复建议 |
|------|------|------|---------|
| **健康检查为占位实现** | `src/aicbc/monitoring/health.py` | 假阳性风险：MongoDB、Redis、LLM API 的健康检查均为 `TODO` 注释，实际只检查 API Key 是否存在，无法检测真实服务不可用 | 立即实现真正的 `ping` 逻辑；在依赖不可用时 `ReadinessStatus` 应返回 `not_ready` |
| **缺少熔断器（Circuit Breaker）** | `src/aicbc/llm/client.py` | 连续失败时仍机械重试，可能加剧 LLM 服务端压力 | 为 Anthropic/OpenAI 调用增加 Circuit Breaker（如连续 5 次失败则进入 60s 半开状态） |
| **缺少非 LLM 降级路径** | 全系统 | LLM 完全不可用时，系统无任何备用方案（如规则模板、缓存回放） | 设计「离线模式」：当 EMERGENCY 触发时，可基于历史模板批量生成基础画像 |
| **CSP 响应头缺失** | `SecurityHeadersMiddleware` | 存在 XSS 载荷注入风险 | 增加 `Content-Security-Policy` 头 |

**判定**：⚠️ **有条件通过**。核心熔断/重试机制已生效，但健康检查造假、缺 Circuit Breaker 需在阶段 4 前修复。

### 2.2 国际化（i18n）

**审查标准**：多语言支持是否完整，切换流程是否顺畅。

#### 🔴 阻塞级发现

| 发现项 | 证据 | 合规影响 |
|--------|------|---------|
| **前端语言硬编码为中文** | `frontend/src/main.tsx` 仅导入 `zhCN`；`antd` 的 `ConfigProvider` 未配置多语言切换 | 系统无法支持非中文用户 |
| **HTML lang 属性固定** | `frontend/index.html` `<html lang="zh-CN">` 无动态切换能力 | 屏幕阅读器、搜索引擎对语言判定错误 |
| **无 i18n 框架** | 未使用 `react-i18next`、`lingui` 或任何国际化库 | 所有界面文本为硬编码中文，无法翻译 |
| **后端无本地化** | API 返回的错误消息、偏见审计报告均为中文硬编码 | 非中文业务用户无法理解审计结果 |
| **无语言切换 UI** | 全系统未见语言选择器或 `Accept-Language` 处理 | 无法按需切换 |

**判定**：🔴 **不通过，一票否决风险**。系统定位为「通用联合分析平台」，但当前为纯中文单语言实现。根据 `docs/系统成功标准书.md` 的「通用能力」要求，系统需具备跨产品、跨人群适配能力，国际化是其中隐含要求。建议在阶段 4 前引入 `react-i18next` 并完成中英双语覆盖。

### 2.3 SEO 与社交分享

**审查标准**：服务端渲染/预渲染效果；Meta 标签与 Open Graph 配置。

#### 🔴 阻塞级发现

| 发现项 | 证据 | 合规影响 |
|--------|------|---------|
| **纯 SPA 无 SSR/预渲染** | `frontend/index.html` + Vite 构建，无 Next.js/Nuxt/SSR 方案 | 搜索引擎爬虫难以索引动态内容（画像详情、分析报告） |
| **缺少 Meta Description** | `index.html` 无 `<meta name="description">` | 搜索摘要为空，降低点击率 |
| **缺少 Open Graph 标签** | 无 `og:title`、`og:description`、`og:image`、`og:url` | 社交分享时卡片信息缺失，影响专业传播 |
| **缺少 Twitter Card 标签** | 无 `twitter:card` 系列标签 | 无法在 Twitter/LinkedIn 生成预览卡片 |
| **缺少 Canonical URL** | 无 `<link rel="canonical">` | 重复内容 SEO 惩罚风险 |
| **无 robots.txt / sitemap.xml** | 未见相关文件 | 搜索引擎抓取策略不可控 |
| **无语义化路由标题** | `PersonaDetail.tsx` 等页面未动态更新 `document.title` | 所有页面共享同一标题，SEO 权重分散 |

**判定**：🔴 **不通过，一票否决风险**。作为面向业务用户的「决策平台」，报告和画像详情页面需要可被分享和检索。当前 SEO 状态相当于「不可被搜索引擎发现」。建议：
1. 为 `index.html` 补充完整 Meta 和 OG 标签；
2. 为关键页面（画像详情、分析报告）增加动态 `document.title` 和 `meta` 更新；
3. 考虑为公开报告页引入静态预渲染或 Prerender 中间件。

### 2.5 公平性合规

**审查标准**：6 条公平性硬规则是否嵌入；偏见审计是否自动化。

#### ✅ 已实现的措施

| 措施 | 实现 | 状态 |
|------|------|------|
| **24 模式刻板印象库** | `src/aicbc/core/scoring/stereotype_patterns.py` | 24 个模式覆盖性别(6)、年龄(5)、地域(4)、职业(4)、民族(3)、收入(2)，全部对齐 `docs/虚拟消费者公平性规范.md` 附录 B |
| **偏见审计器** | `src/aicbc/core/scoring/bias_auditor.py` | 实现关键词匹配 + 语言偏见检测 + 多样性检测，支持单画像和批量审计 |
| **自动化审计入口** | `ConsumerGeneratorAgent` 集成 `BiasAuditor` | 生成后自动审计（需确认是否真正执行） |
| **统计阈值对齐** | `src/aicbc/config/settings.py` | `cramers_v_threshold=0.2`，与规范 v1.1 一致 |
| **偏见审计测试** | `tests/test_bias_audit_batch.py` | 24 画像批量审计测试通过，覆盖 CRITICAL/HIGH/LOW 级别 |
| **真实性评分** | `src/aicbc/core/scoring/authenticity_scorer.py` | 7 维度评分（内在一致性、情境敏感性、认知有限性、社会摩擦感、时间延续性、语言自然度、知识边界感） |

#### 🔴 阻塞级发现

| 发现项 | 证据 | 合规影响 |
|--------|------|---------|
| **6 条公平性硬规则未嵌入 Prompt 模板** | `configs/prompts/persona_generation.txt` 中**完全未出现**任何公平性约束、性别解耦、地域中性、年龄中性等指令；`ConsumerGeneratorAgent` 的 `RuleInjection` 也未包含公平性硬规则 | `docs/系统成功标准书.md` v1.2 §4.9 明确将「6条公平性硬规则已全部嵌入消费者生成Prompt」列为 **Must Have + 一票否决** 项。当前缺失此嵌入，构成直接一票否决条件 |
| **Prompt 泄漏检测与偏见防护冲突** | `LLMClient._detect_prompt_leakage()` 会拦截包含「系统指令」「约束条件」等关键词的响应，但公平性约束的 Prompt 注入若包含这些关键词，可能被误判为泄漏 | 需验证泄漏检测白名单是否包含公平性约束文本 |
| **多样性检测仅覆盖「平均化模板」** | `BiasAuditor._check_demographic_diversity()` 仅检查 5 个 demographic 字段的「平均人」模板，未实现规范要求的 `age_entropy >= 0.7`、`gender_balance 0.4-0.6`、`city_tier_coverage >= 3` 等批次级统计检验 | 无法自动化满足规则 6（多样性保障）的量化要求 |
| **无卡方/KS 统计检验实现** | `bias_auditor.py` 中仅有关键词匹配和简单计数，未见规范要求的卡方独立性检验、KS 检验、Cramér's V 计算、Shannon 熵计算 | 无法自动化满足规则 1-5 的统计检验要求，只能依赖事后人工审查 |

**判定**：🔴 **不通过，一票否决风险**。

这是本次审查中最严重的合规缺陷。虽然偏见模式库和审计器实现质量很高，但**公平性硬规则未嵌入生成 Prompt** 意味着偏见检测是「事后发现」而非「事前预防」。根据 `docs/系统成功标准书.md` v1.2 §4.9 和 §10.2，此缺陷直接触发小伦的一票否决权。

**强制修复清单**：
1. 在 `configs/prompts/persona_generation.txt` 中增加「公平性与偏见防护指令」章节（参考 `docs/伦理与偏见审计规范.md` §2.2）；
2. 在 `ConsumerGeneratorAgent` 的 `RuleInjection` 中显式增加 6 条硬规则约束；
3. 在 `BiasAuditor` 中实现批次级统计检验（卡方、KS、Cramér's V、Shannon 熵）；
4. 在生成流程中增加「偏见审计前置闸口」：单画像未通过偏见审计时，自动触发重新生成（`reject_and_regenerate`）。

| 风险 | 说明 | 缓解建议 |
|------|------|---------|
| **无人工审核闸口** | 当前流程为「生成 → 自动审计 → 入库」，无强制人工复核环节 | 在阶段 4 业务验收前，增加「小伦审核通过」人工闸口 |
| **刻板印象模式库依赖关键词匹配** | 未使用 Embedding 语义相似度（文档要求但未实现），新型偏见可能绕过关键词检测 | 引入 Embedding 语义扫描作为第二道防线 |
| **「宝妈」细分群体设计风险** | `docs/项目成功标准书.md` §3.2 将「宝妈」作为目标人群，但 `docs/虚拟消费者公平性规范.md` 明确将「宝妈容易被种草」列为违规示例 | 需在生成 Prompt 中增加对「宝妈」群体的特殊公平性约束，避免将婚姻/育儿与消费动机过度绑定 |
| **偏见审计通过 ≠ 伦理无害** | 审计器只能检测已知模式，无法覆盖全部伦理风险 | 建立「伦理红队」机制，定期人工注入边缘案例进行对抗测试 |

**判定**：⚠️ **有条件通过**。伦理框架文档完备，但自动化审计缺少语义层检测，且缺少人工审核闸口。建议在阶段 4 前引入红队测试机制。

## 四、合规验收判定
