| 检查项 | 是否拦截 | 说明 |
|--------|---------|------|

| 提交消息格式违规 | ✅ 是 | `exit 1` |

| bandit 发现高危漏洞 | ❌ 否 | `|| true` 掩盖失败，仅生成报告 |
| pip-audit 发现依赖漏洞 | ❌ 否 | `|| true` 掩盖失败 |
| Trivy 发现 CRITICAL/HIGH 漏洞 | ❌ 否 | `exit-code: '0'` |
| 单元测试失败 | ✅ 是 | pytest 默认会失败 |
| 覆盖率 < 60% | ✅ 是 | `--cov-fail-under=60` 会失败（但当前 28% 根本无法通过） |

**关键问题**：
1. **当前 CI 实际上无法通过**：因为覆盖率 28% 低于 60%，`pytest --cov-fail-under=60` 会在每次 CI 运行中失败。如果这是期望的，说明团队已知差距并正在改进；如果不是，CI 已被破坏。
2. **安全扫描形同虚设**：`bandit`、`pip-audit`、`Trivy` 都使用了 `|| true` 或 `exit-code: '0'`，高危漏洞不会阻止构建。这不符合安全最佳实践。
3. **没有集成测试阶段**：`test_integration_pipeline.py` 和 `test_e2e_full_pipeline.py` 在 CI 中与普通单元测试一起运行，没有区分阶段，也没有使用更真实的环境（如临时数据库）。
4. **没有前端 CI 阶段**：`frontend/` 目录在 CI 中完全未被构建或测试，前端代码变更无法被验证。

- **ruff 配置优秀**：规则选择合理，覆盖了代码风格、导入排序、命名规范、bug 预防、简化建议。

- **建议补充**：创建 `.pre-commit-config.yaml`，配置 ruff、mypy、bandit 的 pre-commit hooks，与 CI 保持一致。

- 需要排查 `.coverage` 数据收集问题，确保 `source = ["src"]` 配置正确，且没有 `concurrency` 或 `parallel` 问题导致数据丢失。

### 6.1 前后端覆盖率缺口概述

对 AI_CBC 全部后端 API 端点与前端页面的调用关系进行逐端点对照审计后，发现**前后端 API 契约存在严重断层**：

| 指标 | 数值 | 风险评级 |
|------|------|---------|
| 后端总端点数 | **31** | — |
| 前端已调用端点数 | **19** | — |
| 前端未调用端点数 | **12** | 🔴 **高** |
| 覆盖率 | **~61%** | 🔴 **未达标** |
| 已调用但参数僵化端点数 | **5** | 🔴 **高** |
| 前端 E2E 测试覆盖端点数 | **0** | 🔴 **极高** |

**模块级覆盖率分布**：

| 模块 | 后端端点数 | 前端已使用 | 前端未暴露 | 覆盖率 |
|------|-----------|-----------|-----------|--------|
| 问卷配置 | 6 | 3 | 3 | 50% |
| 画像管理 | 6 | 3 | 3 | 50% |
| 行为模拟 | 3 | 1 | 2 | 33% |
| 作答模拟 | 3 | 2 | 1 | 67% |
| 分析管线 | 9 | 8 | 1 | 89% |
| 系统运维 | 4 | 2 | 2 | 50% |
| **合计** | **31** | **19** | **12** | **~61%** |

**核心结论**：
1. **12 个端点完全未暴露**：后端已实现的功能（如画像重新验证、批量采访、购买决策模拟、逐条作答记录查看、 dashboard 聚合端点等）前端无任何入口。
2. **5 个已使用端点存在参数硬编码**：前端将关键参数（`model_type`、`mode`、`attributes`、`design_parameters`、`test_type`）写死，后端灵活性被前端架空。
3. **前端零测试**：由于前端没有任何测试框架，上述 19 个已调用端点的前端调用逻辑**没有任何自动化测试覆盖**，参数僵化问题在现有测试中完全无法被发现。

### 6.3 参数僵化导致的测试盲区分析

除端点缺失外，以下 5 个已使用端点的前端实现将关键参数硬编码，导致后端参数校验逻辑和灵活性在现有测试中无法被验证：

| 页面 | 端点 | 硬编码参数 | 后端实际支持 | 测试盲区 |
|------|------|-----------|------------|---------|
| `StudyCreate.tsx` | `POST /studies` | 不传 `attributes`/`design_parameters` | 完整自定义属性 + 设计参数 | 后端对自定义属性的校验逻辑（边界值、必填项、格式）在前端调用路径中从未被触发 |
| `AnalysisStatus.tsx` | `POST .../analyze` | `model_type: 'hb'` | `hb` / `mnl` / `latent_class` | 后端对 `mnl` 和 `latent_class` 的分析流程在前端调用路径中从未被测试 |
| `ResponseSimulator.tsx` | `POST .../simulate-responses` | `mode: 'stochastic'` | `stochastic` / `llm` / `deterministic` | 后端对 `llm` 和 `deterministic` 模式的响应构造逻辑在前端调用路径中未验证 |
| `SegmentComparison.tsx` | `GET .../segment-comparison` | 测试类型硬编码 | `hotelling` / `welch` / `permutation` | 后端多检验类型的差异计算逻辑在前端调用路径中未验证 |
| `InterviewLab.tsx` | `POST .../converse` | 只用单轮对话 | 支持多轮 `interview` + `purchase-decision` | 后端批量采访和购买决策的状态追踪逻辑在前端调用路径中未验证 |

**测试盲区的影响**：
1. **后端参数校验缺陷被掩盖**：如果后端对 `model_type='mnl'` 的处理存在 bug，现有后端单元测试可能通过（因为测试直接构造请求），但前端硬编码 `'hb'` 导致该 bug 在生产环境中永远不会被触发——直到某天需要支持 `mnl` 时才会暴露。
2. **接口契约退化风险**：前端不传 `attributes`/`design_parameters`，后端虽然支持，但前端调用路径从未验证过这些字段的传递、解析和持久化。当未来需要开放这些字段时，可能发现前后端契约早已不一致。
3. **测试金字塔底部失效**：现有 28% 的覆盖率集中在后端单元测试，但单元测试只能验证“模块在隔离环境下正确”，无法验证“模块在实际前端调用路径下正确”。参数僵化导致前端调用路径与后端单元测试的输入空间严重偏离，形成了**测试金字塔底部的虚假安全感**。

### 6.4 测试策略改进建议

#### 6.4.1 引入前端组件测试覆盖端点调用逻辑

| 建议项 | 具体方案 | 优先级 |
|--------|---------|--------|
| **前端单元测试** | 引入 Vitest + React Testing Library，对以下组件进行重点测试：① `StudyCreate.tsx` 的表单提交逻辑（验证是否正确构造 `attributes`/`design_parameters`）；② `AnalysisStatus.tsx` 的分析触发逻辑（验证 `model_type` 是否从 UI 状态正确传递）；③ `ResponseSimulator.tsx` 的模拟模式选择逻辑（验证 `mode` 参数是否从下拉框正确传递） | 🔴 P0 |
| **前端 API 集成测试** | 使用 MSW（Mock Service Worker）拦截前端 API 调用，验证请求参数构造、响应处理、错误状态流转。重点覆盖 5 个参数僵化页面。 | 🔴 P0 |
| **前端 E2E 测试** | 引入 Playwright 或 Cypress，覆盖完整的用户链路（创建 Study → 配置属性 → 生成问卷 → 生成画像 → 运行分析 → 查看结果），在真实浏览器中验证端点调用。 | 🟡 P1 |

#### 6.4.2 引入契约测试确保前后端契约一致

| 建议项 | 具体方案 | 优先级 |
|--------|---------|--------|
| **OpenAPI 契约验证** | 生成后端的 OpenAPI 规范（FastAPI 原生支持），使用 `schemathesis` 或 `dredd` 对全部 31 个端点进行契约验证，确保请求/响应 schema 与实现一致。 | 🔴 P0 |
| **Pact 消费者驱动契约测试** | 将前端定义为“消费者”，后端为“提供者”。前端测试中生成 Pact 契约文件，CI 中验证后端是否满足全部契约。重点覆盖 12 个未使用端点——即使前端暂不调用，也应通过契约测试明确“这些端点的前端期望行为”。 | 🟡 P1 |
| **参数僵化专项修复** | 对 5 个硬编码参数页面进行前端改造，开放参数选择 UI，并在改造后立即补充对应的组件测试和契约测试。 | 🔴 P0 |

#### 6.4.3 将 API 覆盖率作为新的质量门禁

| 建议项 | 具体方案 | 优先级 |
|--------|---------|--------|
| **API 覆盖率指标定义** | 定义“API 覆盖率 = 前端实际调用的后端端点数 / 后端总端点数”，当前目标：从 61% 提升至 ≥ 90%；已调用端点的“参数自由度”从 0%（当前全部硬编码）提升至 ≥ 80%（关键参数可通过 UI 控制）。 | 🔴 P0 |
| **CI 门禁配置** | 在 CI 中增加前端构建和测试阶段，要求：① 前端组件测试通过；② Playwright E2E 关键链路通过；③ API 覆盖率 ≥ 90%。 | 🔴 P0 |
| **未使用端点清理策略** | 对 12 个未使用端点进行“保留/废弃/延迟开发”决策：若保留，则必须补充前后端契约测试；若废弃，则从后端路由中移除，避免维护负担。 | 🟡 P1 |

#### 6.5.3 对验收结论的修正

**原结论**（2026-06-13）：
> 当前测试体系不满足上线要求，需完成整改后方可通过验收。核心缺口：覆盖率 28%（目标 60%）、前端零测试、缺少 pre-commit。

**修正后结论**：
> 当前测试体系**严重不满足上线要求**。除原有的覆盖率不足（28% vs 60%）和前端零测试外，新增发现的前后端 API 覆盖率缺口（61%）和参数僵化问题（5 个硬编码端点）揭示了**测试体系存在系统性的“虚假覆盖”风险**——后端测试覆盖了一些端点，但前端不调用；前端调用了一些端点，但参数被硬编码。这导致大量后端逻辑（多模型分析、多模式模拟、自定义属性配置、批量采访、购买决策追踪）在**实际的端到端链路中完全未被验证**。
>
> **必须完成的整改项**：
> 1. 修复 5 个参数硬编码问题，开放后端能力给前端 UI；
> 2. 引入前端组件测试（Vitest + RTL）覆盖端点调用逻辑；
> 3. 引入契约测试（OpenAPI 验证 / Pact）覆盖全部 31 个端点；
> 4. 将 API 覆盖率（≥ 90%）和参数自由度（≥ 80%）作为新的质量门禁；
> 5. 对 12 个未使用端点进行保留/废弃决策，并补充对应测试。
>
> 在完成上述整改并重新验收前，**不建议将系统投入生产环境**。
