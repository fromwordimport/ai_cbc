## 跨模块关联问题

以下问题横跨多个模块，单一模块修复无法根治，需**跨模块协调**统一修复：

### 关联 1：内存存储与限流器（数据层 ↔ 服务端 ↔ 性能 ↔ 安全 ↔ 可观测性）

- **表现**：`core/store.py` 纯内存存储、`InMemoryRateLimiter` 内存级限流、`cost_state.json` 本地文件状态
- **影响**：数据层无持久化（D-001）、服务端高并发阻塞（S-004）、性能多实例限流失效（PERF-002）、安全 DDoS 可绕过（SEC-W01）、可观测性多副本成本状态不一致（OBS-003）
- **根治方案**：统一引入 **Redis** 作为共享状态层（缓存 + 限流 + 成本状态 + 分析任务状态机）；同步引入 **MongoDB**（motor/beanie）作为持久化层。两者必须在同一迭代中完成，否则限流和成本状态在"内存→Redis"迁移后仍缺乏持久化兜底。
- **牵头人**：后端架构（小示/小数）+ DevOps（小维）
- **预估工时**：2-3 周（含数据迁移脚本 + 回滚方案）

### 关联 2：API Key 硬编码与认证绕过（前端 ↔ 服务端 ↔ 安全）

- **表现**：前端 `api.ts` 硬编码 `dev-key-change-in-prod`；服务端 `settings.py` 默认同样密钥；`main.py` debug 模式完全绕过认证
- **影响**：前端源码泄露直接暴露密钥（F-001）；服务端默认值未覆盖则认证失效（S-002）；安全测试证实全端点无认证（SEC-001）
- **根治方案**：① 前端改为环境变量注入（`import.meta.env.VITE_API_KEY`）；② 服务端移除默认值，启动时强制校验；③ 认证中间件与 `debug` 解耦；④ 生产环境禁用 `DEBUG=true`；⑤ CI 中增加 `detect-secrets` 扫描。
- **牵头人**：后端安全（小安）+ 前端（小前）+ DevOps（小维）
- **预估工时**：1 天（代码改动）+ 0.5 天（CI 集成）

### 关联 3：健康检查虚假通过（性能 ↔ 数据层 ↔ 可观测性 ↔ 合规）

- **表现**：`monitoring/health.py` 中 MongoDB/Redis/LLM 检查均为 TODO 占位，仅返回 `status="ok"`
- **影响**：性能维度故障流量无法切断（PERF-005）；数据层假阳性导致备份/恢复策略失效（D-W01）；可观测性缺失真实依赖健康指标；合规维度容错机制失效（COMP-W01）
- **根治方案**：完成 motor/aioredis 客户端接入后，统一替换 TODO 桩代码；实现真实的异步 ping；Readiness 失败时返回 `503`；K8s 探针根据 Readiness 自动切断流量。
- **牵头人**：后端架构（小数）+ DevOps（小维）
- **预估工时**：4h（代码实现）+ 2h（K8s 探针调优）

### 关联 4：分析任务状态一致性（数据层 ↔ 服务端 ↔ 性能）

- **表现**：`AnalysisStore` 状态机仅在内存中维护；Celery Worker 多实例无法共享状态
- **影响**：数据层分布式一致性缺失（D-003）；服务端大批量同步阻塞（S-004）导致分析任务队列堆积；性能维度无 HPA 验证（PERF-004）
- **根治方案**：将 `analysis_jobs` 集合作为唯一真相源；MongoDB 原子操作实现乐观锁；Celery Worker 通过数据库轮询获取任务；同步为 Worker 增加基于队列长度的 HPA（KEDA）。
- **牵头人**：后端架构（小数）+ DevOps（小维）
- **预估工时**：3-5 天（状态机迁移）+ 3 天（KEDA HPA）

### 关联 5：安全与合规的 Prompt 治理（安全 ↔ 合规 ↔ 服务端）

- **表现**：系统 Prompt 未做抗提取加固；公平性硬规则未嵌入生成 Prompt；静态黑名单可被注入绕过
- **影响**：安全维度 Prompt 泄露与注入绕过（SEC-004）；合规维度公平性一票否决（COMP-001）；服务端 `ConsumerGeneratorAgent` 的 `RuleInjection` 缺失约束
- **根治方案**：① 在 `persona_generation.txt` 中增加公平性约束章节；② 在 `ConsumerGeneratorAgent` 的 `RuleInjection` 中显式增加 6 条硬规则；③ 将输入清洗从静态黑名单升级为语义检测 + 结构化输入隔离；④ 在系统 Prompt 末尾增加抗提取防御指令；⑤ 输出层增加 `OutputContentFilter`。
- **牵头人**：Prompt 规则（小示）+ 伦理审计（小伦）+ 安全（小安）
- **预估工时**：1-2 周（Prompt 重构 + 统计检验实现 + 红队测试回归）

### 关联 6：前后端 API 覆盖率缺口（前端 ↔ 服务端 ↔ 测试）

- **表现**：后端 31 个 API 端点中 12 个完全未在前端暴露；已使用端点中 `model_type`、`mode`、`attributes` 等关键参数被硬编码
- **影响**：后端已修复/已实现的功能（如自定义属性、MNL/Latent Class 模型、批量采访、购买决策模拟）因前端不可见而无法被用户触达；分析管线、问卷配置、行为模拟的灵活性被严重架空
- **根治方案**：① 补齐 12 个未暴露端点的前端入口（页面、按钮、表单）；② 为已使用端点增加参数选择器（`model_type`、`mode`、`test_type`、`attributes`）；③ 引入前后端契约测试（Pact / OpenAPI 验证器），确保 API 变更时自动捕获契约破坏
- **牵头人**：前端（小前）+ 后端架构（小数）+ 测试（小测）
- **预估工时**：1-2 周（前端页面补齐 + 参数选择器重构 + 契约测试接入）
- **参考**：`docs/superpowers/review/2026-06-13-前后端API覆盖率缺口审查报告.md`
