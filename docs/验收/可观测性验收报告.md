## 2. 逐维度审查详情

### 2.1 集中化日志

#### 审查发现
- **日志框架**：项目使用 `structlog`（≥24.2.0）作为结构化日志库，所有核心模块均通过 `structlog.get_logger("aicbc.xxx")` 获取 logger，命名空间规范。
- **日志配置**：**未找到 `structlog.configure()` 或等效配置**。当前依赖 structlog 的默认配置，输出格式、处理器、级别未做统一编排，可能导致不同环境（本地 / Docker / K8s）输出格式不一致。
- **日志聚合**：**无 Loki、ELK、Fluentd、Splunk 等日志聚合平台**。`docker-compose.yml` 中仅将 `./logs:/app/logs` 挂载到容器本地目录，日志停留在文件级，无法跨多副本/多节点统一检索。
- **Celery Worker / Beat 日志**：Worker 和 Beat 服务仅通过 `--loglevel=info` 输出到 stdout，未与 API 服务日志做统一收集。

#### 问题清单
| 编号 | 问题 | 严重程度 | 建议 |
|------|------|----------|------|
| LOG-01 | 缺少 `structlog` 全局配置（处理器、渲染器、过滤器） | 中 | 在 `src/aicbc/main.py` 或独立 `logging_config.py` 中显式配置 structlog，确保 JSON 输出与时间戳统一 |
| LOG-02 | 无日志聚合与搜索平台 | 高 | 引入 Grafana Loki（与现有 Grafana 同栈）或对接云日志服务；所有服务（api/worker/beat）日志通过 `promtail` 或 `fluent-bit` 采集 |
| LOG-03 | 日志未携带统一 `trace_id` / `request_id` | 中 | 在中间件中生成 `request_id` 并通过 structlog 的 `threadlocal`/`contextvars` 绑定到所有日志条目，便于关联排查 |

### 2.2 应用性能监控（APM）

#### 审查发现
- **指标框架**：使用 `prometheus-client`（≥0.20.0），采用懒加载（`lru_cache` + 动态导入）策略避免 Windows 启动时 C 扩展崩溃，设计合理。
- **指标命名**：遵循 `aicbc_<domain>_<metric>_<unit>` 规范，已定义 14 个指标，覆盖业务、系统、安全三大域。
- **中间件**：`MetricsMiddleware` 已注册于 `main.py`，自动采集所有 HTTP 请求的 `duration` 和 `count`；`SecurityHeadersMiddleware` 补充安全头。
- **Grafana 仪表盘**：`docker/grafana/dashboards/aicbc-dashboard.json` 提供 9 个面板，涵盖 API 请求量/错误率、P95 延迟、LLM 调用与费用、消费者生成成功率、真实性评分、研究成本趋势、安全事件、任务队列长度、Token 使用量。仪表盘时区设置为 `Asia/Shanghai`，刷新周期 30s，符合国内场景。
- **Prometheus 配置**：`docker/prometheus.yml` 包含 `aicbc-api`、`prometheus`、`node-exporter`、`mongodb-exporter`、`redis-exporter`、`blackbox-http` 6 个 job，抓取间隔 15s，评估间隔 15s。
- **健康探针**：`blackbox-exporter` 对 `/health` 和 `/ready` 做 HTTP 探测，利于外部可用性监控。

#### 关键缺陷：指标埋点未接通
| 函数 | 定义位置 | 被调用位置 | 状态 |
|------|----------|------------|------|
| `record_llm_call()` | `metrics.py:175` | **无** | ❌ 未使用 |
| `record_persona_generation()` | `metrics.py:191` | **无** | ❌ 未使用 |
| `record_api_request()` | `metrics.py:205` | `middleware.py:45` | ✅ 已使用 |
| `record_security_event()` | `metrics.py:220` | `middleware.py:31` | ✅ 已使用 |

**影响**：
- `aicbc_llm_api_calls_total`、`aicbc_llm_tokens_total`、`aicbc_llm_latency_seconds` 等核心 LLM 指标在 Grafana 中**永远为零**。
- `aicbc_personas_generated_total`、`aicbc_cost_per_persona_cny`、`aicbc_authenticity_score_average` 等核心业务指标**永远为零**。
- 仪表盘虽配置美观，但大量面板为“空数据”状态，失去 APM 价值。

#### 问题清单
| 编号 | 问题 | 严重程度 | 建议 |
|------|------|----------|------|
| APM-01 | `record_llm_call()` 未被调用 | **高** | 在 `LLMClient`（`src/aicbc/llm/client.py`）完成每次 API 调用后，调用 `record_llm_call()` 上报模型、Token、耗时 |
| APM-02 | `record_persona_generation()` 未被调用 | **高** | 在 `personas` 路由或 `profile_generator` 生成完成后调用，上报状态、成本、真实性评分 |
| APM-03 | 缺少数据库查询耗时指标 | 中 | 增加 `aicbc_db_query_duration_seconds`（按操作类型 `find/insert/update/aggregate` 打标签），在 MongoDB/Motor 操作层埋点 |
| APM-04 | 缺少 Celery Worker 任务指标 | 中 | 通过 `celery-prometheus-exporter` 或自定义 Counter/Gauge 采集任务执行时间、失败率、重试次数 |
| APM-05 | Prometheus 引用未部署的 exporter | 低 | `node-exporter`、`mongodb-exporter`、`redis-exporter` 在 `docker-compose.yml` 中未定义，会导致 Prometheus 目标 Down；建议补齐服务或注释掉未启用的 job |

### 2.3 分布式链路追踪

#### 审查发现
- **追踪系统**：**完全缺失**。代码中未引入 `opentelemetry`、`jaeger-client`、`zipkin` 等任何分布式追踪库。
- `ToolCaller` 中的 `enable_tracing`：仅为**本地调用历史记录**（`self._call_history`），记录 call_id、tool_name、duration，不具备跨进程/跨服务传播能力。
- `ToolCallRecord` 虽有 `parent_call_id` 字段，但**无分布式 trace_id / span_id 生成与传递机制**。
- `main.py` 的 middleware 链中无追踪中间件，HTTP Header 未注入/提取 `traceparent`。
- 微服务架构（API → Worker → MongoDB/Redis/LLM）下，一次请求经过多个进程，无法追踪完整调用链。

#### 问题清单
| 编号 | 问题 | 严重程度 | 建议 |
|------|------|----------|------|
| TRACE-01 | 无分布式链路追踪实现 | **高** | 引入 `opentelemetry-distro` + `opentelemetry-instrumentation-fastapi` + `opentelemetry-instrumentation-celery`，自动采集 HTTP 与 Celery 调用链 |
| TRACE-02 | 无 trace_id 注入日志 | 高 | 在 OpenTelemetry 自动注入基础上，将 `trace_id` / `span_id` 通过 structlog 处理器写入日志，实现“日志-追踪-指标”三联 |
| TRACE-03 | LLM 调用无 Span 覆盖 | 中 | 为 `LLMClient` 的每次调用手动创建 Span，记录模型、Token、异常信息，便于定位慢调用与失败根因 |
| TRACE-04 | 无追踪数据存储与可视化 | 高 | 部署 Jaeger Collector + Query（或 Grafana Tempo），与现有 Grafana 集成展示调用拓扑图 |

### 2.4 智能告警

#### 审查发现
- **Alertmanager**：`docker/prometheus.yml` 中 `alerting.alertmanagers` 的 `targets` 为空列表，且 `docker-compose.yml` 中未部署 `alertmanager` 服务。告警通知渠道**完全未建立**。
- **告警规则**：`rule_files` 指向 `/etc/prometheus/rules/*.yml`，但项目目录中**不存在 `docker/prometheus/rules/` 目录**，无任何规则文件。
- **阈值设置**：无 Prometheus 告警表达式，无 P95 延迟、错误率、成本、队列积压等阈值策略。
- **成本熔断通知**：`CostTracker.notify_if_changed()` 仅通过 `structlog` 输出日志（`cost_fuse_warning`、`cost_fuse_degrade_model`、`cost_fuse_triggered`），**无邮件/钉钉/飞书/企业微信/Webhook 等外部通知**。若运维人员不实时查看日志，无法感知熔断事件。
- **健康检查**：`docker-compose.yml` 中各服务均配置了 `healthcheck`，但健康检查失败仅由 Docker 处理重启，未触发告警。

#### 问题清单
| 编号 | 问题 | 严重程度 | 建议 |
|------|------|----------|------|
| ALERT-01 | Alertmanager 未部署 | **高** | 在 `docker-compose.yml` 中增加 `alertmanager` 服务，配置邮件/IM Webhook 接收器 |
| ALERT-02 | 告警规则文件缺失 | **高** | 创建 `docker/prometheus/rules/aicbc-alerts.yml`，至少包含：<br>1. `APIErrorRate > 5%`（5m 窗口）<br>2. `APIP95Latency > 5s`（对应 `settings.slow_request_threshold`）<br>3. `LLMCallFailureRate > 10%`<br>4. `CostFuseStatus == FUSE/EMERGENCY`<br>5. `QueueSize > 100` |
| ALERT-03 | 成本熔断无外部通知 | **高** | 在 `CostTracker.notify_if_changed()` 中集成 Webhook 通知（或调用 Alertmanager API push），确保预算耗尽时即时通知到值班通道 |
| ALERT-04 | 无告警抑制与分组 | 中 | 在 Alertmanager 配置中设置 `group_by=['alertname','environment']` 与 `repeat_interval: 4h`，防止告警风暴 |

### 2.5 成本可观测性

#### 审查发现
- **成本追踪器**：`CostTracker` 设计优秀，具备以下能力：
  - 五维核算：`per_study` / `daily` / `weekly` / `monthly` / `global`
  - 持久化：`./data/cost_state.json` 保存状态，进程重启不丢失
  - 自动预算重置：跨日/跨周/跨月自动清空历史累积
  - 熔断状态机：`NORMAL → WARNING → DEGRADE → FUSE → EMERGENCY`，五级状态
  - 线程安全：`threading.Lock` 保护并发写入
- **成本熔断**：`CostFuse` 与 `LLMClient` 集成，支持：
  - 预调用检查 `pre_call_check()`，自动降级模型（`claude-haiku-4-5`）
  - 后调用记录 `record_call()`，提取 Token 与费用
  - 模型降级与紧急阻断逻辑清晰
- **API 暴露**：`/cost-status` 端点实时返回 fuse 状态、消费比率、阈值详情。

#### 关键缺陷
- **指标未上报**：`record_persona_generation()` 和 `record_llm_call()` 未被调用，导致 Prometheus 中 `aicbc_cost_per_persona_cny`、`aicbc_cost_per_study_cny` 等成本指标为零，Grafana 成本面板无数据。
- **无成本告警**：虽然代码内有 `notify_if_changed()` 日志，但无外部通知渠道，预算突破时无法主动告警。
- **无成本预测**：只有事后统计，无基于历史增速的“预计今日/本周超预算”预测能力。
- **单点状态**：`cost_state.json` 为本地文件，多副本部署时各实例状态不一致，需改用 Redis 或共享存储。

#### 问题清单
| 编号 | 问题 | 严重程度 | 建议 |
|------|------|----------|------|
| COST-01 | 成本指标未上报 Prometheus | **高** | 在 `LLMClient` 和 `CostFuse` 中调用 `record_llm_call()` / `record_persona_generation()`，确保成本数据进入时序数据库 |
| COST-02 | 多副本状态不一致 | 高 | 将 `CostTracker` 持久化后端从本地 JSON 改为 Redis（使用 Redis Hash + INCRBY），支持水平扩展 |
| COST-03 | 无成本预测与趋势告警 | 中 | 在 Prometheus 中增加 `aicbc_cost_dailyBurn_rate`（基于 `increase` 函数），并配置告警规则：若 `predict_linear(cost_daily, 3600*4) > daily_cny` 则触发“预计 4 小时后超预算” |
| COST-04 | 无成本分摊/归因 | 低 | 在 `cost_per_study_cny` 标签中增加 `team_id` / `user_id`，实现多租户成本归因 |

## 5. 整改建议优先级

### 🔴 P0 — 阻塞上线（必须修复）
1. **APM-01 / APM-02**：在 `LLMClient` 和 Persona 生成流程中接通 `record_llm_call()` 与 `record_persona_generation()`，确保核心指标有数据。
2. **ALERT-01 / ALERT-02**：部署 Alertmanager，编写至少 5 条核心告警规则（API 错误率、P95 延迟、LLM 失败率、成本熔断、队列积压）。
3. **COST-01 / COST-02**：将成本追踪器数据接入 Prometheus；将 `cost_state.json` 改为 Redis 共享存储，支持多副本。

### 🟡 P1 — 上线后短期内完成（1-2 周）
4. **LOG-01 / LOG-02**：统一 structlog 配置（JSON 输出 + 时间戳）；引入 Loki + Promtail 做日志聚合，Grafana 中增加日志查询面板。
5. **TRACE-01 / TRACE-02**：引入 OpenTelemetry 自动插桩，覆盖 FastAPI 与 Celery；将 `trace_id` 注入日志。
6. **ALERT-03**：为成本熔断事件增加 Webhook 通知（钉钉/飞书/企业微信）。

### 🟢 P2 — 持续优化（1 个月内）
7. **APM-03 / APM-04**：增加数据库查询耗时与 Celery 任务指标。
8. **TRACE-03 / TRACE-04**：为 LLM 调用手动创建 Span，部署 Jaeger/Tempo 做可视化。
9. **COST-03**：增加成本 burn-rate 预测告警。
10. **APM-05**：补齐 Node Exporter / MongoDB Exporter / Redis Exporter，或清理 Prometheus 中无效 target。

---

## 6. 结论

AI_CBC 项目在可观测性方面**具备良好的基础架构骨架**（Prometheus + Grafana + structlog + 成本追踪器），但当前处于“**有骨骼、无血肉**”的状态：

- 核心指标埋点（LLM 调用、成本、消费者生成）**未接通**，导致仪表盘无法反映真实运行状态。
- 告警体系**完全缺失**，无 Alertmanager、无规则、无通知渠道，系统异常时无法主动感知。
- 分布式链路追踪**未启动**，微服务调用链不可见，排障效率低。
- 日志**未做集中汇聚**，多节点环境下搜索困难。

**建议**：在正式上线前，必须完成 P0 整改项，使监控与告警真正具备“发现故障、定位根因、通知到人”的闭环能力。
