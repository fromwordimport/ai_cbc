## 一、验收结论概览

| 维度 | 评分 | 说明 |
|------|------|------|
| API 设计 | ⚠️ 良 | RESTful 基本合规，但缺少统一错误码体系、admin 路由未注册 |
| 认证与授权 | 🔴 差 | 仅有全局 API Key，无 RBAC；admin 端点无认证保护 |
| 分层架构 | 🟡 中 | 控制器/数据访问层清晰，但缺少 Service 业务层；路由直接耦合生成器 |
| 异步与队列 | 🟡 中 | 分析任务使用 Celery ✅，但画像生成、模拟作答仍同步阻塞 |
| 数据校验 | ⚠️ 良 | Pydantic + 输入清洗覆盖核心端点；但 `/admin/settings` 直接收 `dict` |
| 错误处理 | ⚠️ 良 | 全局异常处理区分了环境；但缺少业务异常体系 |
| 配置安全 | 🟡 中 | 敏感项走环境变量，但存在不安全默认值且生产校验不强制 |
| **综合结论** | **⚠️ 有条件通过** | 存在 5 个 🔴 阻塞问题，必须修复后方可上线；建议 8 个 🟡 警告和 6 个 🔵 优化 |

## 二、问题清单（按严重度分级）

### 🔴 阻塞问题（必须修复）

#### 🔴 B-001：admin 端点未注册到 FastAPI，且无任何认证保护
- **文件**：`src/aicbc/api/routes/health.py`（第 26–62 行）、`src/aicbc/main.py`（第 61 行）
- **描述**：`api/routes/health.py` 中定义了 `/admin/settings`（GET 与 PUT）、`/dashboard/summary`、`/cost-status` 等敏感端点，但 `main.py` 实际注册的是 `aicbc.monitoring.health` 的 router，**`api/routes/health.py` 中的 router 未被 `include_router` 到 app 上**。即使未来被注册，当前代码中 `/admin/settings` 的 `PUT` 端点**没有任何 `Depends` 认证依赖**，直接接受外部 `dict` payload，可被任意调用修改运行时的 LLM 温度、真实性阈值等关键参数。
- **影响**：运行配置可被未授权篡改，导致模型输出质量不可控或成本激增。
- **修复建议**：
  1. 确认 `/admin/settings` 是否需要保留；若保留，必须注册到 app 并添加 `Depends(api_key_admin)` 或 JWT 鉴权。
  2. `PUT /admin/settings` 的 payload 必须定义为 Pydantic Model（`AdminSettingsUpdate`），用白名单字段+类型校验，拒绝未知字段。
  3. 生产环境应通过 `ENV` 驱动配置，禁止运行时热更新安全相关参数。

#### 🔴 B-002：全局 API Key 存在硬编码默认值，且认证中间件在 debug 模式下完全绕过
- **文件**：`src/aicbc/config/settings.py`（第 111 行）、`src/aicbc/main.py`（第 42–43 行）
- **描述**：`api_key = Field(default="dev-key-change-in-prod", alias="API_KEY")`；`APIKeyMiddleware` 在 `settings.debug` 为 `True` 时直接 `return await call_next(request)`，跳过了所有认证。开发者本地测试后极易将 `DEBUG=true` 带入生产环境。
- **影响**：若部署时忘记设置 `API_KEY` 或误开启 `DEBUG`，整个 API 完全暴露。
- **修复建议**：
  1. 移除 `api_key` 的默认值，在 `__post_init__` 或启动时强制校验：若 `is_production` 且 `api_key` 包含 `dev-key`，则抛出 `RuntimeError` 阻止启动。
  2. 认证中间件不应与 `debug` 强耦合；建议通过 `settings.environment` 区分，或引入 `DISABLE_AUTH` 显式开关，并在日志中输出 `SECURITY: authentication disabled` 告警。

#### 🔴 B-003：所有数据存储为 In-Memory，无持久化，进程重启数据全失
- **文件**：`src/aicbc/core/store.py`（全文件）
- **描述**：`PersonaStore`、`QuestionnaireStore`、`ResponseStore` 均为纯内存字典实现，虽然加了 `threading.Lock`，但**没有 MongoDB/Redis 持久化**。`store.py` 顶部注释明确说明 "This is a temporary implementation for Week 1-2 development"。
- **影响**：进程重启、容器重调度、滚动更新后所有研究数据、画像、问卷、作答记录全部丢失。不符合 MVP 上线要求。
- **修复建议**：立即接入 MongoDB（motor/beanie）或至少使用 Redis 持久化。在 `get_store()` 中根据环境切换实现，并添加数据迁移脚本。

#### 🔴 B-004：大批量同步操作阻塞主线程，无队列保护
- **文件**：`src/aicbc/api/routes/personas.py`（第 57–220 行）、`src/aicbc/api/routes/responses.py`（第 48–184 行）
- **描述**：`POST /api/v1/personas/generate` 和 `POST /api/v1/studies/{study_id}/simulate-responses` 均为**同步循环处理**：逐个生成画像、逐个模拟作答。每次请求最多 100 个画像/作答，当使用 `llm` 模式时每个 persona 需一次 LLM 调用，总耗时可达到数十分钟，完全阻塞 FastAPI 事件循环中的 worker 线程。
- **影响**：高并发或大批量请求会导致所有 worker 被占满，后续请求无法响应；若客户端超时重试，会进一步加剧资源耗尽。
- **修复建议**：
  1. 画像生成 >10 个或模拟作答 >10 个时，拆分为 Celery 异步任务，返回 `202 Accepted` + `job_id` 供轮询。
  2. 或引入 `BackgroundTasks` + 结果缓存作为过渡方案，但长期仍应走 Celery。

#### 🔴 B-005：速率限制器未实现真正的线程安全锁
- **文件**：`src/aicbc/api/middleware/rate_limit.py`（第 51 行）
- **描述**：`InMemoryRateLimiter` 的 `_lock` 字段被声明为 `dict[str, None]`，实际没有使用 `threading.Lock`，`is_allowed()` 方法中没有任何同步机制。在高并发下，多个请求同时进入 `is_allowed` 会导致 token 计数竞态条件，极限情况下可造成限流失效。
- **影响**：DDoS 或暴力调用时，速率限制无法可靠保护后端。
- **修复建议**：将 `_lock` 改为 `threading.Lock()`，在 `is_allowed` 入口和 `reset` 中使用 `with self._lock:`。生产环境应切换为 Redis 限流（如 `fastapi-limiter`）。

### 🟡 警告问题（建议修复）

#### 🟡 W-001：缺少角色权限控制（RBAC）
- **描述**：当前仅有单一全局 `X-API-Key` 认证，没有区分管理员、研究员、只读用户等角色。`DELETE /personas/{persona_id}` 等敏感操作与普通查询使用同一套认证。
- **影响**：密钥一旦泄露，攻击者拥有全部权限。
- **建议**：引入基于 JWT 的 RBAC，或至少为不同端点配置 `scopes`（如 `admin:write` vs `read`）。

#### 🟡 W-002：依赖注入使用全局可变单例，非线程安全
- **文件**：`src/aicbc/api/dependencies.py`（全文件）
- **描述**：所有依赖通过模块级全局变量（`_llm_client`, `_profile_generator` 等）实现懒加载单例。虽然 Python GIL 在一定程度上保护，但多 worker（Uvicorn 多进程）场景下每个进程有自己一份，尚可接受；但**测试并发重置时**（`reset_dependencies()`）可能存在竞态。
- **建议**：在 `get_settings_dep()` 之外，使用 `functools.lru_cache` 或 `fastapi.Depends` 的 `use_cache=True` 替代手工全局变量管理。为每个请求生命周期注入独立实例，避免状态泄漏。

#### 🟡 W-003：`/admin/settings` 直接接收 `dict` payload，无 schema 校验
- **文件**：`src/aicbc/api/routes/health.py`（第 27 行）
- **描述**：`admin_update_settings` 的签名是 `payload: dict`，而不是 Pydantic Model。FastAPI 无法对该 payload 进行字段类型校验、长度限制、注入检测。攻击者可能传入超大 JSON 导致内存压力，或注入非预期的字段类型。
- **建议**：定义 `AdminSettingsUpdate(BaseModel)`，使用 Pydantic 的 `Field(...)` 限定类型和范围，并额外做 `sanitize_text` 防御。

#### 🟡 W-004：Health Check 中的 MongoDB/Redis 连通性检查为假实现（TODO）
- **文件**：`src/aicbc/monitoring/health.py`（第 43–88 行）
- **描述**：`_check_mongodb()` 和 `_check_redis()` 仅返回 `status="ok"`，实际未执行 ping 操作。Readiness 检查不能真实反映依赖健康状态，Kubernetes 可能将流量路由到数据库未就绪的 Pod。
- **建议**：接入 motor 和 redis-py，实现真实的异步 ping，并在失败时返回 `503` 状态码。

#### 🟡 W-005：模拟作答端点在 Cost Fuse 触发后中断，但已保存部分数据无回滚机制
- **文件**：`src/aicbc/api/routes/responses.py`（第 135–139 行）
- **描述**：当 `CostFuseError` 触发时，循环 `break`，但此前已经成功保存的 persona_response 和 raw_dataset 不会被清理。下次重试时可能造成数据重复（`save_dataset` 虽有 merge 逻辑，但 `save_response` 是按 `response_id` 覆盖，而 `response_id` 是每次生成的新 UUID，不会覆盖）。
- **建议**：为 batch simulation 引入事务或幂等性设计；在响应摘要中标注 `partial_batch`，让客户端知道数据不完整。

#### 🟡 W-006：画像生成跳过验证后仍存储未验证数据，且无显式标记
- **文件**：`src/aicbc/api/routes/personas.py`（第 116–126 行）
- **描述**：当 `skip_validation=False` 时，若验证失败，代码只是 `log.warning` 然后继续存储。验证失败的画像没有 `validation_failed` 标记，前端或下游无法区分。
- **建议**：增加 `validation_passed` 布尔字段到 `PersonaProfile` 和 `PersonaSummary`，或在 `bias_audit_status` 旁增加 `validation_status`。

#### 🟡 W-007：成本追踪器使用相对路径保存状态文件，存在路径不确定性
- **文件**：`src/aicbc/cost/tracker.py`（第 30 行）
- **描述**：`_STATE_FILE = Path("./data/cost_state.json")`，工作目录不确定时文件可能写到错误位置。Docker 中如果 `WORKDIR` 不是预期路径，状态文件会丢失或无法共享。
- **建议**：使用 `Path(__file__).resolve().parent.parent.parent / "data" / "cost_state.json"` 或从 `settings` 中读取 `DATA_DIR` 环境变量。

#### 🟡 W-008：全局异常处理在 debug 模式下返回完整错误信息，可能被用于信息探测
- **文件**：`src/aicbc/main.py`（第 69–85 行）
- **描述**：`global_exception_handler` 在 `settings.debug` 为 `True` 时返回 `{"detail": str(exc)}`。如果生产环境因误配置开启 debug，攻击者可获得内部异常信息（如文件路径、库版本）。
- **建议**：即使 `debug=True`，也仅对受信 IP 或管理员返回详细错误，普通客户端始终返回通用消息。

### 🔵 建议问题（优化项）

#### 🔵 S-001：未使用的健康路由文件存在，与 monitoring 路由重复
- **描述**：`src/aicbc/api/routes/health.py` 中定义了 `/health`、`/ready`、`/metrics` 等端点，但 `main.py` 注册的是 `aicbc.monitoring.health` 的 router。`api/routes/health.py` 处于孤儿状态，未来维护时容易混淆。
- **建议**：统一合并到一个健康模块，删除或归档 `api/routes/health.py`。

#### 🔵 S-002：API 端点缺少 ETag/Cache-Control 和条件请求支持
- **描述**：分析结果、问卷详情等只读数据不支持缓存，重复请求会重复执行数据库查询和序列化。
- **建议**：对 `/studies/{id}/questionnaire`、`/analysis/{id}` 等端点添加 `ETag` 或 `Last-Modified`，利用 FastAPI 的 `Response` 对象设置 header。

#### 🔵 S-003：缺少 OpenAPI 文档中安全方案的声明
- **描述**：`app = FastAPI(...)` 未配置 `openapi_tags` 和 `security_schemes`，Swagger UI 中不显示 `X-API-Key` 的认证入口。
- **建议**：添加 `security_schemes={"ApiKeyAuth": {"type": "apiKey", "in": "header", "name": "X-API-Key"}}` 并在受保护端点标记 `dependencies=[Depends(verify_api_key)]`。

#### 🔵 S-004：分页响应缺少 Link Header 或总页数信息
- **描述**：`PersonaListResponse` 和 `StudyListResponse` 返回 `total`、`page`、`page_size`，但缺少 `total_pages` 和 `has_next`/`has_prev` 字段，前端需要自行计算。
- **建议**：增加 `total_pages: int` 和 `has_next: bool` 字段，提升 API 易用性。

#### 🔵 S-005：Celery 任务配置在 route 中直接构造 JSON，容易遗漏字段
- **文件**：`src/aicbc/analysis/routes.py`（第 147–152 行）
- **描述**：`config_json = json.dumps({...})` 手动序列化，若 `AnalyzeRequest` 新增字段，此处容易漏传。且 `json.dumps` 对 `None` 和默认值处理不一致。
- **建议**：使用 `request.model_dump_json()` 直接序列化，或定义 Celery 任务的 Pydantic 输入模型。

#### 🔵 S-006：分析 Agent 的 `run` 方法同步执行重型计算，缺少进度暴露
- **文件**：`src/aicbc/agents/analysis_agent.py`（第 124 行）
- **描述**：`AnalysisAgent.run()` 内部调用 HBEngine（PyMC），在单线程中执行 MCMC 采样。虽然上层路由已用 Celery 异步，但 Agent 内部没有提供 progress callback 或 checkpoint 机制，长任务无法被中断或查看进度。
- **建议**：在 `HBEngine` 中支持 `arviz` 的采样回调，或将采样步骤拆分为更细粒度的子任务。

## 三、逐模块审查详情

### 3.1 FastAPI 入口与中间件 (`main.py`)

| 审查项 | 结论 | 说明 |
|--------|------|------|
| 应用初始化 | ✅ | 使用 `FastAPI()` 标准初始化，文档开关根据 `debug` 控制 |
| 中间件顺序 | ⚠️ | RateLimit → Metrics → SecurityHeaders → APIKey；顺序合理，但 SecurityHeaders 在 APIKey 之后，意味着认证失败（401）的响应也会被打上安全头，这没问题。但 Metrics 在 APIKey 之前，意味着未认证请求也会被计入 metrics，是否预期需确认 |
| 全局异常处理 | ⚠️ | `debug=True` 时暴露 `str(exc)`，有信息泄露风险（见 B-002） |
| 路由注册 | ✅ | 统一使用 `/api/v1` 前缀，版本管理清晰 |
| 生命周期事件 | ✅ | `startup` / `shutdown` 有日志记录 |

### 3.2 配置管理 (`config/settings.py`)

| 审查项 | 结论 | 说明 |
|--------|------|------|
| 环境变量驱动 | ✅ | 使用 `pydantic-settings`，支持 `.env` 文件 |
| 敏感配置默认值 | 🔴 | `api_key`、`secret_key` 存在硬编码默认值，生产未强制覆盖（见 B-002） |
| 配置分层 | ✅ | LLM、Database、CostFuse、Study、Authenticity 等子配置分离清晰 |
| secret_key 校验器 | ⚠️ | `_validate_secret_key` 仅对非空做检查，未在生产环境强制拒绝默认值 |
| 额外字段处理 | ✅ | `extra="ignore"` 避免未知字段导致启动失败，合理 |

### 3.3 API Schema (`api/schemas.py`)

| 审查项 | 结论 | 说明 |
|--------|------|------|
| Pydantic Model 覆盖 | ✅ | 请求/响应模型完整，覆盖 BatchGenerate、Persona、Study、Questionnaire、Response、Simulation |
| 字段校验 | ✅ | 广泛使用 `Field(..., ge=1, le=100, pattern=...)`，边界校验到位 |
| 嵌套模型 | ✅ | `ChoiceSetView` → `AlternativeView` 等嵌套结构清晰 |
| 类方法工厂 | ✅ | `from_profile()`、`from_study()` 等工厂方法减少了路由中的序列化代码 |
| 缺少 admin schema | 🟡 | `/admin/settings` 的 payload 没有对应的 Pydantic Model（见 W-003） |

### 3.4 依赖注入 (`api/dependencies.py`)

| 审查项 | 结论 | 说明 |
|--------|------|------|
| 单例懒加载 | ✅ | LLMClient、ProfileGenerator、Validators 等按需初始化 |
| 循环依赖管理 | ⚠️ | `get_profile_generator` 依赖 `get_llm_client`，`get_behavior_simulator` 也依赖 `get_llm_client`，层级扁平，无深层循环 |
| 测试重置 | ✅ | `reset_dependencies()` 提供全局清理，方便测试隔离 |
| 线程安全 | 🟡 | 全局变量方式在多 worker 场景下尚可，但测试并发时存在竞态（见 W-002） |

### 3.5 画像 API (`api/routes/personas.py`)

| 审查项 | 结论 | 说明 |
|--------|------|------|
| RESTful 路径 | ✅ | `POST /personas/generate`, `GET /personas/{id}`, `DELETE /personas/{id}` 标准 |
| 输入清洗 | ✅ | `sanitize_id` 用于 `study_id`，防止注入 |
| 错误消息安全 | ✅ | `_safe_error_message` 在生产环境隐藏内部错误 |
| 偏见重试机制 | ✅ | 最多 3 次尝试（1 次原始 + 2 次 retry），符合规范 |
| 验证失败处理 | 🟡 | 验证失败仅打日志仍存储，无显式标记（见 W-006） |
| 批量处理 | 🔴 | 同步循环，无队列，大批量阻塞（见 B-004） |
| 成本累加 | ✅ | 每次生成/重试都累加 `total_cost`，结果精确到 4 位小数 |

### 3.6 问卷 API (`api/routes/questionnaires.py`)

| 审查项 | 结论 | 说明 |
|--------|------|------|

| 设计参数解析 | ✅ | `_parse_design_params` 直接 `DesignParameters(**raw)`，由 Pydantic 内部校验 |
| 问卷生成后更新状态 | ✅ | 生成后自动将 study status 设为 `READY` 并保存 |

### 3.7 作答 API (`api/routes/responses.py`)

| 审查项 | 结论 | 说明 |
|--------|------|------|
| 批量模拟 | ✅ | 支持 `rule` 和 `llm` 两种模式，按请求参数切换 |
| 输入清洗 | ✅ | `sanitize_id` 和 `sanitize_string_list` 用于 `study_id` 和 `persona_ids` |

### 3.8 模拟 API (`api/routes/simulations.py`)

| 审查项 | 结论 | 说明 |

### 6.2 端点利用率分析（各模块使用率排名）

从 API 设计视角看，低利用率不仅反映前端实现滞后，也提示部分端点的职责划分或返回值结构可能未与前端页面模型充分对齐：

1. **分析管线（89%）**：端点覆盖最好，说明 API 粒度与前端仪表盘页面高度匹配。但存在关键参数硬编码问题（见 6.3）。
2. **作答模拟（67%）**：基本覆盖，但缺少逐条记录查看端点，前端只能通过导出获取完整数据。
3. **问卷配置（50%）**：后端已支持自定义属性与设计参数，但前端表单未扩展；`PUT` 编辑端点完全闲置，导致创建后无法修正研究配置。
4. **画像管理（50%）**：筛选端点未使用，说明前端列表页未对接后端多维筛选能力；单层获取端点未使用，丧失按需加载优化空间。
5. **系统运维（50%）**：聚合端点和管理端点被前端自行拼凑或本地存储替代，后端统一收口逻辑被架空。
6. **行为模拟（33%）**：最差，仅单轮对话端点被使用，批量采访和购买决策端点完全未暴露，后端已实现的阶段追踪与置信度评估能力无法触达用户。

---

### 6.3 参数僵化问题分析（5 项）

报告发现 5 个已使用端点的关键参数被前端硬编码。以下从 API 设计角度逐项判定根因：

| # | 页面 | 端点 | 硬编码参数 | 后端实际支持 | 根因判定 | 严重度 |
|---|------|------|-----------|-------------|---------|--------|
| 1 | `StudyCreate.tsx` | `POST /studies` | 不传 `attributes`/`design_parameters` | 完整自定义属性 + 设计参数 | **前端适配缺失**：后端 `CreateStudyRequest` 已开放 schema，前端表单未同步扩展字段。但后端默认值为"洗碗机"示例，降低了前端接入紧迫性。 | 🟡 |
| 2 | `AnalysisStatus.tsx` | `POST .../analyze` | `model_type: 'hb'` | `hb` / `mnl` / `latent_class` | **API 默认值策略缺失**：后端 schema 中 `model_type` 为必填，未提供自动选择逻辑（如样本量 < 50 默认 MNL），前端被迫硬编码。 | 🟡 |
| 3 | `ResponseSimulator.tsx` | `POST .../simulate-responses` | `mode: 'stochastic'` | `stochastic` / `llm` / `deterministic` | **前端适配缺失**：三种模式差异大，前端需提供 UI 选择器。但后端 OpenAPI 描述未充分说明各模式适用场景与成本差异，增加了前端对接难度。 | 🟡 |
| 4 | `SegmentComparison.tsx` | `GET .../segment-comparison` | 测试类型硬编码 | `hotelling` / `welch` / `permutation` | **API 契约不透明**：后端未在 schema 中说明三种检验的适用条件（如样本分布、方差齐性），前端缺乏决策依据。 | 🔵 |
| 5 | `InterviewLab.tsx` | `POST .../converse` | 仅使用单轮对话 | 已提供 `interview` / `purchase-decision` 专用端点 | **前端功能缺失**：后端提供了语义更清晰的专用端点，但前端页面未扩展。属于前端未对接，非 API 设计缺陷。 | 🔵 |

**总体评估**：参数僵化并非 API 契约设计缺陷，而是**后端灵活性未通过前端 UI 释放**。但其中 2 项（模型类型、统计检验类型）与后端**默认值策略和字段说明不足**直接相关，建议后端补充：
- `AnalyzeRequest.model_type` 设为可选，后端根据样本量自动选择并回写实际使用模型；
- `segment-comparison` 的统计检验参数增加 `Field(description=...)`，说明三种检验的假设前提与推荐场景。

### 6.4 专用端点被绕过问题

#### 🔴 B-006：Dashboard 聚合端点 `/dashboard/summary` 被前端自行拼凑替代

- **描述**：前端 `Dashboard.tsx` 使用 `getStudies` + `getPersonas` 两次独立调用获取数据，而非使用后端已提供的专用聚合端点 `GET /dashboard/summary`。这导致：
  - 网络请求翻倍，增加延迟和服务器负载；
  - 前端需自行维护聚合逻辑，与后端数据口径不一致风险；
  - 若后端聚合逻辑升级（如加入缓存、成本汇总、实时计数），前端无法受益。
- **根因**：前端未对接该端点；同时该端点因 `api/routes/health.py` 未被注册到 FastAPI app（见 B-001），即使前端想调用也无法成功。
- **修复建议**：
  1. 先完成 B-001（注册 `health.py` router），使 `/dashboard/summary` 可用；
  2. 后端确认 `/dashboard/summary` 返回字段覆盖 Dashboard 组件全部数据需求（如 study 总数、persona 总数、最近活跃、成本概览）；
  3. 若数据结构不匹配，优先调整后端聚合 schema，而非让前端继续拼凑；
  4. 前端切换至单次聚合调用，移除双请求逻辑。

---

### 6.5 管理端点未对接问题

#### 🟡 W-014：`GET /admin/settings` 存在但前端从本地存储读取

- **描述**：前端 `Settings.tsx` 直接从 `localStorage` 读取默认值，而非通过 `GET /admin/settings` 拉取服务端当前运行时配置。这导致：
  - 服务端配置与前端视图不一致（如运维人员通过 API 修改了温度参数，前端仍显示旧值）；
  - `PUT /admin/settings` 写回时可能覆盖其他客户端已更新的配置，产生静默丢失。
- **根因**：`GET /admin/settings` 同样因 B-001 未注册到 app；且前端未实现初始化拉取逻辑。
- **修复建议**：B-001 修复后，前端在 `Settings.tsx` 挂载时调用 `GET /admin/settings` 同步服务端状态，再以 `PUT` 写回。

### 6.6 对服务端验收结论的影响评估

本次前后端覆盖率审查对服务端验收结论的影响如下：

1. **未引入新的服务端阻塞缺陷**：所有覆盖率缺口均属于"后端已实现但前端未使用"或"参数未暴露"，后端代码本身没有新增安全、性能或数据一致性缺陷。
2. **B-001 的连锁影响被放大**：由于 `/dashboard/summary` 和 `/admin/settings` 等端点未注册且无认证，前端被迫采用替代方案（本地存储、拼凑调用），这些替代方案反而增加了数据不一致风险和安全暴露面。
3. **后端默认值策略需补充**：部分参数硬编码源于后端未提供合理的默认值或智能选择逻辑。若后端补充默认值，前端即使暂不实现选择器，也能获得更优行为（如小样本自动选 MNL）。
4. **验收结论不变**：综合结论仍为 **⚠️ 有条件通过**，但需将前后端端点对齐纳入 P1 跟踪项，作为上线前必须完成的集成检查。
