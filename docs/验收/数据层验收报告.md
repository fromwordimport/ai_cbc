## 1. 执行摘要

AI_CBC 项目当前**所有核心数据存储均为进程内内存实现（in-memory）**，MongoDB 与 Redis 虽已纳入 Docker Compose 编排并配置环境变量，但**业务代码层完全未接入**持久化存储与分布式缓存。数据层处于“Week 1-2 临时方案”阶段，距离生产可用存在系统性差距。本报告识别出 4 个阻塞项（必须修复方可进入下一阶段）、3 个严重缺陷和 5 项优化建议。

## 2. 详细审查结果

### 2.1 数据库选型与建模

#### 2.1.1 现状评估

| 维度 | 现状 | 评级 |
|------|------|------|
| 数据库选型 | MongoDB 6（容器化）已部署，但代码中无 Motor/Beanie 客户端；实际使用 `dict` 内存存储 | 🔴 阻塞 |
| 数据模型 | `PersonaProfile` 为四层嵌套 Pydantic 模型，天然契合文档型数据库；但无 ODM 映射 | 🟡 严重 |
| 索引设计 | 内存层实现了 segment / city / bias_status 三套倒排索引；数据库层零索引 | 🟡 严重 |
| 集合规划 | 无集合（Collection）拆分设计，study / questionnaire / response / persona / analysis 全部混于内存字典 | 🟡 严重 |

#### 2.1.2 关键代码证据

- `src/aicbc/core/store.py` 第 1-5 行明确声明：
  > *"In-memory persona storage with query support. This is a temporary implementation for Week 1-2 development. Production will switch to MongoDB (Beanie) or a persistent store."*
- `src/aicbc/analysis/store.py` 第 1-4 行同样声明：

  > *"TODO: Implement actual Redis ping when redis client is available"*

#### 2.1.3 阻塞项

**B-001：生产环境不可用——全量数据存储于进程内存**
- **影响**：容器重启、服务滚动更新、OOM Kill 会导致全部画像、问卷、分析任务丢失。
- **修复要求**：
  1. 引入 `motor` + `beanie` 作为 MongoDB 异步 ODM；
  2. 将 `PersonaStore`、`QuestionnaireStore`、`ResponseStore`、`AnalysisStore` 重构为基于 Beanie `Document` 的仓储实现；
  3. 保留现有内存实现作为 `MemoryStore` 后端，供测试与本地开发使用（通过 `ENVIRONMENT` 切换）。

#### 2.1.4 建议优化

**O-001：模型命名与集合规划**
- `PersonaProfile` 嵌套深度 4 层，建议拆分为独立子文档（`Layer1Demographics`、`Layer2Behavior` 等），在 Beanie 中利用 `Link` 或内嵌文档视查询频率决定。
- 建议集合命名：
  - `personas` — 画像主文档
  - `studies` — 研究配置
  - `questionnaires` — 问卷设计
  - `responses` — 个体选择响应
  - `datasets` — 聚合数据集（可物化视图）
  - `analysis_jobs` — 分析任务状态
  - `analysis_results` — 分析结果（收敛诊断、WTP、重要性等）

**O-002：索引设计草案**
- `personas`：复合索引 `{ persona_id: 1 }`（唯一）、`{ segment: 1, created_at: -1 }`、`{ "layer1_demographics.city": 1 }`、`{ bias_audit_status: 1 }`
- `studies`：索引 `{ study_id: 1 }`（唯一）、`{ product_category: 1 }`
- `analysis_jobs`：索引 `{ analysis_id: 1 }`（唯一）、`{ status: 1, created_at: -1 }`（用于队列扫描）
- `responses`：复合索引 `{ study_id: 1, respondent_id: 1 }`

### 2.2 多级缓存策略

#### 2.2.1 现状评估

| 缓存层级 | 现状 | 评级 |
|----------|------|------|
| 浏览器缓存 | API 响应未设置 `Cache-Control`、`ETag`、`Last-Modified`；`/personas/{id}` 等读端点完全无缓存 | 🟡 严重 |
| CDN / 边缘缓存 | Nginx 配置中无 `proxy_cache` 或 `fastcgi_cache` 指令；无 CDN 回源策略 | 🟢 建议优化 |
| 应用层缓存（Redis） | Redis 容器已运行，但业务代码零调用；Celery 仅使用 Redis 作为 Broker | 🔴 阻塞 |
| 数据库查询缓存 | 未使用 MongoDB；无 query cache | 🟢 建议优化 |
| 进程内缓存 | `LLMClient` 有 LRU 线程安全缓存（`OrderedDict` + `threading.Lock`）；`ProfileGenerator` 有 `functools.lru_cache` | 🟢 良好 |

#### 2.2.2 关键代码证据

- `docker-compose.yml` 第 128-147 行部署 Redis 7-alpine，但代码中无 `redis.asyncio` 或 `aioredis` 导入。
- `docker/nginx.conf` 第 86-101 行 API 路由配置中无缓存相关 header 或 `proxy_cache`。
- `src/aicbc/llm/client.py` 第 88-92 行实现了线程安全 LRU 缓存，这是当前唯一合理的应用层缓存。

#### 2.2.3 阻塞项

**B-002：Redis 缓存未接入业务层**
- **影响**：高并发查询画像列表、分析结果时，全部压力直接落到（尚未存在的）数据库或内存字典；且分析结果（WTP、市场模拟）计算成本高，无法复用。
- **修复要求**：
  1. 引入 `redis.asyncio` 客户端，封装 `CacheService`；
  2. 读多写少场景启用缓存：画像列表、分析结果（收敛诊断、WTP、属性重要性）；
  3. 设置合理的 TTL：画像数据 `TTL=300s`，分析结果 `TTL=600s`（或按任务状态动态失效）；
  4. 缓存键规范：`aicbc:{entity}:{id}:{version}`，如 `aicbc:persona:list:segment=xxx:page=1`。

#### 2.2.4 建议优化

**O-003：API 响应缓存头**
- 静态分析结果（如 `/analysis/{id}/wtp`）可在响应头添加：
  ```
  Cache-Control: private, max-age=600
  ETag: "{analysis_id}:{result_hash}"
  ```
- 动态生成中的画像列表应明确禁用缓存：
  ```
  Cache-Control: no-store, no-cache, must-revalidate
  ```

**O-004：Nginx 代理缓存（可选）**
- 对于 `/api/personas` 等列表接口，可在 Nginx 层配置 `proxy_cache_path` + `proxy_cache_valid 200 10s`，减轻应用层压力。

### 2.3 数据一致性保障

#### 2.3.1 现状评估

| 一致性维度 | 现状 | 评级 |
|------------|------|------|
| 事务支持 | 无数据库事务；内存操作使用 `threading.Lock` 粗粒度锁 | 🔴 阻塞 |
| 乐观锁 | 无 `version` 字段或 `_etag`；无 CAS 语义 | 🟡 严重 |
| 悲观锁 | 无分布式锁；`threading.Lock` 仅单进程有效 | 🟡 严重 |
| 状态机一致性 | `AnalysisStore` 实现了合法状态转换校验（`PENDING→QUEUED→RUNNING→COMPLETED`），但仅在内存中 | 🟡 建议优化 |
| 去重机制 | `PersonaStore` 使用 SHA-256 指纹去重，设计合理 | 🟢 良好 |

#### 2.3.2 关键代码证据

- `src/aicbc/core/store.py` 第 149-157 行：`PersonaStore` 维护了 `_segment_index`、`_city_tier_index`、`_bias_status_index` 三套倒排索引，且 `_lock` 为 `threading.Lock`。
- `src/aicbc/analysis/store.py` 第 50-58 行：定义了 `_VALID_TRANSITIONS` 状态机，但 `update_job_status` 仅回退非法转换，不保证跨进程一致性。
- `src/aicbc/api/middleware/rate_limit.py` 第 23 行：`InMemoryRateLimiter` 使用全局字典，多实例部署时失效。

#### 2.3.3 阻塞项

**B-003：分析任务状态机缺乏持久化与分布式一致性**
- **影响**：Celery Worker 多实例部署时，任务状态（`PENDING→RUNNING→COMPLETED`）在内存字典中维护，Worker 无法共享状态，导致任务重复执行或状态不一致。
- **修复要求**：
  1. 将 `analysis_jobs` 集合作为任务状态唯一真相源；
  2. 利用 MongoDB 原子操作（`find_one_and_update` + 条件 `$eq`）实现乐观锁状态转换；
  3. Celery Worker 通过轮询/监听数据库变更获取任务，而非内存队列。

#### 2.3.4 建议优化

**O-005：画像版本控制**
- `PersonaProfile` 已包含 `version` 字段（默认 `"0.1.0"`），建议将其提升为数据库乐观锁字段：
  - 更新时要求 `version` 匹配，成功后递增 `version`（如 `0.1.0` → `0.1.1`）；
  - 并发更新冲突时返回 `409 Conflict`，由客户端重试或合并。

#### 2.4.3 阻塞项

**B-004：缺失定期备份与恢复策略**
- **影响**：数据丢失后无法恢复；Redis 作为 Celery Broker，若宕机未开启 AOF，任务队列全部丢失。
- **修复要求**：
  1. **MongoDB**：添加 `mongodump` 定时任务（Celery Beat 或宿主机 cron），每日 02:00 执行，保留 7 份滚动备份；备份目录 `./backups/mongo/$(date +%Y%m%d)`。
  2. **Redis**：在 `docker-compose.yml` 中覆盖 Redis 启动命令，启用 AOF：
     ```yaml
     redis:
       image: redis:7-alpine
       command: redis-server --appendonly yes --save 60 1000
     ```
  3. **恢复脚本**：编写 `scripts/restore_mongo.sh` 与 `scripts/restore_redis.sh`，并在 `docs/运维/` 中记录恢复演练步骤（RTO ≤ 30 min，RPO ≤ 24 h）。

#### 2.5.2 建议优化

**O-006：显式初始化 MongoDB 与 Redis 连接池**
- 在 `src/aicbc/config/database.py` 中新增：
  ```python
  from motor.motor_asyncio import AsyncIOMotorClient
  import redis.asyncio as aioredis
  from aicbc.config.settings import get_settings

  _mongo_client: AsyncIOMotorClient | None = None
  _redis_pool: aioredis.Redis | None = None

  def get_mongo_client() -> AsyncIOMotorClient:
      global _mongo_client
      if _mongo_client is None:
          settings = get_settings()
          _mongo_client = AsyncIOMotorClient(
              settings.database.mongodb_url,
              maxPoolSize=settings.database.mongodb_max_connections,
              minPoolSize=10,
              serverSelectionTimeoutMS=5000,
          )
      return _mongo_client

  def get_redis_pool() -> aioredis.Redis:
      global _redis_pool
      if _redis_pool is None:
          settings = get_settings()
          _redis_pool = aioredis.from_url(
              settings.database.redis_url,
              password=settings.database.redis_password or None,
              max_connections=20,
              decode_responses=True,
          )
      return _redis_pool
  ```
- 在应用关闭时（FastAPI `shutdown` 事件）正确关闭连接池，防止句柄泄漏。

**O-007：连接池监控指标**
- 通过 Prometheus 暴露 `mongodb_active_connections`、`redis_pool_usage` 等 Gauge 指标，便于容量规划。
