## 一、回归测试策略

### 1.1 回归范围

当前 AI_CBC v0.1.0 (MVP) 因存在 **38 项 🔴 阻塞问题** 未通过验收。回归测试将在以下阻塞问题修复后执行：

| 优先级 | 修复阶段 | 回归测试范围 |
|--------|---------|-------------|
| P0-1 | 安全整改 | 红队测试全量回归（防御率 ≥ 95%）、认证/授权/API 安全测试 |
| P0-2 | 数据持久化 | MongoDB/Redis 数据一致性测试、备份恢复演练、连接池压力测试 |
| P0-3 | 合规整改 | 公平性硬规则嵌入验证、隐私合规接口测试、国际化切换测试 |
| P0-4 | 测试覆盖 | 覆盖率回归（目标 ≥ 60%）、agents/tools/llm 核心模块测试回归 |
| P1 | 性能/DevOps | 压力测试回归、CI/CD 全链路回归、部署回滚演练 |
| P1 | 可观测性 | 指标埋点回归、告警触发测试、成本追踪一致性测试 |

### 1.2 回归测试触发条件

回归测试仅在以下条件全部满足后启动：
1. 对应模块的全部 🔴 阻塞问题已标记为「已修复」
2. 修复责任人提交代码变更并附带修复说明
3. 相关专家（小验安/小验规/小验数等）初审通过

## 三、回归测试通过标准

每条回归测试必须满足以下条件方可视为通过：

| 条件 | 说明 |
|------|------|
| 原问题复现失败 | 使用原问题的复现步骤，确认问题已不可复现 |
| 无副作用 | 回归测试范围内的关联功能未出现新的异常 |
| 专家确认 | 对应模块的验收专家书面确认「通过」 |
| 证据留存 | 保留测试命令输出、截图、日志等证据 |

## 四、回归测试流程

```
开发团队提交修复
    ↓
小验分配回归任务至对应专家
    ↓
专家执行回归测试 + 填写本记录表
    ↓
测试通过 → 标记为 ✅ 已关闭
测试失败 → 退回开发团队重新修复
    ↓
全部 38 项阻塞问题回归通过
    ↓
小验汇总，启动最终复验
    ↓
全模块重新审查（抽样）
    ↓
出具最终验收结论
```
