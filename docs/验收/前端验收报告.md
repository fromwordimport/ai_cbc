⚠️ **有条件通过**

## 详细发现

### 1. 响应式适配

- **Layout 布局在移动端无折叠/隐藏适配** (🟡 警告 / `Layout.tsx`)
  - Sider 固定宽度 220px，无 `breakpoint` / `collapsed` / `collapsedWidth` 的响应式策略，在 768px 以下屏幕会占据大部分视口宽度，主内容区被严重挤压。
  - 建议：为 `Sider` 增加 `breakpoint="lg"` 或 `useMediaQuery` 方案，在移动端自动收起为图标抽屉。

- **MarketSimulator 场景配置行在小屏幕下溢出** (🟡 警告 / `MarketSimulator.tsx`)
  - 每个场景使用 `Row gutter={[12, 12]} align="middle">` 内含 8 个 `Col span={3}`，总计 24 列刚好占满，但在 `xs` / `sm` 屏幕下不会自动折叠，导致横向滚动或布局崩坏。
  - 建议：将 `Col span={3}` 改为响应式断点组合，如 `xs={24} sm={12} md={6} lg={3}`，或改用折叠卡片/表单形式。

- **Settings 页 Slider 固定宽度 300px** (🔵 建议 / `Settings.tsx`)
  - `style={{ width: 300 }}` 在小屏容器内可能溢出，建议改用百分比或 `maxWidth: '100%'`。

- **整体评估**：Dashboard、ImportanceDashboard、Settings 已使用 Ant Design 的响应式栅格（`xs={24} md={12}` 等），但核心布局骨架和表单密集页缺失移动端适配。

---

### 2. 性能感知优化

- **路由级懒加载已实施** (✅ 通过 / `router.tsx`)
  - 全部 12 个页面使用 `React.lazy()` + `Suspense`，减少首屏包体积。

- **Vite 手动分包策略合理** (✅ 通过 / `vite.config.ts`)
  - 将 `react`、`antd`、`echarts`、`axios` 分别拆为独立 chunk，避免重复打包，提升缓存命中率。

- **缺少资源预加载 / Prefetch** (🔵 建议 / `router.tsx`)
  - 对高频页面（如 Dashboard、ImportanceDashboard）未配置 `link rel="prefetch"` 或 `import(/* webpackPrefetch: true */ ...)`，用户悬停菜单时无提前加载策略。

- **多轮轮询缺少页面可见性优化** (🟡 警告 / `AnalysisStatus.tsx`、`Settings.tsx`)
  - `AnalysisStatus` 每 3 秒、Dashboard 每 60 秒、Settings 每 30 秒轮询。Dashboard 在 `window.addEventListener('focus', ...)` 做了聚焦刷新，但 `AnalysisStatus` 和 `Settings` 的轮询在后台标签页依然持续运行，浪费资源。
  - 建议：统一使用 `document.visibilitychange` 监听，页面不可见时暂停轮询。

- **QuestionnaireConfig 存在 N+1 请求** (🟡 警告 / `QuestionnaireConfig.tsx`)
  - 加载研究列表后，对每个 `hasQuestionnaire` 的研究单独发起 `getQuestionnaire` 请求，若研究数量大，将产生并发请求风暴。当前 `pageSize=100` 已做上限，但仍非最优。
  - 建议：后端提供批量查询接口，或前端改为展开行时才加载单个问卷。

- **无 Service Worker / PWA 支持** (🔵 建议 / 全局)
  - 离线可用性和缓存策略未考虑。

### 3. 交互直觉性

- **ResponseSimulator 使用固定假进度条** (🔴 阻塞 / `ResponseSimulator.tsx` 第97行)
  - `Progress percent={50} status="active"` 在模拟过程中始终固定在 50%，不会随真实进度变化，严重误导用户，属于数据/交互诚信缺陷。
  - 建议：若后端无 SSE/WebSocket 实时进度，应移除假进度条，改用 `loading` 状态 + 文字说明，或实现轮询进度接口。

- **InterviewLab 对话历史无自动滚动** (🟡 警告 / `InterviewLab.tsx`)
  - 新消息加入 `history` 后，列表不会自动滚动到底部，用户需手动下滑查看最新回复。
  - 建议：在 `history` 变化后使用 `scrollIntoView` 或 `List` 的 ref 滚动。

- **Dashboard 初始加载无全局 Loading** (🟡 警告 / `Dashboard.tsx`)
  - `<Spin spinning={false}>` 永远为 false，首次加载数据时用户看到的是空白统计卡片和空表格，没有加载感知。
  - 建议：绑定 `loading` state 到 `Spin`。

- **API 错误展示方式不统一** (🟡 警告 / 全局)
  - 部分页面用 `message.error()`（如 `PersonaManager`、`InterviewLab`），部分用 `Alert` 组件（如 `Dashboard`、`ImportanceDashboard`），部分两者都用。错误提示在页面切换时可能被忽略。
  - 建议：全局错误统一用 `Alert` 在页面内展示，网络瞬时失败用 `message` 轻提示。

- **SegmentComparison 结果空状态缺失** (🔵 建议 / `SegmentComparison.tsx`)
  - 未点击对比前，页面下方为空白，没有引导空状态。
  - 建议：添加 `Empty` 组件提示用户输入参数并运行对比。

- **PersonaManager 分页器独立在 Table 外** (🟡 警告 / `PersonaManager.tsx`)
  - 使用独立的 `Pagination` 组件，而非 `Table` 内置 `pagination`，导致切换页码时表格无内置 loading 联动，且 Table 固定 `pagination={false}` 时缺少 `showSizeChanger` 等默认功能。
  - 建议：统一使用 `Table pagination={{ ... }}`。

- **分析任务状态条件逻辑冗余** (🟡 警告 / `AnalysisStatus.tsx` 第57行)
  - `(status.status === 'COMPLETED' || status.status === 'FAILED') && status.status === 'COMPLETED'` 等价于 `status.status === 'COMPLETED'`，冗余代码增加维护成本。

- **StudyCreate 问卷生成失败后的回滚提示清晰** (✅ 通过 / `StudyCreate.tsx`)
  - 两阶段提交（创建研究 → 生成问卷）在第二阶段失败时给出明确说明和手动操作指引，体验良好。

### 4. 无障碍（A11y）

- **ECharts 图表对屏幕阅读器不可访问** (🟡 警告 / `ImportanceDashboard.tsx`、`MarketSimulator.tsx`)
  - 所有 ECharts 实例使用 Canvas 渲染，没有提供 `aria-label`、数据表格替代或 `role="img"` 说明，视障用户无法感知图表内容。
  - 建议：为图表容器添加 `aria-label` 概述图表主题，并在下方提供可访问的数据表格（已部分实现）。

- **缺少动态内容播报机制** (🔵 建议 / `AnalysisStatus.tsx`、`ImportanceDashboard.tsx`)
  - 分析任务完成、轮询结果更新等动态变化，没有 `aria-live="polite"` 区域，屏幕阅读器用户不会收到状态变化通知。

- **部分表单控件缺少显式标签** (🔵 建议 / 全局)
  - 如 `SegmentComparison` 的 `Input`（分析ID）仅使用 `placeholder`，没有 `label` 或 `aria-label`。
  - 建议：所有输入控件增加 `aria-label` 或显式 `label` 关联。

- **无障碍基础达标项** (✅ 通过)
  - `index.html` 已设置 `lang="zh-CN"`。
  - Ant Design 的 Menu、Button、Select 等组件自带基础键盘导航和 ARIA 属性。
  - 没有使用 `div` 伪装按钮的反模式。

- **缺少 Skip Link** (🔵 建议 / `Layout.tsx`)
  - 没有提供跳到主内容的跳转链接，键盘用户需逐层遍历侧边导航。

### 5. 前端架构稳健性

- **API 服务层硬编码 API Key** (🔴 阻塞 / `api.ts` 第44行)
  - `config.headers['X-API-Key'] = 'dev-key-change-in-prod'` 为硬编码字符串，直接暴露于源码中，存在严重安全风险（密钥泄露、仓库历史残留）。
  - 建议：改为从环境变量读取（`import.meta.env.VITE_API_KEY`），并在构建流水线中注入；生产环境禁用回退默认值。

- **缺少全局 Error Boundary** (🟡 警告 / `main.tsx`)
  - `ReactDOM.createRoot` 外层没有包裹 `<ErrorBoundary>`，任何页面渲染异常会导致整个应用白屏，无法优雅降级。
  - 建议：增加 `ErrorBoundary` 组件，捕获渲染异常并展示友好错误页。

- **缺少 404 路由与兜底页面** (🟡 警告 / `router.tsx`)
  - `createBrowserRouter` 未配置 `path: '*'` 的 404 页面，用户输入错误 URL 时看到空白或 Layout 内无内容。
  - 建议：增加 `NotFoundPage` 组件并注册到 `*` 路由。

- **缺少自定义 Hooks，业务逻辑与 UI 耦合** (🔵 建议 / 全局)
  - 所有数据获取、轮询、状态管理逻辑直接写在页面组件内，如 `ImportanceDashboard` 超过 600 行，重复逻辑（如 `getStudies` 加载）在多个页面中复制。
  - 建议：提取 `useStudies`、`usePolling`、`useAnalysis` 等自定义 Hooks。

- **无请求去重 / 缓存层** (🔵 建议 / `api.ts`)
  - 多次进入同一页面会重复请求相同接口，没有 SWR / React Query 等缓存策略。
  - 建议：引入 `@tanstack/react-query` 或 `swr`，统一处理缓存、去重、重试、失效。

- **无输入数据校验层** (🔵 建议 / 全局)
  - 前端仅依赖 Ant Design 的 `rules` 做表单校验，API 响应数据没有 Zod / Yup 运行时校验，一旦后端契约变更，TypeScript 类型无法在生产环境保护应用。
  - 建议：对核心 API 响应增加 Zod schema 校验。

- **状态管理简洁合理** (✅ 通过 / `appStore.ts`)
  - Zustand 存储结构清晰，仅包含跨页面共享状态（选中研究、分析任务、运行中任务列表），不滥用全局状态。

- **目录结构清晰** (✅ 通过 / 全局)
  - `pages / components / services / stores / types` 职责分离明确。

### 6. 类型安全

- **SegmentComparison 核心业务结果使用 `any`** (🔴 阻塞 / `SegmentComparison.tsx` 第15行)
  - `const [result, setResult] = useState<any | null>(null)` 使该页面最重要的对比分析结果完全丧失类型保护，后续访问 `result.overall_test`、`result.per_attribute` 均无编译期检查，极易因后端字段变更导致运行时错误。
  - 建议：严格使用 `SegmentComparisonResponse | null` 类型。

- **TypeScript 严格模式已启用** (✅ 通过 / `tsconfig.json`)
  - `strict: true`、`noUnusedLocals: true`、`noUnusedParameters: true` 已配置，编译期约束较强。

- **API 层类型导入存在内联 import** (🟡 警告 / `api.ts` 第167行)
  - `Promise<import('@/types/api').ConvergenceDiagnostics>` 使用内联类型导入，虽然功能正确，但破坏代码一致性，建议统一在顶部 `import type` 声明。

- **Admin Settings 返回类型过于宽泛** (🔵 建议 / `api.ts` 第291–300行)
  - `getAdminSettings` 和 `updateAdminSettings` 使用 `Record<string, unknown>`，丢失了具体字段提示。
  - 建议：定义 `AdminSettingsResponse` 接口。

- **部分页面 useState 缺少显式泛型** (🔵 建议 / 全局)
  - 如 `InterviewLab` 的 `history` 状态类型已显式声明，但 `loading`、`personaLoading` 等布尔值依赖推断。虽然无错误，但显式声明更利于维护。

- **WTP 和 AnalysisResult 中存在 `Record<string, unknown>`** (🟡 警告 / `types/api.ts`)
  - `AnalysisResultResponse.wtp: Record<string, unknown>` 和 `RawDatasetExportResponse.choice_records: Record<string, unknown>[]` 在类型层面未细化，实际消费时缺乏类型安全。
  - 建议：在 TODO 阶段尽快补全具体接口。

### 7. 错误处理

- **API 拦截器全局 message 可能堆叠** (🟡 警告 / `api.ts` 第54–80行)
  - 对每一次请求失败都调用 `message.error()`，当页面同时发起多个请求（如 QuestionnaireConfig 的 N+1 加载）时，错误消息会堆叠弹出，遮挡界面。
  - 建议：使用 `message.config({ maxCount: 3 })` 限制，或错误消息统一收敛到页面级 Alert。

- **部分轮询错误被静默吞掉** (🟡 警告 / `ImportanceDashboard.tsx`、`AnalysisStatus.tsx`)
  - `catch { /* ignore polling errors */ }` 和 `catch { /* polling error */ }` 将轮询失败完全静默，如果后端持续不可用，用户看到的状态将永远停滞，不会收到任何提示。
  - 建议：连续轮询失败 3 次后，降级为页面级 Alert 提示“状态更新中断”。

- **ResponseSimulator 错误信息丢失** (🟡 警告 / `ResponseSimulator.tsx`)
  - `catch (err) { message.error('导出失败') }` 未展示具体错误原因，且未写入 `error` state 供 Alert 展示。

- **axios timeout 60 秒合理** (✅ 通过 / `api.ts`)
  - 对分析类长请求（如 `analyzeStudy`）设置了 60 秒超时，符合业务场景。

- **缺少全局请求重试机制** (🔵 建议 / `api.ts`)
  - 网络瞬时抖动时直接失败，没有指数退避重试。
  - 建议：对 GET 类幂等请求增加 1–2 次自动重试。

- **StudyCreate 问卷生成失败有降级回滚提示** (✅ 通过 / `StudyCreate.tsx`)
  - 在问卷生成失败时明确告知用户“研究已创建但无问卷”，并给出后续操作指引，避免用户困惑。

### 对前端验收结论的影响评估

**结论：维持「有条件通过」，但新增 2 项阻塞项、5 项警告项。**

原有 3 项阻塞问题（API 密钥硬编码、假进度条、`any` 类型）仍优先修复。本次 API 覆盖率审查发现，**`model_type` 与 `simulate mode` 两类核心参数被硬编码，直接架空后端已实现的分析模型切换与模拟模式选择能力**，用户无法根据业务场景选择 `mnl` / `latent_class` 分析模型，也无法使用 `llm` / `deterministic` 模拟模式。这属于“核心功能不可控”级别缺陷，应**追加为 2 项新增阻塞项**：

- 🔴 **阻塞-4**：`AnalysisStatus.tsx` 等硬编码 `model_type: 'hb'`，不暴露 `mnl` / `latent_class` 选择器。
- 🔴 **阻塞-5**：`ResponseSimulator.tsx` 硬编码 `mode: 'stochastic'`，不暴露 `llm` / `deterministic` 选择器。

其余 `StudyCreate.tsx` 不传 `attributes/design_parameters`、`SegmentComparison.tsx` 测试类型硬编码、`InterviewLab.tsx` 仅单轮对话 3 项归入新增警告；12 个未使用端点中，行为模拟入口缺失（批量采访、购买决策）和画像筛选器缺失归入新增警告，其余端点接入工作归入建议级优化。

> 注：本审查同时揭示前期后端 P0-API-001（硬编码洗碗机属性）已在后端修复，但前端 `StudyCreate.tsx` 表单未同步增加对应字段，导致后端灵活性被前端架空——此类“后端已修复/已实现但前端不可见”的功能断层，需在复验时一并闭环。

### 复验条件更新

除原有 3 项阻塞问题外，新增以下进入复验的硬性要求：

1. **分析模型可切换**：`AnalysisStatus` 或同链路页面允许用户选择 `model_type`（至少暴露 `hb` 与 `mnl`），并能正确发起对应分析任务。

3. **分析 ID 下拉框可用**：`MarketSimulator` 或 `SegmentComparison` 的分析 ID 下拉框通过 `GET /studies/{id}/analyses` 或等效数据源填充，不再为空。

原有“警告项完成 60% 以上”要求不变，但建议将 **画像筛选器**、**研究编辑页**、**Dashboard 聚合端点接入** 纳入优先修复范围，以尽快将整体覆盖率从 61% 提升至 80% 以上。
