## 二、逐维度审查详情

### 2.1 前端加载优化

#### 2.1.1 已实施项
- **代码分割（Code Splitting）**：`frontend/vite.config.ts` 中通过 `rollupOptions.output.manualChunks` 将第三方库拆分为 4 个独立 chunk：
  - `vendor-react`（react、react-dom、react-router-dom）
  - `vendor-antd`（antd、@ant-design/icons）
  - `vendor-echarts`（echarts、echarts-for-react）
  - `vendor-axios`（axios）
  这可避免单个 JS 文件过大，提升缓存命中率。
- **路由懒加载**：`frontend/src/router.tsx` 中**所有 13 个页面**均使用 `React.lazy()` + `Suspense` 包裹，首屏仅加载当前路由所需代码，配合 `<Spin>` 加载态，体验良好。
- **构建工具**：采用 Vite 5.4，内置 ESBuild 压缩与 Tree Shaking，构建产物已做基础优化。

#### 2.1.2 缺失项与风险
| 问题 | 风险等级 | 说明 |
|------|----------|------|
| 无 CDN 配置 | 中 | 静态资源未配置 CDN 域名分发，远距离用户加载延迟高 |
| 无图片优化 | 中 | 未见 WebP/AVIF 格式、`<img loading="lazy">`、响应式图片策略 |
| 无字体加载优化 | 中 | `index.html` 未配置 `font-display: swap`，Ant Design 默认字体可能阻塞渲染 |
| 无 Brotli/Gzip 显式配置 | 低 | Vite 构建产物默认未压缩，Nginx 虽可开 gzip，但配置中未显式启用 `gzip_static` |
| 无资源预加载 | 低 | `index.html` 缺少 `<link rel="preconnect">`、`<link rel="preload">` 等关键资源提示 |
| 无 Service Worker / PWA 缓存 | 低 | 未利用 Workbox 等工具做前端离线缓存与增量更新 |

#### 2.1.3 整改建议
1. **高优先级**：在 Nginx 或 CDN 层开启 Brotli/Gzip 压缩；为静态资源配置独立 CDN 域名并添加 `preconnect`。
2. **中优先级**：评估图片资源使用 WebP 格式；为 Ant Design 字体添加 `font-display: swap` 或自托管字体子集化。
3. **低优先级**：引入 `vite-plugin-pwa` 或 Workbox 做构建时 Service Worker 生成。

### 2.2 后端水平扩展

#### 2.2.1 已实施项
- **无状态服务设计**：API 服务不依赖本地内存会话，认证通过 `X-API-Key` 请求头完成，符合水平扩展要求。
- **K8s 多副本与滚动更新**：`k8s/deployment.yaml` 配置 `replicas: 3`，RollingUpdate 策略 `maxSurge: 1, maxUnavailable: 0`，实现零停机发布。
- **HPA 自动扩缩容**：`k8s/deployment.yaml` 中 HorizontalPodAutoscaler 配置：
  - `minReplicas: 3`, `maxReplicas: 10`
  - CPU 目标利用率 70%，内存 80%
  - 扩容策略：15 秒内最多扩容 100% 或 4 Pod；缩容策略：5 分钟稳定窗口，每分钟最多缩容 10%
  该配置能应对日常流量波动，但最大 10 副本可能在突发流量下仍显不足。
- **Nginx 负载均衡**：`docker/nginx.conf` 中 upstream 使用 `least_conn` 算法，并启用 `keepalive 32` 长连接，减少 TCP 握手开销。
- **K8s 探针完善**：同时配置了 Liveness、Readiness、Startup Probe，避免流量打到未就绪实例。

#### 2.2.2 缺失项与风险
| 问题 | 风险等级 | 说明 |
|------|----------|------|
| API_WORKERS=1 | **高** | `docker-compose.yml` 与 `k8s/configmap.yaml` 均设置 `API_WORKERS=1`，Uvicorn 仅单 worker，无法利用多核 CPU；K8s 单 Pod 也只有 1 核限额，整体并发能力被严重限制 |
| 内存限流非分布式 | **高** | `src/aicbc/api/middleware/rate_limit.py` 使用 `InMemoryRateLimiter`，在多 Pod 场景下每个实例独立计数，限流策略失效；注释已明确说明“production should use Redis-backed store” |
| 无 Pod 拓扑分布 | 中 | 未配置 `podAntiAffinity` 或 `topologySpreadConstraints`，多个副本可能调度到同一节点，存在单点故障风险 |
| 无 Graceful Shutdown | 中 | `main.py` 中 `shutdown_event` 仅打印日志，未实现请求排空（drain），滚动更新时可能中断正在处理的 LLM 长连接请求 |
| K8s `readOnlyRootFilesystem: false` | 低 | 容器根文件系统可写，降低安全级别，虽然与性能无直接关联，但影响弹性伸缩时的镜像拉取与启动速度 |

#### 2.2.3 整改建议
1. **高优先级**：将 `API_WORKERS` 调整为基于 CPU 核心数的动态值（如 `$(($(nproc) * 2 + 1))`），或直接使用 Gunicorn + Uvicorn Worker 模式；K8s 中建议单 Pod 资源 limits 与 workers 数量匹配。
2. **高优先级**：将限流中间件迁移至 Redis 后端（Redis + Lua 原子脚本或使用 `fastapi-limiter` 等成熟库），确保多实例共享限流状态。
3. **中优先级**：在 Deployment 中添加 `podAntiAffinity`，确保副本跨节点/可用区分布。
4. **中优先级**：在 `shutdown_event` 中实现 Graceful Shutdown，捕获 SIGTERM 后停止接收新请求并等待现有请求完成（FastAPI 原生支持，需显式配置）。

### 2.3 数据库扩展方案

#### 2.3.1 已实施项
- **连接数上限配置**：`settings.py` 中定义 `mongodb_max_connections=50`，避免了无限制连接耗尽资源。
- **Redis 作为缓存与消息代理**：承担 Celery Broker 与缓存角色，减轻了 MongoDB 直接压力。

#### 2.3.2 缺失项与风险
| 问题 | 风险等级 | 说明 |
|------|----------|------|
| MongoDB 单实例，无副本集 | **高** | 单点故障，无读写分离，无法水平扩展；数据量增长后查询性能将线性下降 |
| 未使用异步驱动 | **高** | `health.py` 中 TODO 明确提到“when motor client is available”，说明当前可能使用同步 PyMongo；在 FastAPI 异步场景下，同步 DB 调用会阻塞事件循环，严重降低并发能力 |
| 连接池配置粗糙 | 中 | 仅设置 `max_connections=50`，未配置 `minPoolSize`、`maxIdleTimeMS`、`waitQueueTimeoutMS`、`serverSelectionTimeoutMS` 等关键参数，无法应对连接抖动 |
| 无分库分表策略 | 中 | 随着 study/consumer 数据增长，单 collection 查询性能将下降；无数据归档或分片策略 |
| Redis 单实例 | 中 | 单点故障，未配置 Redis Sentinel 或 Cluster；Beat 调度依赖 Redis，故障将导致定时任务中断 |
| 健康检查虚假通过 | 中 | `_check_mongodb()`、`_check_redis()` 中实际 ping 逻辑被注释，仅返回固定 `status="ok"`，无法真实检测数据库故障，可能导致 K8s 在 DB 异常时仍向故障 Pod 导流 |

#### 2.3.3 整改建议
1. **高优先级**：引入 `motor` 异步 MongoDB 驱动，替换同步 PyMongo，确保 FastAPI 事件循环不被阻塞。
2. **高优先级**：搭建 MongoDB 副本集（至少 1 Primary + 2 Secondary），配置读写分离（读偏好 `secondaryPreferred`）。
3. **中优先级**：细化连接池参数，配置 `minPoolSize=10`、`maxPoolSize=50`、`maxIdleTimeMS=60000` 等。
4. **中优先级**：为 Redis 配置 Sentinel 或 Cluster 模式，确保高可用；同时启用 Redis 持久化（RDB/AOF）防止数据丢失。
5. **中优先级**：完成 `health.py` 中 TODO，实现真实的数据库 ping 检查，确保 readiness probe 有效。

### 2.4 资源限制

#### 2.4.1 已实施项
- **Docker Compose 资源约束**：
  - API：limit 2G / 1.0 CPU，reservation 512M / 0.25 CPU
  - Worker：limit 2G / 1.0 CPU
  - MongoDB：limit 1G
  - Redis：limit 512M
- **K8s 资源约束**：
  - API：requests 512Mi / 250m，limits 2Gi / 1000m
- **安全上下文**：K8s 中设置 `runAsNonRoot: true`、`runAsUser: 999`、`allowPrivilegeEscalation: false`、`capabilities: drop: [ALL]`，符合安全最佳实践。

#### 2.4.2 缺失项与风险
| 问题 | 风险等级 | 说明 |
|------|----------|------|
| Beat 无资源限制 | 中 | `docker-compose.yml` 中 `beat` 服务未设置 `deploy.resources`，存在资源耗尽风险（虽然负载低，但缺乏边界） |
| MongoDB/Redis 无 CPU 限制 | 低 | 仅限制了内存，未限制 CPU，极端情况下可能抢占宿主机 CPU |
| Worker 无资源预留 | 低 | 仅设置了 limits，未设置 reservations，在资源紧张时可能被频繁驱逐 |
| Dockerfile 复制 `/usr/local/bin` 过于宽泛 | 低 | 多阶段构建中从 builder 复制整个 `/usr/local/bin`，可能引入不必要的二进制文件，增加镜像体积与攻击面 |

#### 2.4.3 整改建议
1. 为 `beat` 补充资源 limits（建议 512M / 0.5 CPU）。
2. 为 MongoDB 和 Redis 补充 CPU limits（建议各 1.0 CPU）。
3. 优化 Dockerfile，仅复制运行所需的二进制文件（如 `uvicorn`、`celery`），而非整个 `/usr/local/bin`。

### 2.5 性能监控

#### 2.5.1 已实施项
- **Prometheus + Grafana 基础设施**：`docker-compose.yml` 中完整部署了 Prometheus（v2.51）和 Grafana（10.4），并配置数据卷持久化。
- **Metrics 命名规范**：`metrics.py` 中遵循 `aicbc_<domain>_<metric>_<unit>` 命名，包含业务、系统、安全三大类指标。
- **Histogram 延迟分布**：
  - `aicbc_api_request_duration_seconds`：buckets 覆盖 0.01s ~ 30s
  - `aicbc_llm_latency_seconds`：buckets 覆盖 0.1s ~ 60s
  - `aicbc_simulation_duration_seconds`：buckets 覆盖 1s ~ 600s
- **Counter 吞吐量指标**：`aicbc_api_requests_total`、`aicbc_llm_api_calls_total`、`aicbc_llm_tokens_total` 等。
- **安全事件监控**：`aicbc_injection_attempts_total`、`aicbc_authentication_failures_total` 等。
- **懒加载初始化**：`metrics.py` 通过 `lru_cache` 延迟导入 `prometheus_client`，避免 Windows 环境下 C 扩展启动崩溃，设计巧妙。
- **健康与成本监控**：`/health`、`/ready`、`/metrics`、`/cost-status` 四个端点覆盖健康、依赖、指标、成本四大维度。

#### 2.5.2 缺失项与风险
| 问题 | 风险等级 | 说明 |
|------|----------|------|
| 无 P95/P99 告警规则 | **高** | 虽然 histogram 可以计算分位数，但项目未提供 Prometheus Recording Rules 或 Alertmanager 告警配置；缺乏对长尾延迟的主动监控 |
| 无吞吐量（RPS/QPS）专用指标 | 中 | `api_requests_total` 是 Counter，但缺少基于时间的 RPS 计算面板或告警；无法快速判断流量突增 |
| 无错误预算（Error Budget）监控 | 中 | 未配置 SLO/SLI（如 99.9% 的请求延迟 < 2s），无法量化系统可靠性 |
| 慢请求阈值偏宽松 | 低 | `SLOW_REQUEST_THRESHOLD=5.0` 秒，对普通 REST API 而言阈值过高，可能掩盖后端性能退化 |
| 无 APM（分布式追踪） | 低 | 未集成 Jaeger/Tempo 等分布式追踪，LLM 调用链路长，故障定位困难 |
| Grafana Dashboard 未纳入版本控制 | 低 | `docker/grafana/dashboards` 目录存在，但无法确认是否包含完整的关键性能面板定义 |

#### 2.5.3 整改建议
1. **高优先级**：在 Prometheus 中增加 Recording Rules，计算 `aicbc_api_request_duration_seconds` 的 P95/P99；配置 Alertmanager 告警规则（如 P99 > 3s 持续 5 分钟触发告警）。
2. **中优先级**：创建 Grafana Dashboard，展示 RPS、错误率、P95/P99 延迟、LLM 调用成功率、队列长度等核心面板。
3. **中优先级**：定义 SLO（如 99.5% 的 API 请求 < 2s），基于错误预算配置 burn rate 告警。
4. **低优先级**：评估接入 OpenTelemetry + Jaeger，对 LLM 调用、Celery 任务执行链路进行分布式追踪。

### 2.6 压力测试

#### 2.6.1 现状
- **无任何压力测试脚本、方案或报告**。审查范围包括 `tests/`、`scripts/`、`docs/` 目录及项目根目录，均未发现以下任一工具的使用痕迹：
  - k6、Locust、JMeter、wrk、ab、ghz、gatling 等。
- **仅有的性能相关测试**：`tests/analysis/test_hb_convergence_validation.py` 中包含 `test_benchmark_n50`，但这是算法级微基准测试，非系统级压力测试。
- **无任何性能基线数据**：未记录系统在正常/峰值负载下的 QPS、P95 延迟、CPU/内存使用率、错误率等基线指标。

#### 2.6.2 风险
| 问题 | 风险等级 | 说明 |
|------|----------|------|
| 无容量规划数据 | **高** | 无法回答“系统能承载多少并发用户”、“单实例 QPS 上限是多少”等核心问题 |
| 无法验证 HPA 有效性 | **高** | HPA 配置基于 CPU 70% 扩容，但未验证在真实负载下 Pod 数是否能正确伸缩 |
| 无法发现性能瓶颈 | **高** | 未经过负载测试，MongoDB 同步驱动、单 worker、内存限流等瓶颈未暴露 |
| 上线后风险未知 | **高** | 缺乏压测通过标准，无法判断系统是否具备生产环境承载能力 |

#### 2.6.3 整改建议
1. **高优先级**：制定并执行压力测试方案，建议覆盖以下场景：
   - **基线测试**：50 并发用户，持续 5 分钟，记录 QPS、P95/P99、错误率、资源使用率。
   - **峰值测试**：200 并发用户，持续 2 分钟，观察 HPA 是否能正确扩容至 10 Pod。
   - **稳定性测试**：100 并发用户，持续 30 分钟，验证内存泄漏、连接池耗尽、Redis/MongoDB 稳定性。
   - **关键路径测试**：重点压测 `/personas/generate`、`/studies/*/simulate-responses` 等 LLM 密集型接口（注意控制 LLM 调用成本，可使用 mock LLM 服务端）。
2. **高优先级**：引入压测工具。推荐方案：
   - 使用 **k6**（JavaScript/TypeScript 脚本，易于 CI 集成）编写 API 压测脚本。
   - 或 **Locust**（Python 脚本，团队技术栈一致）。
3. **中优先级**：在 CI/CD 流水线中集成性能回归测试（如每次合并前执行 5 分钟基准压测，对比 QPS 退化 > 10% 则阻断）。
