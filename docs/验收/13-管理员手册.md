### 2.2 Docker Compose 部署（推荐 — MVP 阶段）

#### Step 1：获取代码

```bash
git clone <repository-url> /opt/aicbc
cd /opt/aicbc
```

#### Step 2：配置环境变量

```bash
# 复制示例配置文件
cp .env.example .env

# 编辑 .env，填写以下必填项
vim .env
```

**必填环境变量**：

| 变量名 | 说明 | 示例 |
|--------|------|------|
| `ANTHROPIC_API_KEY` | Anthropic Claude API 密钥 | `sk-ant-xxx` |
| `OPENAI_API_KEY` | OpenAI GPT API 密钥（备用） | `sk-xxx` |
| `GRAFANA_ADMIN_PASSWORD` | Grafana 管理员密码 | `changeme` |

**可选环境变量**（使用默认值即可）：

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `ENVIRONMENT` | `production` | 运行环境 |
| `LOG_LEVEL` | `INFO` | 日志级别 |
| `API_WORKERS` | `1` | API 工作进程数 |
| `SLOW_REQUEST_THRESHOLD` | `5.0` | 慢请求阈值（秒） |

#### Step 3：启动服务

```bash

docker compose up -d

# 查看服务状态
```

### 2.4 Kubernetes 部署（生产环境）

> ⚠️ 以下配置已就绪，但需在 K8s 集群上验证后投入生产。

#### 前置条件

- K8s 集群 ≥ 1.28
- kubectl 已配置
- 镜像仓库可访问（如 Harbor、ECR、ACR）

#### 快速部署

```bash
# 1. 创建命名空间
kubectl apply -f k8s/namespace.yaml

# 2. 创建配置和密钥
kubectl apply -f k8s/configmap.yaml
# 编辑 k8s/secret.yaml，将 API Key 进行 base64 编码后填入
kubectl apply -f k8s/secret.yaml

# 3. 部署核心服务
kubectl apply -f k8s/deployment.yaml

# 4. 暴露服务和 Ingress
kubectl apply -f k8s/ingress.yaml

# 5. 验证部署
kubectl get pods -n aicbc
kubectl get svc -n aicbc
kubectl get ingress -n aicbc
```

#### K8s 资源说明

| 文件 | 资源 | 说明 |
|------|------|------|
| `namespace.yaml` | Namespace | 隔离生产环境 |
| `configmap.yaml` | ConfigMap | 非敏感配置（数据库 URL、日志级别） |
| `secret.yaml` | Secret | 敏感信息（API Key、数据库密码） |
| `deployment.yaml` | Deployment + HPA | API 服务、Worker、Beat；自动扩缩容 |
| `ingress.yaml` | Ingress | Nginx Ingress Controller 路由规则 |

#### K8s 扩缩容参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 初始副本数 | 3 | API 服务 |
| 最小副本数 | 3 | 保证高可用 |
| 最大副本数 | 10 | CPU > 70% 时触发扩容 |
| Worker 副本 | 2 | 可独立扩容至 20 |

## 三、备份与恢复

### 3.1 MongoDB 备份

#### 自动备份脚本（每日）

```bash
#!/bin/bash
# /opt/aicbc/scripts/backup-mongo.sh

BACKUP_DIR="/opt/aicbc/backups/mongo"
DATE=$(date +%Y%m%d_%H%M%S)
DB_NAME="aicbc"
RETENTION_DAYS=30

# 创建备份
mkdir -p ${BACKUP_DIR}
docker compose exec -T mongo mongodump --db=${DB_NAME} --archive > ${BACKUP_DIR}/${DB_NAME}_${DATE}.archive

# 压缩
gzip ${BACKUP_DIR}/${DB_NAME}_${DATE}.archive

# 清理旧备份
find ${BACKUP_DIR} -name "*.archive.gz" -mtime +${RETENTION_DAYS} -delete

echo "Backup completed: ${BACKUP_DIR}/${DB_NAME}_${DATE}.archive.gz"
```

#### 添加定时任务（crontab）

```bash
# 每天凌晨 2 点执行备份
0 2 * * * /opt/aicbc/scripts/backup-mongo.sh >> /var/log/aicbc-backup.log 2>&1
```

#### 手动备份

```bash
# 全库备份
docker compose exec mongo mongodump --out=/backup/aicbc_$(date +%Y%m%d)

# 备份文件在宿主机路径：./backups/mongo/
ls -la ./backups/mongo/
```

### 3.2 MongoDB 恢复

```bash
# 从备份恢复（注意：会覆盖当前数据）
docker compose exec -T mongo mongorestore --db=aicbc --archive < ./backups/mongo/aicbc_20260613.archive

# 或从目录恢复
docker compose exec mongo mongorestore --db=aicbc /backup/aicbc_20260613/aicbc

# 恢复后验证数据完整性
docker compose exec mongo mongosh --eval "db.getCollectionNames()"
```

### 3.3 Redis 备份

```bash
# 手动触发 RDB 保存
docker compose exec redis redis-cli BGSAVE

# 备份文件在 Docker 卷中，宿主机路径通常位于：
# /var/lib/docker/volumes/aicbc_redis_data/_data/dump.rdb

# 如需复制备份到安全位置
cp /var/lib/docker/volumes/aicbc_redis_data/_data/dump.rdb /opt/aicbc/backups/redis/redis_$(date +%Y%m%d).rdb
```

### 3.4 Redis 恢复

```bash
# 停止 Redis 容器
docker compose stop redis

# 替换 dump.rdb
cp /opt/aicbc/backups/redis/redis_20260613.rdb /var/lib/docker/volumes/aicbc_redis_data/_data/dump.rdb

# 启动 Redis 容器
docker compose start redis
```

### 3.5 备份策略总结

| 数据类型 | 备份频率 | 保留期 | 存储位置 | 加密 |
|---------|---------|--------|---------|------|
| MongoDB | 每日自动 + 升级前手动 | 30 天 | 本地 + 异地（建议） | 建议加密 |
| Redis | 每日（RDB）+ 升级前手动 | 7 天 | 本地 | — |
| 配置文件 | 每次变更 | 永久 | Git | — |
| Grafana 仪表盘 | 配置变更时 | 永久 | Git + Grafana 导出 JSON | — |

### 4.3 核心告警规则

在 Prometheus 中配置告警规则（`docker/prometheus/rules/`）：

```yaml
# 业务告警
- alert: HighCostPerPersona
  expr: cost_per_persona_cny > 5
  for: 1h
  labels:
    severity: warning
  annotations:
    summary: "单个消费者成本过高"
    description: "当前成本 ¥{{ $value }}/人，超过阈值 ¥5"

- alert: LowAuthenticityScore
  expr: authenticity_score_average < 8
  for: 30m
  labels:
    severity: critical
  annotations:
    summary: "消费者真实性评分过低"
    action: "暂停生成，检查Prompt模板"

# 系统告警
- alert: HighErrorRate
  expr: rate(api_requests_total{status=~"5.."}[5m]) / rate(api_requests_total[5m]) > 0.1
  for: 5m
  labels:
    severity: critical
  annotations:
    summary: "API错误率过高"

- alert: LLMAPIRateLimit
  expr: rate(llm_api_calls_total{status="429"}[5m]) > 0
  for: 1m
  labels:
    severity: warning
  annotations:
    summary: "LLM API触发速率限制"

- alert: QueueBacklog
  expr: queue_size > 100
  for: 10m
  labels:
    severity: warning
  annotations:
    summary: "任务队列堆积"

# 安全告警
- alert: InjectionAttackDetected
  expr: rate(injection_attempts_total{blocked="true"}[5m]) > 5
  for: 1m
  labels:
    severity: critical
  annotations:
    summary: "检测到注入攻击"
```

## 五、应急处理

### 5.1 成本熔断

**触发条件**：
- 单次研究成本 > ¥800（警戒线）
- 单日累计成本 > ¥2,000
- 单个消费者成本 > ¥5/人

**自动熔断动作**（系统已内置）：
1. 暂停新的画像生成和模拟任务
2. 正在进行的任务允许完成，但不再接收新任务
3. 发送告警通知（小控 + 小P）

**管理员手动处理**：

```bash
# 查看当前队列
docker compose exec redis redis-cli LLEN celery

# 如需清空队列（极端情况）
docker compose exec redis redis-cli FLUSHDB

# 查看近期成本日志
docker compose logs api | grep "cost"

# 检查 .env 中是否配置了成本上限
# COST_PER_STUDY_LIMIT=500
# DAILY_COST_LIMIT=2000
```

**恢复服务**：
1. 确认成本异常原因（检查是否有人为超大样本请求、LLM 模型异常消耗等）
2. 修复成本配置或限制参数
3. 重启 API 服务：`docker compose restart api`

---

### 5.2 LLM API 不可用

**现象**：
- 画像生成或模拟任务大量失败
- 日志中出现 `429 Rate Limit`、`500 Internal Server Error`、`Connection Timeout`

**排查步骤**：

```bash
# 1. 检查 API Key 是否有效
curl -H "x-api-key: $ANTHROPIC_API_KEY" https://api.anthropic.com/v1/models

# 2. 检查 API 配额
docker compose logs api | grep -i "rate limit\|quota\|auth"

# 3. 检查模型路由状态
docker compose logs api | grep -i "model_router\|fallback"
```

**应急处理**：

| 场景 | 处理动作 | 命令 |
|------|---------|------|
| 主 LLM 服务商故障 | 切换至备用模型（OpenAI） | 在设置页面切换模型路由；或修改环境变量重启 |
| API Key 过期 | 更新密钥 | 修改 `.env` 中 `ANTHROPIC_API_KEY`，执行 `docker compose up -d` |
| 速率限制 | 降低并发 | 减少 Worker 并发数：`celery -A aicbc.tasks worker --concurrency=1` |
| 全部 LLM 不可用 | 暂停服务 | 在 Nginx 层返回 503，提示用户暂不可用 |

### 5.3 服务重启与恢复

#### 单个服务重启

```bash
# API 服务重启
docker compose restart api

# Worker 重启（如 Worker 卡住）
docker compose restart worker

# 全部服务重启
docker compose restart
```

#### 数据卷损坏后的重建

```bash
# 1. 停止所有服务
docker compose down

# 2. 备份现有数据（即使可能损坏）
cp -r /var/lib/docker/volumes/aicbc_mongo_data /tmp/mongo_data_backup

# 3. 删除损坏卷（⚠️ 谨慎操作）
docker volume rm aicbc_mongo_data

# 4. 重新创建并启动
docker compose up -d

# 5. 从备份恢复数据（如有可用备份）
docker compose exec -T mongo mongorestore --db=aicbc --archive < ./backups/mongo/aicbc_latest.archive
```

#### 灾难恢复时间目标（RTO/RPO）

| 场景 | RTO（恢复时间目标） | RPO（恢复点目标） | 恢复方式 |
|------|-------------------|-------------------|---------|
| 单服务故障 | < 5 分钟 | 0 | 自动重启 / 手动 restart |
| 单节点故障 | < 30 分钟 | 最近日备份 | Docker Compose 重建 + 数据恢复 |
| 全系统故障 | < 2 小时 | 最近日备份 | 从备份重建 + 验证 |
| API Key 泄露 | < 15 分钟 | 0 | 紧急轮换密钥 + 滚动更新 |

### 7.1 网络层安全

```nginx
# docker/nginx.conf 示例：基础认证 + IP 白名单
server {
    listen 80;
    
    # 基础认证（临时替代方案）
    auth_basic "AI_CBC Admin";
    auth_basic_user_file /etc/nginx/.htpasswd;
    
    # IP 白名单（仅限内网）
    allow 10.0.0.0/8;
    allow 172.16.0.0/12;
    allow 192.168.0.0/16;
    deny all;
    
    location / {
        proxy_pass http://api:8000;
    }
}
```

