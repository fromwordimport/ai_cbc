# 消费者模拟 Agent — 输出信息规范

> **版本**：v1.0
> **定位**：定义消费者模拟 Agent 在所有交互模式下必须输出的核心信息，以及可按需启用的附加输出
> **使用说明**：输出规范与 `08-Agent输入信息规范.md` 形成对称的 I/O 契约。输入决定"演什么"，输出决定"演得怎么样"

---

## 一、基本输出信息（Minimum Required Outputs）

缺少任何一类，输出就不完整，无法通过 Harness 校验。

---

### 1.1 第一人称反应（First-Person Response）

**定义**：Agent 以消费者画像的口吻、语言风格和认知水平，对当前情境做出的即时反应。这是消费者"演出来"的部分，**不是第三人称分析**。

**输出要求**：
- 必须使用第一人称："我觉得""我看到""我刚想"
- 必须符合人物语言风格：口头禅、句式长度、网络用语、方言特征
- 必须体现认知边界：不知道的不要说，不确定的用"大概""可能"
- 允许碎片化、不连贯、有停顿

**示例**：

```markdown
✅ 合格：
"又是这种…'熬夜急救'。讲真我第一眼是有点心动的，毕竟这周每天都在赶需求。
但评论区一水儿的好评让我有点警觉，太像刷的了。'以油养肤'到底是真有用还是新智商税？
我脸本来就油，再抹油不会闷痘吗…"

❌ 不合格：
"作为一个月收入2万元的一线城市产品经理，我对'以油养肤'这一新兴护肤概念持谨慎态度。
我需要进一步研究产品成分和用户评价后再做决策。"
（问题：过于理性、结构化、像市场分析报告）
```

---

### 1.2 情绪状态标注（Emotional State）

**定义**：明确标注 Agent 在当前情境下的主导情绪，以及情绪的来源和强度变化。

**输出格式**：

```markdown
【情绪状态】
- 主导情绪：{从标准情绪词中选择}
- 强度：{低(1-3) / 中(4-6) / 高(7-10)}
- 触发源：{具体是什么引发了这种情绪}
- 情绪变化路径：{上一状态 → 当前状态 → 预期下一状态}
```

**标准情绪词表**：

| 情绪类别 | 具体情绪 | 典型消费行为 |
|---------|---------|------------|
| 正向 | 兴奋、满足、期待、安心 | 冲动购买、溢价接受、主动分享 |
| 负向 | 焦虑、疲惫、防御、后悔、愤怒 | 比价延长、容易放弃、投诉倾向 |
| 中性 | 平静、无聊、漠然、犹豫 | 惯性决策、拖延、依赖默认选项 |
| 复杂 | 矛盾、愧疚、侥幸、怀旧 | 凑单、买后后悔、为情怀付费 |

---

### 1.3 采取的行动（Action Taken）

**定义**：Agent 在情境中实际做出的行为或决定。"行动"不等于"购买"，而是人在面对信息时的真实反应动作。

**输出格式**：

```markdown
【采取的行动】
- 即时反应（3秒内）：{下意识行为}
- 短期行为（1-5分钟）：{接下来会做的事}
- 最终决策：{购买/放弃/延迟/咨询/分享}
- 决策置信度：{高(>80%) / 中(50-80%) / 低(<50%) / 摇摆}
```

**行动分类参考**：

```markdown
信息处理类：滑动浏览 / 点击查看详情 / 搜索比价 / 查看中差评 / 询问朋友
决策推进类：直接购买 / 加入购物车 / 加入收藏 / 领券等待 / 凑单计算 / 关闭离开
情绪表达类：写评价 / 点赞收藏 / 投诉退货 / 向朋友吐槽
```

---

### 1.4 决策理由（Decision Rationale）

**定义**：解释 Agent 为什么会这样想/这样做。包括**表面理由**（说得出口的）和**深层理由**（不愿公开承认的真实动机）。

**输出格式**：

```markdown
【决策理由】
- 表面理由：{可以告诉别人的原因}
- 深层理由：{不愿公开承认的真实动机}
- 关键影响因素：{哪些信息/标签/情境起了决定性作用}
- 放弃/犹豫的原因：{如果有的话}
```

**示例**：

```markdown
- 表面理由："成分不太熟悉，想先研究一下，不想盲目跟风"
- 深层理由："其实我很怕被归类为那种'看到广告就买的韭菜'，
  尤其是我在闺蜜圈里一直自诩理性。直接买会让我觉得自己输了。"
- 关键影响因素：
  1. 信任模式 = "怀疑一切"，对广告天然不信任
  2. 信息处理风格 = "口碑党"，需要熟人背书
  3. 情绪状态 = 防御（最近刚买过一个踩雷产品）
- 犹豫原因："没有看到真实熟人的使用反馈 + 评论区太像刷的 + 价格不算低"
```

## 二、四类输出的逻辑链条

```
        情境输入
           │
           ▼
    ┌──────────────┐
    │ 第一人称反应 │  ← "我现在这么想/说"
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  情绪状态    │  ← "我为什么这么想/说"
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  采取的行动  │  ← "我做了什么"
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  决策理由    │  ← "我为什么这么做"
    └──────────────┘
```

**检查标准**：从"第一人称反应"到"决策理由"应该能形成一条可解释的链条。如果四类输出之间出现断裂，说明 Agent 在"背诵标签"而非"扮演人物"。

## 三、可选输出信息（Enhanced Outputs）

根据业务场景和研究目标，可以选择性要求 Agent 输出附加信息。

---

### 3.1 注意力分配（Attention Allocation）

**定义**：Agent 在面对信息时，注意力是如何分布的——先看什么、再看什么、忽略什么。

**适用场景**：UX 诊断、广告效果测试、页面设计优化

**输出格式**：

```markdown
【注意力分配】
- 第一眼注意到：{信息点}
- 第二眼扫到：{信息点}
- 完全忽略：{信息点}
- 回头再看：{信息点}
- 注意力停留时间预估：{极短<3s / 短3-10s / 中10-30s / 长>30s}
```

---

### 3.2 认知负荷变化（Cognitive Load Trajectory）

**定义**：Agent 在交互过程中的认知资源消耗变化轨迹。

**适用场景**：流程复杂度评估、信息架构优化、决策摩擦点识别

**输出格式**：

```markdown
【认知负荷变化】
- 初始负荷：{高/中/低}
- 峰值负荷时刻：{哪个环节最费脑子}
- 负荷骤降时刻：{哪个环节让 Agent 突然"放弃思考"}
- 过载表现：{回归默认选项 / 依赖单一线索 / 延迟决策}
```

---

### 3.3 信息缺口（Information Gap）

**定义**：Agent 认为还需要什么信息才能做出决策——这些缺口是业务可以补足的。

**适用场景**：内容策略优化、详情页设计、客服话术完善

**输出格式**：

```markdown
【信息缺口】
- 缺失信息1：{具体缺什么}
  - 重要性：{高/中/低}
  - 如果提供该信息，决策会如何变化：{预测}
- 缺失信息2：...
```

---

### 3.4 内心冲突过程（Internal Conflict）

**定义**：Agent 在决策过程中的心理拉锯——大脑说不要，身体很诚实。

**适用场景**：理解购买阻力、设计说服路径、优化转化漏斗

**输出格式**：

```markdown
【内心冲突】
- 正方（想买）：{理由}
- 反方（不买）：{理由}
- 天平倾向：{正 / 反 / 平衡}
- 打破平衡的关键信息：{什么能 tipping the scale}
```

---

### 3.5 输出元数据（Output Metadata）

**定义**：关于本次模拟本身的结构化信息，便于后续分析和归档。

**适用场景**：批量分析、A/B 测试、质量追踪、洞察提取

**输出格式**：

```yaml
output_metadata:
  persona_id: "成都-28F-产品经理-焦虑型-001"
  scenario: "护肤品新品测试"
  task_type: "购买决策模拟"
  interaction_duration: "5分32秒"
  turn_count: 12
  final_state:
    emotion: "焦虑→犹豫→轻微后悔"
    decision: "加入购物车，未付款"
    authenticity_score: 10.5
  key_moments:
    - timestamp: "T+0:15"
      event: "看到'限量'标签，情绪转为焦虑"
    - timestamp: "T+2:30"
      event: "发现需要凑单，产生抵触"
    - timestamp: "T+4:45"
      event: "最终因'不想研究成分'而放弃"
  business_insights:
    - "该画像对'限量'高度敏感，但对'成分党话术'无感"
    - "决策放弃主因：凑单门槛过高，非价格本身"
```

## 六、输出质量控制清单

每次 Agent 输出后，用以下清单检查基本输出是否合格：

```markdown
□ 第一人称反应是否使用了画像本人的口吻？（非第三人称分析）
□ 情绪状态是否与画像标签和当前情境一致？
□ 情绪是否有明确的触发源？（非凭空出现）
□ 采取的行动是否具体？（非笼统的"考虑一下"）
□ 行动是否与情绪状态匹配？（焦虑时不会果断下单）
□ 决策理由是否区分了表面理由和深层理由？
□ 深层理由是否与画像的"秘密动机"一致？
□ 四类输出之间是否形成了逻辑链条？
□ 是否有"认知边界"的体现？（不知道的不会胡说）
□ 是否有"社会摩擦感"？（犹豫、面子、矛盾等真实特征）
```

## 七、与输入规范的映射关系

| 输入（08） | 输出（09） | 映射关系 |
|-----------|-----------|---------|
| 消费者画像 | 第一人称反应 | 画像决定"我是谁"，反应体现"我怎么说" |
| 当前情境 | 情绪状态 | 情境决定情绪起点 |
| 交互任务 | 采取的行动 | 任务类型决定行动分类 |
| 心理引擎（秘密动机） | 决策理由（深层理由） | 秘密动机是深层理由的源头 |
| 场景反应库 | 情绪状态 + 行动 | 输出验证场景反应库是否准确 |
| 产品/服务信息 | 注意力分配 | 产品信息是注意力分配的靶子 |
| 历史记忆 | 决策理由 | 历史记忆影响理由的权重 |

## 八、与 Harness 校验的衔接

本规范中的四类基本输出，是 Harness 中 **Authenticity Scorer（真实感评分）** 的重要输入：

| 输出类别 | 主要校验维度 |
|---------|------------|
| 第一人称反应 | 语言自然度、知识边界感 |
| 情绪状态 | 情境敏感性、内在一致性 |
| 采取的行动 | 认知有限性、社会摩擦感 |
| 决策理由 | 时间延续性、内在一致性 |

如果 Agent 在输出中缺失某一类信息，或四类信息之间逻辑断裂，Harness 应拒绝通过并要求补充。

## 九、存储设计（高并发版）

> **目标**：支持 **1000 QPS/TPS** 的高并发读写。
> 
> **核心设计**：文件系统方案仅适用于 **< 50 QPS** 的轻量场景。超过 100 QPS 必须升级为**分层存储架构**（内存缓存 + 文档数据库 + 对象存储）。

---

### 9.1 并发瓶颈分析（为什么文件系统扛不住）

| 瓶颈点 | 问题描述 | 1000 QPS 下的表现 |
|--------|---------|-------------------|
| **文件句柄耗尽** | 每次读取需 `open()` 文件 | Linux 默认单进程句柄上限 1024，瞬间打满 |
| **索引文件锁竞争** | `simulation-index.jsonl` 追加写需串行化 | 1000 条/秒全部排队，延迟指数级上升 |
| **JSON 解析 CPU 开销** | 每条记录需完整解析 JSON | 1000 次/秒 × 平均 5KB ≈ 5MB/s 解析压力，单核 CPU 100% |
| **目录遍历成本** | `by-persona/{id}/` 仍需 `readdir()` | 画像数量 > 1000 时，遍历耗时 > 10ms |
| **写放大** | 更新索引 + 写记录 + 更新缓存 = 3 次 IO | 1000 TPS × 3 IO × 4KB ≈ 12MB/s，延迟不稳 |

**结论**：文件系统方案适合 **< 50 QPS**，超过 100 QPS 必须架构升级。

---

### 9.2 分层存储架构总览

```
┌─────────────────────────────────────────────┐
│  Layer 1: 内存缓存（热数据）                   │
│  Redis / KeyDB（单机 10万+ QPS）              │
│  - 最近活跃的画像                             │
│  - 最近 24 小时的模拟记录                      │
│  - 热点标签索引                               │
├─────────────────────────────────────────────┤
│  Layer 2: 文档数据库（温数据）                 │
│  MongoDB / PostgreSQL(JSONB)                  │
│  - 全部画像（10万级）                         │
│  - 全部模拟记录（百万级）                      │
│  - 全文检索 + 标签索引                         │
│  - 读写分离：Primary 写，Replica 读           │
├─────────────────────────────────────────────┤
│  Layer 3: 对象存储（冷数据）                   │
│  S3 / MinIO / OSS                             │
│  - 归档的原始 Markdown 报告                    │
│  - 超过 90 天的 JSON Lines 批次文件            │
│  - 压缩存储，按需拉取                          │
└─────────────────────────────────────────────┘
```

---

### 9.3 Layer 1: 内存缓存（Redis）

**职责**：扛住 90% 的读取流量。

```yaml
redis_config:
  部署模式: "Cluster（3主3从）或 单机（<50 QPS时）"
  单节点内存: 64GB
  总集群内存: 192GB（可用约 128GB）
  
  数据结构:
    画像缓存:
      类型: Hash
      Key: "persona:{id}"
      字段: [profile_json, metadata, ttl=7天]
    
    模拟记录缓存:
      类型: String
      Key: "sim:{id}"
      内容: JSON 序列化后的完整记录
      TTL: 24小时（热数据）/ 7天（温数据）
    
    标签索引缓存:
      类型: Set
      Key: "tag:{tag_name}:{tag_value}"
      示例: "tag:age_group:25-34"
      用途: O(1) 按标签筛选画像
    
    场景聚合缓存:
      类型: SortedSet
      Key: "scenario:{name}"
      分值: 模拟时间戳
      用途: 快速获取某场景下的全部模拟记录
  
  缓存策略:
    读路径: "先读Redis → 命中返回 / 未命中读DB → 回填Redis"
    写路径: "先写DB → 再写Redis → 异步更新索引"
    淘汰: "LRU + maxmemory-policy allkeys-lru"
  
  性能指标:
    单机QPS: 100,000+
    网络延迟: < 1ms（同机房）
    集群模式: 可横向扩展至百万级QPS
```

### 9.5 Layer 3: 对象存储（S3/MinIO）

**职责**：归档冷数据，降低数据库成本。

```yaml
object_storage_config:
  存储内容:
    - 原始 Markdown 画像报告（人类审阅用）
    - 超过 90 天的 JSON Lines 批次文件
    - 压缩后的历史模拟记录（gzip，压缩率约 70%）
  
  生命周期规则:
    0-7天: "仅存在于 MongoDB + Redis"
    8-90天: "MongoDB（查询用）+ S3（备份）"
    90天+: "仅保留于 S3，MongoDB 中删除，需要时从 S3 恢复"
  
  读取策略:
    常规查询: 走 MongoDB
    历史归档查询: 从 S3 拉取 + 本地解压缓存
```

### 9.6 数据流设计

#### 写入流程（模拟记录生成）

```
Agent 生成输出
    │
    ▼
┌─────────────────┐     ┌─────────────────┐
│ 1. 写 MongoDB   │────→│ 2. 发 Kafka     │
│    Primary      │同步  │    消息队列     │异步削峰
└─────────────────┘     └────────┬────────┘
                                 │
                    ┌────────────┼────────────┐
                    ▼            ▼            ▼
            ┌──────────┐  ┌──────────┐  ┌──────────┐
            │ 3. 写    │  │ 4. 更新  │  │ 5. 写 S3 │
            │ Redis    │  │ Redis索引│  │ （可选） │
            │ 缓存     │  │          │  │          │
            └──────────┘  └──────────┘  └──────────┘
```

**关键设计**：
- 步骤 1 同步写（保证一致性），步骤 2-5 异步写（削峰）
- Kafka 缓冲写入峰值，消费者批量消费（100 条/批）
- MongoDB 使用 **Bulk Write**，减少网络往返

#### 读取流程（查询模拟记录）

```
用户请求
    │
    ▼
┌─────────────────┐
│ 1. 查 Redis     │ ← 先查缓存
│    (sim:{id})   │
└────────┬────────┘
    命中？ │
   是 / 否
    │     ▼
    │  ┌─────────────────┐
    │  │ 2. 查 MongoDB   │ ← 缓存未命中
    │  │    Secondary    │
    │  └────────┬────────┘
    │      命中？│
    │     是 / 否
    │      │     ▼
    │      │  ┌─────────────────┐
    │      │  │ 3. 查 S3        │ ← 数据库也没有
    │      │  │    归档文件     │
    │      │  └─────────────────┘
    │      │
    ▼      ▼
┌─────────────────┐
│ 4. 回填 Redis   │ ← 无论哪里读到，都回填缓存
│    (下次命中)   │
└─────────────────┘
```

### 9.7 容量测算（1000 QPS/TPS）

| 指标 | 测算 | 结果 |
|------|------|------|
| **峰值写入** | 1000 TPS × 5KB/条 | 5MB/s 写带宽 |
| **峰值读取** | 1000 QPS × 5KB/条 | 5MB/s 读带宽 |
| **日写入量** | 1000 × 86400 × 5KB | 432GB/天 |
| **Redis 内存** | 热数据 7 天 × 432GB × 10% | ~300GB（需集群） |
| **MongoDB 存储** | 温数据 90 天 × 432GB | ~39TB（需分片） |
| **S3 年存储** | 冷数据归档，压缩后 | ~100TB/年 |

**最低 viable 部署规格**：

```yaml
redis_cluster:
  节点数: 6（3主3从）
  单节点内存: 64GB
  总内存: 192GB（可用约 128GB）
  QPS能力: 300,000+

mongodb_cluster:
  分片数: 3
  每分片: 1 Primary + 2 Secondary
  单节点存储: 2TB SSD
  总存储: 18TB（3分片 × 3副本 × 2TB）
  QPS能力: 15,000+（混合读写）

kafka_cluster:
  节点数: 3
  用途: 削峰填谷，缓冲写入峰值
  吞吐量: 100,000+ TPS（批量写入）
```

### 9.8 降级预案

| 故障场景 | 降级策略 | 影响 |
|---------|---------|------|
| **Redis 故障** | 直接读 MongoDB | 延迟从 1ms → 10ms，服务可用 |
| **MongoDB Primary 故障** | 禁止写入，Secondary 提升为 Primary | 写中断 10-30 秒，读不受影响 |
| **MongoDB 全集群故障** | 降级为文件系统模式（只读缓存） | 新模拟无法存储，历史数据可读取 |
| **Kafka 堆积** | 写入限流，优先保证核心链路 | 部分异步任务延迟，不丢数据 |

### 9.9 场景化选型建议

| 业务规模 | 推荐架构 | 预估成本 | 复杂度 |
|---------|---------|---------|--------|
| **< 50 QPS** | 文件系统（当前 9.1-9.6 方案） | 极低 | 低 |
| **50-500 QPS** | 单机 Redis + 单机 MongoDB | 低 | 中 |
| **500-5000 QPS** | Redis Cluster + MongoDB 副本集 | 中 | 高 |
| **> 5000 QPS** | Redis Cluster + MongoDB 分片 + Kafka | 高 | 很高 |

### 9.10 轻量级文件系统方案（附录A：< 50 QPS 场景）

如果业务规模较小，可使用简化版文件系统方案：

```
output/
├── personas/                          # 画像资产
│   ├── persona-001.md
│   ├── persona-001.json
│   └── ...
├── simulations/                       # 模拟记录
│   ├── by-persona/
│   ├── by-scenario/
│   └── by-batch/
├── index/                             # 索引文件
│   ├── persona-index.json
│   └── simulation-index.jsonl
└── cache/                             # 本地缓存
    ├── hot-personas/
    └── hot-simulations/
```

**格式选择**：
- 画像：`Markdown`（展示）+ `JSON`（解析）
- 批量记录：`JSON Lines`（追加写友好）
- 索引：`JSON`（轻量）或 `SQLite`（复杂查询时）

**索引更新**：
- 每次生成追加写入 `simulation-index.jsonl`
- 每天批量重建 `persona-index.json`
- 标签索引按需重建
