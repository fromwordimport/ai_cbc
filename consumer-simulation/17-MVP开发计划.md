# MVP开发计划

> **版本**：v1.0
> **定位**：基于现有文档体系，制定可执行的消费者模拟Agent产品开发路线图
> **目标读者**：产品经理、技术负责人、开发团队

---

## 一、开发目标

### 1.1 MVP定义

**MVP（Minimum Viable Product）**：一个可通过API调用、能生成消费者画像并执行情境模拟、输出符合规范的最小可用系统。

**MVP不做什么**：
- ❌ 不实现多Agent自动协作（先单机串行）
- ❌ 不实现前端管理界面（先API+CLI）
- ❌ 不实现高并发存储（先单机文件存储）
- ❌ 不实现Bias Detector（先人工抽查）
- ❌ 不实现Kafka消息队列（先同步调用）

**MVP必须做**：
- ✅ 画像生成（批量/单个）
- ✅ 情境模拟（单条/批量）
- ✅ Schema + Logic 基础校验
- ✅ 真实感评分（规则引擎版）
- ✅ 任务状态查询
- ✅ 基础日志和错误处理

### 1.2 成功标准

| 指标 | 目标值 | 验证方式 |
|------|--------|---------|
| 画像生成成功率 | >= 75% | 100个画像批次测试 |
| 模拟输出平均分 | >= 8.5 | 人工抽查30条 |
| 端到端延迟 | < 60秒 | P95延迟测试 |
| API可用性 | >= 99% | 7天持续监控 |
| 代码测试覆盖率 | >= 60% | pytest覆盖率报告 |

## 二、技术栈选型

### 2.1 后端

| 组件 | 选型 | 理由 |
|------|------|------|
| **语言** | Python 3.11+ | LLM生态最成熟（LangChain/OpenAI SDK） |
| **框架** | FastAPI | 异步支持好、自动API文档、类型提示 |
| **ORM** | Pydantic + Beanie | 与FastAPI原生集成，MongoDB异步驱动 |
| **任务队列** | Celery + Redis | 轻量、文档丰富、无需Kafka |
| **缓存** | Redis | 画像缓存、速率限制 |
| **数据库** | MongoDB 6 | 文档型适合画像结构、水平扩展 |
| **存储** | 本地文件系统 | MVP阶段够用，后续迁S3 |

### 2.2 LLM接入

| 场景 | 模型 | 优先级 |
|------|------|--------|
| 画像生成 | Claude Sonnet 4.6 | P0 |
| 模拟执行 | Claude Sonnet 4.6 | P0 |
| 审校评分（语义维度） | Claude Haiku 4.5 | P1 |
| 备用模型 | GPT-4o | P1 |

### 2.3 基础设施

| 组件 | 选型 | 理由 |
|------|------|------|
| **部署** | Docker Compose | MVP阶段足够，后续迁K8s |
| **监控** | Prometheus + Grafana | 开源、生态成熟 |
| **日志** | 结构化JSON日志 | 简单、可搜索 |
| **CI/CD** | GitHub Actions | 免费、与GitHub集成 |

### 2.4 不选型（MVP阶段排除）

| 技术 | 排除理由 | 何时引入 |
|------|---------|---------|
| Kubernetes | 运维复杂度高 | Phase 2（生产部署） |
| Kafka | 需要额外运维 | Phase 2（高并发场景） |
| Elasticsearch | 搜索需求简单 | Phase 2（全文检索） |
| React/Vue前端 | MVP先API | Phase 2（管理后台） |
| 对象存储(S3) | 文件系统够用 | Phase 2（归档场景） |

## 六、风险与应对

### 6.1 技术风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| LLM输出格式不稳定 | 高 | 解析失败，生成率低 | ①增强Prompt格式约束②降低temperature③重试机制 |
| MongoDB性能不足 | 低 | 查询慢 | MVP数据量小，后期加索引/分片 |
| Celery Worker崩溃 | 中 | 任务丢失 | ①持久化任务状态②失败任务重入队③监控告警 |

### 6.2 进度风险

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| LLM API调用调试耗时超预期 | 高 | Week1-2延期 | ①预留buffer时间②先做mock测试③并行开发 |
| 输出质量不达标 | 中 | Week4延期 | ①提前在Week2做质量验证②Prompt A/B测试 |
| 需求变更 | 中 | 返工 | ①严格MVP范围②变更需记录影响③每周对齐 |

### 6.3 成本风险

| 阶段 | LLM预算 | 监控方式 |
|------|--------|---------|
| Week1-2开发 | ¥500 | 每日查看用量 |
