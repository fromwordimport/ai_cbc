# 画像资产化管理规范

> **版本**：v1.0
> **定位**：定义消费者画像作为企业数字资产的版本控制、去重、检索、血缘追踪与协作管理规范
> **使用说明**：当画像数量超过 100 个、需要团队协作、或需要长期运营维护时启用本规范

---

## 一、画像生命周期管理

### 1.1 生命周期状态图

```
[草稿] ──→ 创建中，未通过审校
    │
    ▼
[已审校] ──→ 通过Harness校验，可用
    │
    ├──→ [已发布] ──→ 正式投入模拟使用
    │       │
    │       ├──→ [已归档] ──→ 长期不用，但保留
    │       │
    │       └──→ [已废弃] ──→ 质量问题或过时，标记删除
    │
    └──→ [已驳回] ──→ 审校未通过，退回修改
```

### 1.2 状态转换规则

| 转换 | 触发条件 | 操作人 |
|------|---------|--------|
| 草稿 → 已审校 | Harness 校验通过（总分 ≥ 9） | 系统自动 |
| 已审校 → 已发布 | 研究人员手动确认 | 研究员 |
| 已发布 → 已归档 | 90 天无使用记录 | 系统自动 |
| 已发布 → 已废弃 | 发现严重质量问题 / 标签体系变更 | 管理员 |
| 已驳回 → 草稿 | 修正后重新提交 | 研究员 |
| 已归档 → 已发布 | 重新激活使用 | 研究员 |

### 1.3 状态元数据

```json
{
  "persona_id": "persona-001",
  "status": "PUBLISHED",
  "status_history": [
    {"from": "DRAFT", "to": "REVIEWED", "at": "2026-06-01T10:00:00Z", "by": "system"},
    {"from": "REVIEWED", "to": "PUBLISHED", "at": "2026-06-01T11:00:00Z", "by": "researcher_xxx"}
  ],
  "usage_stats": {
    "total_simulations": 45,
    "last_used_at": "2026-06-08T14:30:00Z",
    "scenarios_used": ["skincare", "digital", "food"]
  }
}
```

## 二、版本控制

### 2.1 版本号规则

采用语义化版本：`主版本.次版本.修订号`

| 版本变化 | 触发条件 | 示例 |
|---------|---------|------|
| **主版本 +1** | 核心身份标签变更（年龄/职业/收入档位） | 1.0.0 → 2.0.0 |
| **次版本 +1** | 行为/心理标签显著变更 | 1.1.0 → 1.2.0 |
| **修订号 +1** | 小传/语言样本优化，标签不变 | 1.1.0 → 1.1.1 |

### 2.2 版本历史

```json
{
  "persona_id": "persona-001",
  "current_version": "1.2.0",
  "versions": [
    {
      "version": "1.0.0",
      "created_at": "2026-06-01T10:00:00Z",
      "change_summary": "初始创建",
      "change_type": "CREATE",
      "author": "researcher_xxx"
    },
    {
      "version": "1.1.0",
      "created_at": "2026-06-05T09:00:00Z",
      "change_summary": "调整核心焦虑：从'同辈压力'改为'育儿焦虑+同辈压力'",
      "change_type": "UPDATE",
      "author": "researcher_yyy",
      "diff": {
        "changed_fields": ["layer3.psychology.core_anxieties"],
        "old_values": {"core_anxieties": ["同辈压力"]},
        "new_values": {"core_anxieties": ["育儿焦虑", "同辈压力"]}
      }
    },
    {
      "version": "1.2.0",
      "created_at": "2026-06-08T14:00:00Z",
      "change_summary": "增加场景反应：添加'产品故障售后'场景",
      "change_type": "UPDATE",
      "author": "researcher_xxx"
    }
  ]
}
```

### 2.3 版本对比

支持两个版本之间的字段级对比：

```
对比 persona-001 v1.1.0 vs v1.2.0

变更字段：
  + layer4.scene_reactions[5] = "产品出现故障需要售后时 → 直接找客服..."
  ~ layer3.psychology.core_anxieties = ["育儿焦虑", "同辈压力"] → ["育儿焦虑"]

影响评估：
  - 场景反应库扩展，可能影响模拟结果
  - 核心焦虑减少一个，人物形象更聚焦
  - 建议重新跑一遍历史模拟，验证一致性
```

## 三、去重策略

### 3.1 重复类型

| 类型 | 定义 | 检测方法 |
|------|------|---------|
| **完全重复** | 两个画像所有字段相同 | 哈希值比对 |
| **语义重复** | 标签组合高度相似，仅语言表述不同 | Embedding相似度 |
| **功能重复** | 标签组合不同，但模拟输出高度一致 | 输出对比 |

### 3.2 去重检测流程

```
新画像生成
    │
    ▼
┌───────────────┐
│ 完全重复检测   │ ← MD5哈希比对
│ (O(1))        │
└───────┬───────┘
    重复？
   是 / 否
    │     ▼
    │  ┌───────────────┐
    │  │ 语义重复检测   │ ← Embedding余弦相似度
    │  │ (O(N))        │
    │  └───────┬───────┘
    │      相似度>0.85？
    │     是 / 否
    │      │     ▼
    │      │  ┌───────────────┐
    │      │  │ 功能重复检测   │ ← 相同场景模拟输出对比
    │      │  │ (抽样验证)     │
    │      │  └───────┬───────┘
    │      │      输出一致？
    │      │     是 / 否
    │      │      │
    ▼      ▼      ▼
  [拒绝] [警告] [通过]
```

### 3.3 去重阈值

| 检测层 | 阈值 | 动作 |
|--------|------|------|
| 完全重复 | 哈希相同 | 自动拒绝，提示"画像已存在" |
| 语义重复 | 相似度 > 0.90 | 拒绝，提示"与 persona-xxx 过于相似" |
| 语义重复 | 相似度 0.80-0.90 | 警告，要求研究员确认差异点 |
| 功能重复 | 模拟输出一致率 > 80% | 警告，建议合并或差异化 |

### 3.4 画像指纹

用于快速去重检测的画像指纹：

```json
{
  "persona_fingerprint": {
    "structural_hash": "sha256(layer1+layer2_tags_sorted)",
    "semantic_vector": [0.12, -0.34, 0.89, ...],
    "behavior_signature": "年龄组_性别_城市层级_收入_核心焦虑_价格敏感度",
    "example": "25-34_女_新一线_15-30万_同辈压力_理性比价"
  }
}
```

## 四、检索与发现

### 4.1 检索维度

| 维度 | 检索方式 | 示例 |
|------|---------|------|
| **标签精确匹配** | 等于/不等于 | `年龄=25-34 AND 性别=女` |
| **标签范围匹配** | 在列表中 | `城市层级 IN [一线, 新一线]` |
| **全文检索** | 模糊匹配 | `小传包含"加班"` |
| **语义检索** | Embedding相似 | `与"焦虑型妈妈"概念相似的画像` |
| **组合检索** | 复杂布尔 | `(收入>30万 AND 极简主义) OR (学生 AND 信贷消费)` |
| **使用记录** | 时间范围 | `最近7天被使用过的画像` |

### 4.2 检索接口设计

```python
class PersonaSearchEngine:
    def search(self, query: SearchQuery) -> SearchResult:
        """
        输入: 复合查询条件
        输出: 画像列表 + 分页信息
        """
        pass

@dataclass
class SearchQuery:
    filters: list[Filter]           # 标签过滤条件
    text_query: str | None          # 全文检索关键词
    semantic_query: str | None      # 语义检索描述
    sort_by: str = "created_at"     # 排序字段
    sort_order: str = "desc"        # asc / desc
    page: int = 1
    page_size: int = 20

@dataclass
class Filter:
    field: str                      # 字段路径，如 "layer1.demographics.age"
    operator: str                   # eq / ne / in / gt / lt / contains
    value: any
```

### 4.3 预置检索模板

为常见业务场景提供一键检索：

```markdown
【快速检索模板】

1. "高价值但价格敏感"
   条件：收入>30万 AND 价格敏感度=高度敏感
   用途：测试高端产品的促销策略

2. "冲动型深夜消费者"
   条件：决策周期=即时决策 AND 购买时间偏好=深夜
   用途：测试深夜直播带货策略

3. "成分党妈妈"
   条件：核心焦虑=育儿焦虑 AND 信息处理=参数党
   用途：测试母婴产品成分沟通

4. "最近未使用的画像"
   条件：last_used_at < 30天前
   用途：激活沉睡画像资产

5. "真实感评分TOP10"
   条件：authenticity_score >= 12
   排序：authenticity_score DESC
   用途：筛选优质画像用于关键研究
```

## 五、血缘追踪

### 5.1 血缘关系类型

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| **派生** | A是B的修改版 | persona-002 由 persona-001 微调而来 |
| **衍生** | 从同一种子生成 | persona-003 和 persona-004 使用相同种子 |
| **引用** | A在模拟中使用了B | batch-001 的模拟使用了 persona-001 |
| **影响** | A的变更影响了B | persona-001 v1.2 导致 simulation-xxx 结果变化 |

### 5.2 血缘图谱

```json
{
  "persona-002": {
    "lineage": {
      "derived_from": "persona-001",
      "derivation_reason": "将城市从'一线城市'改为'新一线'，观察消费行为差异",
      "same_seed_as": ["persona-003"],
      "used_in_batches": ["batch-20250608-001", "batch-20250610-002"],
      "used_in_simulations": ["sim_001", "sim_002", "sim_015"]
    }
  }
}
```

### 5.3 影响分析

当画像发生变更时，自动分析影响范围：

```
persona-001 发生变更：v1.1.0 → v1.2.0
变更字段：layer3.psychology.core_anxieties

影响分析：
├─ 直接引用该画像的批次：3个
│   ├─ batch-20250608-001（45条模拟记录）
│   ├─ batch-20250608-002（30条模拟记录）
│   └─ batch-20250610-001（20条模拟记录）
│
├─ 建议操作：
│   ├─ [ ] 重新运行受影响的模拟（推荐）
│   ├─ [ ] 仅标记"历史数据基于旧版本"
│   └─ [ ] 忽略（如果变更为小修小补）
│
└─ 血缘图谱预览：
    persona-001 v1.2.0
        ├──→ batch-20250608-001 [待更新]
        ├──→ batch-20250608-002 [待更新]
        └──→ batch-20250610-001 [待更新]
```

## 六、权限与协作

### 6.1 角色定义

| 角色 | 权限 | 说明 |
|------|------|------|
| **管理员** | 全部 | 系统配置、用户管理、画像废弃 |
| **高级研究员** | 创建、修改、发布、归档 | 可以修改他人创建的画像 |
| **研究员** | 创建、修改（自己的）、查看全部 | 只能修改自己创建的画像 |
| **分析师** | 查看、使用、评论 | 不能修改画像，只能用于模拟 |
| **访客** | 查看（已发布） | 只能查看已发布的画像 |

### 6.2 协作工作流

```
研究员A创建画像
    │
    ▼
[草稿] ──→ 自动进入 Harness 审校
    │
    ▼
[已审校] ──→ 研究员B（同行）复核
    │           │
    │           ├──→ 提出修改意见（评论）
    │           │
    │           └──→ 确认通过
    │
    ▼
[已发布] ──→ 全员可用
    │
    ▼
分析师C使用该画像执行模拟
    │
    ▼
[模拟记录] ──→ 关联到画像血缘
```

### 6.3 评论与标注

支持对画像的字段级评论：

```json
{
  "comments": [
    {
      "comment_id": "cmt_001",
      "persona_id": "persona-001",
      "field_path": "layer3.psychology.tension_combination",
      "author": "researcher_b",
      "content": "这个矛盾张力解释有点单薄，建议补充童年经历的细节",
      "created_at": "2026-06-02T10:00:00Z",
      "status": "OPEN",  // OPEN | RESOLVED | DISMISSED
      "replies": []
    }
  ]
}
```

## 七、归档与清理

### 7.1 自动归档规则

| 条件 | 动作 | 保留期限 |
|------|------|---------|
| 90天无使用 | 状态改为"已归档" | 永久保留 |
| 真实感评分 < 6 | 状态改为"已废弃" | 保留1年后删除 |
| 画像被新版本替代 | 旧版本归档 | 保留3个历史版本 |
| 批次报告生成后30天 | 原始模拟记录压缩归档 | 保留2年 |

### 7.2 清理策略

```markdown
【定期清理任务】（每月执行一次）

1. 扫描"已废弃"画像
   - 删除超过1年的废弃画像
   - 释放存储空间
   - 更新索引

2. 压缩历史批次
   - 30天前的批次报告保留摘要
   - 原始JSON Lines压缩为 gzip
   - 转存至冷存储（S3）

3. 重建索引
   - 优化检索性能
   - 清理孤儿索引项

4. 生成资产报告
   - 画像总数、分布、质量统计
   - 存储使用量
   - 高频使用画像TOP20
```

## 八、与存储层的衔接

本规范与 `09-Agent输出信息规范.md` 中的存储设计形成"管理-存储"分层：

| 层面 | 文档 | 关注点 |
|------|------|--------|
| **存储层** | `09` 第9章 | 数据怎么存（Redis/MongoDB/S3） |
| **管理层** | `12`（本文档） | 数据怎么管（版本/去重/检索/权限） |

**衔接点**：
- 版本历史存储在 MongoDB 的 `persona_versions` 集合
- 血缘关系存储在 MongoDB 的 `persona_lineage` 集合
- 评论存储在 MongoDB 的 `persona_comments` 集合
- 检索通过 MongoDB 索引 + Redis 缓存实现
- 归档通过生命周期规则自动触发 S3 迁移
